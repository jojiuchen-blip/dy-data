# 页面交互差异 - dy-data（DYDATA-32）

> 生成时间: 2026-07-20
> Skill: page-explainer
> 对比基线: 用户已确认的 DYDATA-32 流程、`docs/rules/account-access-control.md`、真实 `/admin/accounts` 运行页面

## 差异清单

### GAP-001：现有角色选项与目标三角色不一致

- **分类**: `logic_conflict`
- **所在页面**: 账号管理 (`/admin/accounts`)
- **所在模块**: 账号表单、账号列表
- **所在元素**: 角色
- **证据来源**: Codex Browser 实际展开角色选项；`AdminAccountsPage.tsx`；用户确认的 DYDATA-32 规则
- **是否原型 / mock 态**: 否；这是现有生产页面角色文案和提交值
- **现状**: 角色为“门店账号、全局查看、最高管理员”，对应 `store / viewer / admin`；其中 `admin` 显示为最高管理员。
- **预期**: 页面只表达“最高管理员、管理员、门店账号”三种业务角色；迁移后的管理员不再显示为“全局查看”，最高管理员与管理员边界清楚。
- **修改建议**: page-designer 将账号表单和列表统一到目标三角色文案，并在角色选择中体现操作者可管理的角色范围。
- **解决记录**: `resolved` — page-designer loop 1 已在隔离预览统一目标三角色；用户于 2026-07-20 确认。

### GAP-002：缺少“账号列表 / 角色权限”两个页签和角色默认权限矩阵

- **分类**: `design_gap`
- **所在页面**: 账号管理 (`/admin/accounts`)
- **所在模块**: 页面主体
- **所在元素**: 页面页签、角色权限入口
- **证据来源**: Codex Browser 页面截图和 DOM 快照；用户确认的流程 2
- **是否原型 / mock 态**: 否；运行页面只有账号列表、账号表单和未激活门店
- **现状**: 页面没有页签，也没有“角色 × 当前登记页面”的默认页面权限配置。
- **预期**: 页面保持两个页签：“账号列表”和“角色权限”；角色权限页签纵向展示最高管理员、管理员、门店账号，横向展示当前登记页面。最高管理员只读全选。
- **修改建议**: page-designer 在同一路由内增加稳定的两个页签和页面权限矩阵，不增加第三个页签；矩阵必须能表达页面分组但不能把菜单分组当权限项。
- **解决记录**: `resolved` — page-designer loop 1 已增加固定双页签和角色 × 页面矩阵；用户于 2026-07-20 确认。

### GAP-003：缺少单账号差异权限、有效权限预览和恢复默认

- **分类**: `design_gap`
- **所在页面**: 账号管理 (`/admin/accounts`)
- **所在模块**: 账号列表、账号编辑
- **所在元素**: 单账号页面权限
- **证据来源**: Codex Browser 页面截图和账号行操作；用户确认的流程 3
- **是否原型 / mock 态**: 否；运行账号行只有编辑和重置密码
- **现状**: 无法查看角色默认权限、额外允许、额外禁止和最终有效权限，也没有“恢复角色默认权限”。
- **预期**: 有权操作者可从单个账号进入差异权限设置；页面区分默认、额外允许、额外禁止和最终有效结果，并支持单账号恢复默认。
- **修改建议**: page-designer 为账号行或编辑区增加明确入口，使用同一页面目录展示四类权限状态；恢复默认前提供确认，不提供批量重置所有自定义账号。
- **解决记录**: `resolved` — page-designer loop 1 已增加单账号差异、有效权限摘要和恢复默认确认；用户于 2026-07-20 确认。

### GAP-004：缺少默认权限变更影响提示和确认步骤

- **分类**: `design_gap`
- **所在页面**: 账号管理 (`/admin/accounts`)
- **所在模块**: 目标角色权限页签
- **所在元素**: 保存角色默认权限
- **证据来源**: 用户确认的流程 2；当前运行页面无对应入口
- **是否原型 / mock 态**: 否；已确认该提示必须在当前页面显式出现
- **现状**: 页面无法修改角色默认权限，因此也没有变更内容、继承账号数量和自定义账号处理说明。
- **预期**: 保存前显示本次增减页面、受影响继承账号数量，并明确继承账号自动更新、自定义账号最终有效权限保持不变；用户确认后才保存。
- **修改建议**: page-designer 为默认权限保存增加影响预览和确认对话；自定义账号默认保持不变，同时提供明确且默认不勾选的“将自定义账号重置为新默认权限”可选项。
- **解决记录**: `resolved` — page-designer loop 1 已增加默认变更影响确认；继承账号自动更新，自定义账号默认保持，并提供显式可选重置；用户于 2026-07-20 确认。

### GAP-005：缺少权限变更记录入口和筛选查看

- **分类**: `design_gap`
- **所在页面**: 账号管理 (`/admin/accounts`)
- **所在模块**: 页面头部或账号列表上下文
- **所在元素**: 变更记录
- **证据来源**: Codex Browser 页面截图和 DOM 快照；用户确认的流程 4
- **是否原型 / mock 态**: 否；当前页面完全没有相关入口
- **现状**: 无法从账号管理页查看账号、角色默认权限、单账号差异、门店范围和密码重置等变更历史。
- **预期**: 在账号管理上下文提供“变更记录”入口，但不增加第三个页签；支持按账号、操作者、时间和操作类型筛选，不提供第一版导出。
- **修改建议**: page-designer 增加独立入口和记录查看层，明确最高管理员查看全部、管理员仅查看门店账号相关记录、门店账号不可进入。
- **解决记录**: `resolved` — page-designer loop 1 已增加变更记录入口、筛选和“不提供导出”说明；用户于 2026-07-20 确认。

### GAP-006：门店范围交互不能表达目标范围模式和管理员可受限范围

- **分类**: `logic_conflict`
- **所在页面**: 账号管理 (`/admin/accounts`)
- **所在模块**: 账号表单
- **所在元素**: 门店权限
- **证据来源**: Codex Browser 选择“全局查看”后门店权限禁用并显示“全部门店”；`compactPayload`；用户确认的门店范围规则
- **是否原型 / mock 态**: 否；这是当前表单真实联动
- **现状**: 非门店角色固定全部门店且无法编辑；门店账号可保持“未绑定门店”，页面没有显式的“全部门店 / 指定门店 / 无门店数据权限”范围语义。
- **预期**: 最高管理员固定全部门店；管理员默认全部门店但最高管理员可改为至少一个指定门店；门店账号必须至少选择一个门店才能由后台创建。空清单不代表全部门店。
- **修改建议**: page-designer 将范围模式和门店选择分开表达；按角色显示可选模式、最低选择要求和禁止保存原因，并避免用空值表达全部门店。
- **解决记录**: `resolved` — page-designer loop 1 已显式呈现全部门店、指定门店、空范围含义和角色约束；用户于 2026-07-20 确认。

### GAP-007：最高管理员账号治理和失效约束没有可见入口

- **分类**: `design_gap`
- **所在页面**: 账号管理 (`/admin/accounts`)
- **所在模块**: 账号列表、账号表单
- **所在元素**: 新建最高管理员、重置密码、失效
- **证据来源**: Codex Browser 登录环境最高管理员后，账号列表不展示当前环境管理员；当前账号行操作；用户确认的最高管理员规则
- **是否原型 / mock 态**: 否；当前运行页面无法表达已确认的最高管理员治理流程
- **现状**: 表单可以选择“最高管理员”，但页面没有表达任意一名最高管理员可直接创建、重置其他最高管理员密码和使其他最高管理员失效，也没有“不得使自己失效”的可见约束。
- **预期**: 有权最高管理员能够管理其他最高管理员的创建、密码重置和失效；不要求双人确认，不允许使自己失效，不提供直接降级。
- **修改建议**: page-designer 在账号列表和表单中区分当前自己与其他最高管理员，提供允许的动作并对自失效、直接降级显示不可用原因。
- **解决记录**: `resolved` — page-designer loop 1 已呈现当前最高管理员不可自失效、其他最高管理员可重置和失效、无需双人确认且不直接降级；用户于 2026-07-20 确认。

### GAP-008：未激活门店说明容易被理解为预先创建了空范围账号

- **分类**: `logic_conflict`
- **所在页面**: 账号管理 (`/admin/accounts`)
- **所在模块**: 未激活门店
- **所在元素**: 模块说明
- **证据来源**: Codex Browser 运行页面文案；用户确认的自助激活兼容规则
- **是否原型 / mock 态**: 否；这是当前用户可见说明
- **现状**: 页面写“已准备但尚未激活账号的门店”，容易被理解为管理员需要预建账号或存在门店范围为空的占位账号。
- **预期**: 明确“未激活门店”是尚未产生已初始化门店账号的门店；自助激活无需预建账号，并自动绑定核验出的唯一门店。
- **修改建议**: page-designer 调整模块说明，避免“已准备账号”表述；后台手工创建与自助激活两条路径的结果和提示保持一致。
- **解决记录**: `resolved` — page-designer loop 1 已改为双 ID 校验后自动创建或绑定并限定唯一门店的说明；用户于 2026-07-20 确认。

### GAP-010：已激活与未激活列表分成两个大板块，浏览路径偏长

### GAP-011：账号列表点击编辑后缺少可感知的页面指引

- **分类**: `design_gap`
- **所在页面**: 账号管理 (`/admin/accounts?preview=dydata32`)
- **所在模块**: 已激活账号列表、账号配置
- **所在元素**: 编辑、页面权限
- **证据来源**: 用户于 2026-07-20 的浏览器反馈
- **是否原型 / mock 态**: 是；问题发生在隔离交互预览
- **现状**: 点击“编辑”只更新页面状态，长页面中没有滚动、聚焦或提示，用户难以判断结果出现在哪里。
- **预期**: 点击“编辑”后选中对应账号并明确跳转到“账号配置”；页面权限入口采用一致行为。
- **修改建议**: 使用平滑滚动将账号配置区定位到固定顶栏下方；让配置区获得程序化焦点，显示所选账号名称，并以明显的边框和状态提示反馈。
- **解决记录**: `resolved` — page-designer loop 1 已实现选中账号、平滑跳转、配置区焦点边框和包含账号名的状态提示；浏览器验证两条路径均定位到同一配置区，用户于 2026-07-20 确认。
