## 3. 业务目标

- 需求提出者：`【用户确认】 统筹整个系统开发的负责人，负责汇总业务需求和协调开发，本人不是主要日常使用者`
- 目标使用者：`【用户确认】 集团财务、经销商财务、经销商抖音负责人、负责客户电话跟进的经销商客服、经销商侧多店管理人员、查看片区门店的区域管理人员、查看全国经营情况的总部管理人员，以及承担账号权限、规则配置和运行维护的授权后台管理员`
- 被服务业务对象：`【用户确认】 购买相关服务的客户是线索跟进与服务交付流程的业务对象，不是本系统的直接使用者`
- 管理角色层级：`【用户确认】 经销商侧可能由一名管理人员负责多个门店；区域管理人员按所辖片区查看门店；总部管理人员查看全国范围。各层级的核心差异是授权数据范围、汇总粒度和下钻边界`
- 需求证据现状：`【用户确认】 当前判断主要基于对既有业务流程的分析，辅以亲自观察和使用者直接反馈；各角色痛点及量化基线仍需在 BRD 中区分已验证事实与待验证假设`
- 角色利益关系：`【用户确认】 各角色关注点差异判断准确，但目前尚未出现实质利益冲突；差异先作为权限、信息范围和业务口径设计约束处理`
- 主要问题：`【用户确认】 将分散的订单、核销、销售、线索和门店运营数据转为可信、可追溯、可复核的经营决策与业务执行依据；其中分账属于新业务，在实际数据规模和规则复杂度下脱离系统几乎无法人工完成`
- 财务角色痛点：`【用户确认】 分账并非已有成熟人工流程的线上提效，而是系统要建立的新业务能力；集团财务需要系统化计算和复核订单、核销、跨店关系、扣减及结算结果，否则在实际规模下几乎无法完成分账；经销商财务需要清晰可复核的本店结算依据`
- 抖音负责人及客服痛点：`【用户确认】 经销商抖音负责人的销售、订单、核销和线索信息分散，难以判断门店、商品和线索表现并跟进异常；客服的线索分配、联系与跟进过程分散，容易出现重复、遗漏、延迟以及完成情况不可见`
- 管理角色痛点：`【用户确认】 多店、区域和总部管理人员缺少按自身管辖范围统一查看销售、核销、线索跟进、服务交付及异常情况的入口，往往需要逐店收集或依赖人工汇报，难以及时横向比较门店表现、下钻异常原因并督促改进；三个层级的主要区别是可见范围和汇总粒度不同`
- 后台管理员痛点：`【用户确认】 账号与门店权限、SKU 及归属规则、商品可见性、线索分配配置和数据同步运行维护涉及多个业务域；如果缺少统一、可追溯的管理入口，配置容易错配，变更影响范围不清晰，出现数据或业务异常时也难以快速定位原因`
- 结算中心业务定位：`【用户确认】 作为抖音分佣新业务成立所依赖的数字化承接平台，系统化计算并沉淀订单、核销、跨店、分佣与结算参考依据，并支撑后续对账和开票信息能力`
- 线索跟进平台业务定位：`【用户确认】 通过线索分配和跟进约束指导经销商规范服务行为，提升服务购买后的核销率、整体服务质量与服务交付比例`
- 产品边界：`【用户确认】 系统提供经营分析、线索运营、后台配置和分账参考，不执行真实资金划拨`
- 第一版核心目标：`【用户确认】 形成覆盖经营与结算、线索运营、后台管理和数据平台的一体化经营工作台，提供可追溯的经营与分账参考，不执行真实资金动作`
- 核心价值模型：`【用户确认】 准确分佣与有效服务交付构成并列核心价值；经营数据和后台管理用于支撑两个业务闭环，不作为独立首要价值`
- 效率目标：`【用户确认】 以业务可运行性与流程留痕完整性验收。财务侧全部当前已知且持续发生的标准分账场景必须由系统支持，仅源数据缺失、归属争议或首次出现的新规则允许线下处理。线索侧所有权限内合法跟进操作必须成功保存、查询并追溯到订单或线索、分配轮次、门店、操作人和时间，不允许静默丢失；联系结果、首次联系时长和核销转化率不作为本轮基础流程跑通的硬门槛。验收时间与项目阶段目标一致，为 2026 年 7 月 22 日。`
- 当前工作流：`【用户确认】 抖音订单、核销、销售和线索数据进入系统后，与门店、商品及业务规则建立关联；财务复核销售归属、核销门店、跨店关系和结算参考结果，线索按规则分配至门店后由抖音负责人或客服联系客户并记录跟进，管理人员按单店、多店、区域或全国权限查看经营表现、服务交付和异常，后台管理员维护权限、业务规则、线索分配和数据同步并处理异常；系统不执行真实资金动作，对账、开票信息获取和 CLI 接入不计入当前已运行流程`
- 关键风险与兜底：`【用户确认】 完整覆盖源数据质量、财务规则错误、分层权限越界、CLI 只读边界失效、线索过程失真和目标范围过大六类风险；分别通过数据校验与人工复核、规则版本化与试运行、统一鉴权与服务端过滤、只读白名单与审计、关键节点留痕与抽查、按业务域分阶段验收进行兜底`

## 4. 页面与任务定位

- 项目覆盖端：`【用户确认】 覆盖 Web 业务端和 CLI / Agent 严格只读接入端。Web 承载经营与结算、线索运营、经营数据查看和后台管理等页面能力；CLI / Agent 是无传统页面的授权查询渠道。明确不建设面向被服务客户的直接使用端。单店、多店、区域和全国属于数据权限范围，结算、线索、数据表现和后台管理属于业务模块，均不作为独立端`
- 各端目标用户：`【用户确认】 Web 业务端面向集团财务、经销商财务、经销商抖音负责人、经销商客服、经销商侧多店管理人员、区域管理人员、总部管理人员及授权后台管理员，各角色按账号、组织与数据权限使用对应能力。CLI / Agent 端面向全部已授权系统用户，使用同一系统账号并继承角色、组织范围与数据权限，但渠道始终严格只读。购买服务的客户不是任何端的直接使用者`
- 各端主要用途：`【用户确认】 Web 业务端用于完成经营数据查看与分析、财务结算与复核、对账和开票信息获取、线索分配与跟进、后台规则和权限配置，以及数据同步与生产运行管理等完整业务工作。CLI / Agent 端用于在账号权限范围内查询、筛选、汇总和导出业务数据；不得通过该渠道新增、修改、分配、提交、审批、触发同步或执行任何会改变业务状态的操作`
- 页面定位：`【用户确认】 Web 页面按用户进入页面时的主要任务分别定位。查看型页面承载经营看板、门店排名、结算结果、订单明细、线索表现和分层管理汇总；操作型页面承载财务复核与异常处理、对账处理、线索分配与跟进、反馈和数据同步运行；配置型页面承载账号与门店权限、SKU 与归属规则、商品可见性和线索分配规则等后台配置。单个页面可以包含辅助能力，但主定位必须由核心用户任务决定；CLI / Agent 为非页面只读渠道，不适用页面定位分类`
