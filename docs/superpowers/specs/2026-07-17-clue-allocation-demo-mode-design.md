# 线索分配完整 Mock 演示模式设计

## 文档定位

本文档定义线索分配模块的本地完整演示模式。目标是在不连接真实后端、不复制真实业务记录、不暴露门店或客户信息的前提下，使用数百条合成数据完整呈现线索进入、规则匹配、门店分配、跟进、跨轮次流转、总部池兜底和业务结束的现有产品流程。

本设计只增加显式开发环境演示能力，不改变生产认证、真实 API、数据库模型或正式线索分配规则。线索分配业务口径继续以 `docs/superpowers/specs/2026-07-12-clue-allocation-engine-product-design.md` 为准。

## 目标

- 基于现有本地真实导出快照的聚合形态，生成稳定、可复现的合成演示数据。
- 在现有门店线索中心和线索分配后台中呈现同一套互相关联的数据。
- 覆盖筛选、分页、详情、跟进、分配试运行、分配记录、总部池和规则页面。
- 明确展示跟进记录和多轮流转历史，而不是只生成孤立列表行。
- 支持本地免登录预览，并确保该能力无法在生产构建中启用。
- 所有演示写操作只修改浏览器内存，刷新后恢复初始状态。

## 非目标

- 不把真实数据库复制到本地。
- 不把 `local_exports/` 中的原始文件打包进前端或提交到 Git。
- 不建设通用造数平台或可视化数据编辑器。
- 不使用演示模式验证真实后端事务、数据库迁移或并发行为。
- 不让演示数据反向写入本地或远端数据库。
- 不改变正式环境 API 错误处理和认证回退策略。

## 一、演示模式边界

### 1.1 启用条件

新增显式演示开关：

```text
VITE_DEMO_MODE=true
```

演示模式仅在以下两个条件同时满足时启用：

1. `import.meta.env.DEV` 为真。
2. `VITE_DEMO_MODE` 严格等于字符串 `true`。

推荐统一计算：

```text
DEMO_MODE = import.meta.env.DEV && VITE_DEMO_MODE === "true"
```

`DEMO_MODE` 为真时自动使用演示仓库；普通 `VITE_USE_MOCKS=true` 保留现有用途，不自动绕过认证。生产构建中 `import.meta.env.DEV` 为假，即使误传 `VITE_DEMO_MODE=true` 也不能启用演示身份或演示数据。

### 1.2 演示身份

演示模式启动后，`AuthGate` 直接获得一个固定的“演示最高管理员”用户：

- 用户 ID、用户名和门店范围均使用 `DEMO-` 前缀或演示专用值。
- 具备浏览线索中心和线索分配后台所需的最高管理员权限。
- 不调用 `/auth/me`、登录、退出或密码相关真实接口。
- 退出操作只清理前端演示状态，不创建真实会话。

非演示模式继续执行当前真实认证流程。

### 1.3 页面标识

演示模式下，应用外壳持续显示明确但不遮挡操作的状态标识：

```text
演示数据 · 全部为合成数据 · 不写入数据库
```

涉及跟进、试运行、重建、规则发布等写操作时，按钮或确认区补充“仅修改本次浏览器会话，刷新后重置”的说明。正式模式不显示这些标识。

## 二、真实数据只提供聚合形态

### 2.1 聚合参考

现有本地真实导出快照包含约 14.8 万条线索，覆盖订单状态、地区、商品、线索字段缺失情况和时间密度。实施时只提取并人工固化以下非敏感聚合信息：

- 订单状态权重。
- 省市层级的相对分布。
- 商品类型和商品缺失率。
- 关注门店锚点缺失率。
- 时间分布和日内节奏。
- 已分配、未分配及跟进状态的场景权重。

为了让演示页面稳定覆盖全部流程，生成器在参考真实权重的同时为关键状态设置最低样本量。演示结果不承诺与真实数据逐项等比例，只承诺整体形态合理且场景完整。

### 2.2 禁止进入代码库的内容

以下内容不得出现在前端资源、测试快照、文档示例或提交记录中：

- 真实门店名称及其可逆映射。
- 真实门店 ID、POI ID、生活服务账号 ID。
- 真实客户姓名、昵称、手机号、微信号和地址。
- 真实订单号、线索 ID、内容 ID、广告 ID。
- 真实跟进备注或运营人员姓名。
- 能通过组合字段重新识别真实记录的行级样本。

生成器不能在运行时读取 `local_exports/`，聚合权重必须作为不含原始值的静态配置维护。

## 三、合成数据规模

### 3.1 主体规模

默认生成一套固定规模的数据集：

- 480 条唯一线索。
- 48 家合成门店。
- 12 个城市，覆盖多个省级区域。
- 360 条至少成功分配过一次的线索。
- 60 条未创建轮次、直接进入总部池的线索。
- 60 条首次采集时已经核销或退款、未进入分配的线索。

360 条已分配线索按以下轮次结构生成：

- 230 条只有 1 个成功分配轮次。
- 90 条有 2 个成功分配轮次。
- 40 条有 3 个成功分配轮次。

合计约 530 个成功分配轮次。跳过策略和无候选策略只生成决策日志，不生成轮次，保持与正式业务规则一致。

### 3.2 跟进记录规模

默认生成 650 至 750 条跟进记录。每个成功分配轮次生成 0 至 4 条记录，具体数量由固定种子和场景约束决定。

跟进动作覆盖：

- `appointment`：已预约。
- `further_follow_up`：待进一步跟进。
- `unreachable`：未联系上。
- `lost`：暂不需要或战败。
- `request_store_change`：客户要求换店。

每条记录包含：

- 合成的跟进记录 ID。
- 所属订单、分配轮次和轮次编号。
- 当轮跟进门店。
- 跟进动作、合成备注和操作时间。
- 合成操作人。
- 操作后的时效或保护状态。
- 删除状态及删除审计字段，仅在少量演示场景中出现。

### 3.3 分配后台数据

同一生成过程还应产生：

- 规则、规则版本和三类策略配置。
- 待试运行线索和总部池线索。
- 试运行、重建和历史执行批次。
- 每个策略步骤的分配决策记录。
- 候选门店、排除原因、距离和评分快照。
- 48 家合成门店的评分快照。
- 规则发布、试运行、重建和状态变化的审计记录。

这些对象必须引用同一组线索、门店、规则、轮次和批次 ID，不能为不同页面分别生成互不关联的列表。

## 四、脱敏与合成规则

### 4.1 合成标识

- 所有业务 ID 使用 `DEMO-` 前缀。
- 门店名称使用“城市 + 演示门店 + 序号”，例如“深圳演示门店 03”。
- 客户昵称使用“演示客户 + 序号”。
- 手机号掩码使用“演示****0001”一类不可拨打文本。
- 查看完整手机号时返回“DEMO-PHONE-0001”一类演示值。
- 跟进备注从受控的合成话术字典中选择。
- 操作人使用“演示店员 01”“演示运营 01”等合成名称。

任何合成字段都不能通过哈希真实值、截断真实值或保留真实尾号生成，以免形成可逆或可关联映射。

### 4.2 地区与商品

- 省市名称可以使用公开行政区名称，不属于门店身份信息。
- 门店数量使用头部城市较多、长尾城市较少的分布。
- 商品类型覆盖现有线索中心使用的保养、洗美、轮胎等类型。
- 商品名称使用合成产品字典，不复制真实 SKU、商品 ID 或完整商品标题。
- 部分线索保留商品为空、锚点缺失等真实常见数据质量场景。

### 4.3 时间生成

- 生成器以可注入的 `now` 为时间锚点，默认使用本地当前日期。
- 使用固定随机种子保证同一天、同一版本下结果可复现。
- 分配、跟进、轮次结束、核销和退款时间必须按业务顺序递增。
- 历史轮次时间必须早于当前轮次。
- 跟进记录时间必须位于所属轮次的有效时间区间内。

测试通过固定 `now` 保证快照稳定，不直接依赖执行机器的真实时钟。

## 五、跟进与多轮流转模型

### 5.1 轮次内跟进

以下动作保留当前轮次：

- 已预约。
- 待进一步跟进。
- 未联系上。

首次发生上述动作时，当前轮次进入保护状态。后续记录仍绑定同一个 `assignment_round_id`，不会错误创建新轮次。

### 5.2 立即进入下一策略

以下动作结束当前轮次并触发下一策略：

- 暂不需要或战败。
- 客户要求换店。

下一策略选中门店后创建新的成功分配轮次，并满足：

- `round_no` 连续递增。
- 新门店不能是该线索历史已分配门店。
- 上一轮标记为历史失效轮次，不再允许操作。
- 当前轮次指针、当前门店和当前状态指向新轮次。
- 决策日志记录上一轮关闭原因和下一策略结果。

### 5.3 时效触发流转

以下场景同样结束当前轮次并尝试下一策略：

- 首次跟进超时。
- 保护期结束仍未核销。

演示数据需要同时覆盖自动流转成功和无候选两类结果。无候选时不创建虚假轮次，线索进入总部池并记录原因。

### 5.4 业务结束

- 核销后立即关闭当前轮次，不再进入下一策略。
- 退款后立即关闭当前轮次，不再进入下一策略。
- 首次采集时已核销或退款的线索不创建任何分配轮次。

### 5.5 详情展示一致性

线索详情按时间顺序呈现：

```text
进入主池 -> 第 1 轮分配 -> 多次跟进 -> 轮次结束 -> 第 2 轮分配 -> 跟进 -> 核销/退款/总部池
```

历史轮次展示当时的跟进门店、策略、分配时间、结束时间、结束原因和跟进记录。只有当前有效轮次显示可操作控件。

## 六、前端结构

### 6.1 模块边界

新增独立演示模块，建议目录：

```text
apps/web/src/demo/
  clueDemoProfile.ts
  clueDemoGenerator.ts
  clueDemoRepository.ts
  clueDemoAuth.ts
```

职责如下：

- `clueDemoProfile.ts`：非敏感聚合权重、合成词典和默认规模。
- `clueDemoGenerator.ts`：从配置、固定种子和时间锚点生成完整业务图谱。
- `clueDemoRepository.ts`：保存浏览器内存状态，提供查询和演示写操作。
- `clueDemoAuth.ts`：提供仅开发环境可用的演示最高管理员身份。

现有 `api/client.ts` 负责根据 `DEMO_MODE` 路由到真实 API 或演示仓库，不继续内嵌大段造数逻辑。现有 6 条静态线索样例可以保留作兼容测试，但完整演示模式以新仓库为准。

### 6.2 仓库查询能力

演示仓库必须支持现有页面使用的查询语义：

- 条件筛选。
- 排序和分页。
- 汇总指标。
- 订单详情。
- 分配轮次和跟进记录。
- 规则、规则版本和策略配置。
- 待试运行线索。
- 分配批次、决策、评分和审计记录。
- 总部池筛选和分页。
- 合成手机号查看。
- 客户端生成 CSV 导出。

所有列表和详情从同一份内存状态派生，避免汇总数字、分页总数和详情历史不一致。

### 6.3 演示写操作

以下操作在演示模式中允许执行，并只修改当前浏览器内存：

- 新增跟进记录。
- 最高管理员删除跟进记录。
- 预览和执行试运行。
- 重建历史试运行。
- 创建规则和草稿版本。
- 发布或停用规则版本。

写操作完成后，相关列表、汇总指标、详情时间轴、分配记录和总部池必须同步更新。刷新页面后重新运行生成器，恢复初始状态。

## 七、页面覆盖

### 7.1 门店线索中心

覆盖：

- 概览指标。
- 状态、门店、地区和商品筛选。
- 分页列表。
- 线索详情。
- 合成手机号查看。
- 跟进记录新增和删除。
- 历史轮次时间轴。
- 演示数据 CSV 导出。

### 7.2 线索分配后台

覆盖全部现有子页面：

- 分配规则。
- 分配试运行。
- 分配记录。
- 总部线索池。

试运行、记录和总部池页面中的同一 `DEMO-` 线索必须可以相互核对。规则页面展示的规则版本必须与决策记录使用的版本一致。

## 八、错误处理与安全约束

- 演示生成失败时在开发环境明确报错，不能静默改用真实 API。
- 非演示模式的真实 API 错误不能自动切换到免登录演示模式。
- 演示写操作找不到目标记录时返回与真实 API 语义一致的错误状态。
- 生成器在初始化后执行引用完整性校验，发现悬空 ID、重复 ID 或非法状态组合时立即失败。
- 所有演示写操作不得调用 `fetch`、下载生产资源或连接数据库。
- 导出文件名和内容必须明确包含 `demo` 或“演示数据”标识。

## 九、验证方案

### 9.1 数据不变量

自动验证：

- 480 条线索 ID 唯一。
- 所有业务 ID 均使用 `DEMO-` 前缀。
- 所有门店名符合演示命名规则。
- 所有客户、电话、订单和备注来自合成字典。
- 分配轮次编号连续，当前轮次指针唯一。
- 跟进记录只引用存在的轮次和门店。
- 历史轮次不可操作，当前有效轮次可操作。
- 轮次结束原因与下一轮、总部池或业务结束结果一致。
- 规则版本、决策、评分、批次和审计引用完整。
- 页面汇总等于筛选后列表计算结果。

### 9.2 开关与认证

验证：

- 开发环境未设置 `VITE_DEMO_MODE` 时仍使用真实认证。
- 开发环境设置 `VITE_DEMO_MODE=true` 时免登录进入演示身份。
- 生产构建即使设置 `VITE_DEMO_MODE=true` 也不会绕过认证。
- 演示模式不会发出真实业务 API 请求。

### 9.3 页面与交互

验证：

- 线索中心筛选、分页、详情、电话、跟进、删除和导出。
- 多轮线索的完整时间轴和历史轮次只读状态。
- 试运行预览、执行和重建后的跨页面更新。
- 分配记录中的策略、候选、评分和轮次信息。
- 总部池原因筛选和详情对应关系。
- 规则创建、发布和停用的内存状态变化。
- 刷新页面后演示数据恢复初始状态。

### 9.4 构建与浏览器验收

- 运行前端 TypeScript 检查和 Vite 构建。
- 运行现有 pytest 测试及新增的演示模式控制器测试。
- 使用 Playwright 检查至少一个桌面视口和一个移动视口。
- 保存线索中心、多轮详情、分配记录和总部池的最终截图。
- 检查页面无重叠、截断、空白数据区或不可见操作反馈。

## 十、启动与交付

本地演示通过 PowerShell 显式启动：

```powershell
$env:VITE_DEMO_MODE='true'
$env:VITE_USE_MOCKS='true'
npm run dev
```

启动目录为 `apps/web`。实施完成后应提供实际监听 URL、无错误说明、验证命令结果和截图路径。

## 验收结论

当以下条件全部满足时，本需求完成：

1. 本地无需后端和真实账号即可进入完整演示。
2. 线索中心和线索分配后台展示同一套数百条合成数据。
3. 跟进记录和 1 至 3 轮流转历史可完整查看并保持业务一致。
4. 演示操作可在内存中更新相关页面，刷新后恢复。
5. 代码库中不存在真实门店、客户或订单行级数据。
6. 生产认证、真实 API 和正式构建行为不受影响。
7. 自动化测试、构建和桌面/移动端浏览器验收通过。
