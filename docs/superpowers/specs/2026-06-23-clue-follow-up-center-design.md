# 线索跟进中心 MVP 设计

## 背景

线索跟进中心用于帮助门店处理仍有核销价值的抖音来客线索。MVP 只覆盖已进入待核销、履约中、可履约未核销状态的订单；未进入该状态的订单不进入线索池。

线索中心面向两类目标：

- 店端用户快速判断线索是否还能跟进、查看联系方式、登记本次跟进结果。
- 管理端后续可追溯订单状态、分配轮次、轮次失效、手机号查看与复制等内部行为。

## 范围

### 本期范围

- 店端线索明细字段与展示顺序。
- 店端统一线索状态。
- 当前线索跟进详情面板。
- 联系方式查看与复制权限。
- 跟进结果登记规则。
- 跟进历史展示规则。

### 暂不考虑

- 券过期、服务过期、异常关闭、核销撤销后回退等边缘订单状态。
- 门店自行理解或操作系统内部分配轮次 ID。
- 抖音 `follow_life_account_name` 作为我方线索归属展示。
- 再分配算法配置页。
- 呼叫能力。
- 失效后补录跟进结果或申诉流程。

## 核心数据概念

后台保留三类状态字段：

- `order_status`：订单状态，用于判断订单是否仍有跟进价值。
- `round_status`：分配轮次状态，用于判断当前门店这一轮是否仍可处理。
- `store_display_status`：店端显示状态，是店端唯一看到的主状态。

店端不直接展示后台订单状态、后台轮次状态、分配轮次 ID、抖音分配门店或门店 ID。

## 店端线索状态

店端状态枚举如下：

- `待跟进`：当前分配轮次仍有效，门店尚未登记跟进。
- `已跟进`：当前分配轮次仍有效，门店已登记跟进行为。
- `超期失效`：当前分配轮次因超过 SLA 未完成有效推进而失效。
- `主动战败`：门店主动选择线索战败，本轮立即失效。
- `已核销`：订单已经核销，业务目标完成，不再需要跟进。
- `已退款`：订单已退款，不再需要跟进。

`超期失效` 和 `主动战败` 是轮次层状态。`已核销` 和 `已退款` 是订单层结果。店端统一显示为 `线索状态`，不要求用户理解状态来源。

## 线索明细字段

线索明细字段顺序固定如下：

1. `联系方式`
2. `线索状态`
3. `分配轮次`
4. `本轮跟进时间`
5. `商品名称`
6. `商品类型`
7. `线索生成时间`
8. `本轮失效时间`

### 字段口径

`联系方式` 显示脱敏手机号。当前可跟进线索允许查看完整手机号并复制完整手机号；不可跟进线索只显示脱敏手机号和不可操作原因。

`线索状态` 只展示店端状态枚举。

`分配轮次` 显示 `第1轮`、`第2轮` 等自然语言，不展示内部 `assignment_round_id`。

`本轮跟进时间` 只显示当前明细行对应分配轮次的跟进时间。未跟进时为空。

`商品名称` 显示订单商品名称。长名称在列表中截断，详情中可放入商品信息区展示完整值。

`商品类型` 由后台维护的 `sku_id -> product_type` 规则确定，例如 `精诚养车`。

`线索生成时间` MVP 使用线索原始生成时间。后续自有分配系统上线后，再区分线索生成时间和分配给门店的时间。

`本轮失效时间` 只用于 `超期失效`、`主动战败` 等轮次失效状态。`待跟进`、`已跟进`、`已核销`、`已退款` 不填。

## 联系方式权限

明细页和详情面板都提供联系方式复制能力，但复制完整手机号必须与查看完整手机号使用同一权限判断。

### 当前可跟进

- 显示脱敏手机号。
- 显示 `查看完整手机号`。
- 显示复制按钮。
- 点击查看或复制时允许获取完整手机号。
- 查看和复制行为后续应进入审计记录。

### 超期失效

- 显示脱敏手机号。
- 不允许查看完整手机号。
- 不允许复制完整手机号。
- 原操作位置显示 `已失效不可跟进`。

### 主动战败

- 显示脱敏手机号。
- 不允许查看完整手机号。
- 不允许复制完整手机号。
- 原操作位置显示 `已主动战败，不可跟进`。

### 已核销

- 显示脱敏手机号。
- 不允许查看完整手机号。
- 不允许复制完整手机号。
- 原操作位置显示 `订单已完成`。

### 已退款

- 显示脱敏手机号。
- 不允许查看完整手机号。
- 不允许复制完整手机号。
- 原操作位置显示 `不可跟进`。

不复制脱敏手机号，避免店端误以为已复制可联系号码。

## 线索跟进详情面板

详情面板标题为 `线索跟进详情`。面板合并查看、登记和历史记录能力。

### 面板布局决策

开发实现以 `方案B反向结构` 为准：

- 面板居中悬浮在当前线索列表之上，背景列表弱化处理。
- 主体采用左右两栏布局。
- 左侧为信息理解区，宽度更大，包含顶部摘要和 `跟进历史` 时间轴。
- 右侧为跟进工作区，宽度更窄，固定展示联系方式操作和 `跟进操作` 表单。
- 店员打开面板后的阅读顺序应是：先在左侧确认线索状态、订单和历史，再在右侧完成查看号码、复制号码、登记跟进。
- 右侧操作区在视觉上要稳定、明确，但不应压过左侧历史信息；它是当前轮次的工作台。

这个布局对应已经确认的视觉方向：左边看清这条线索是什么、经历过什么；右边马上处理。后续开发和验收均按该结构判断，不再采用原始左操作、右信息的方案B。

### 顶部摘要区

顶部摘要区只显示快速判断所需字段：

- `线索状态`
- `联系方式`
- `订单 ID`
- `商品类型`
- `线索生成时间`
- `分配轮次`

顶部摘要区不完整展示商品名称，避免长商品名称挤压核心信息。完整商品名称可放在下方商品信息区或详情次级区域。

### 当前跟进操作区

当前跟进操作区只在当前分配轮次仍有效时展示可编辑组件。

可编辑组件包括：

- `跟进结果` 单选框。
- `本次跟进结论/备注` 输入框。
- `保存` 按钮。

跟进结果枚举：

- `未接通`
- `线索战败`
- `跟进成功`

`未接通` 算产生跟进行为，但不终止当前轮次。如果超过 SLA 后仍未转为 `跟进成功` 或 `线索战败`，该轮次进入 `超期失效`。

`线索战败` 表示门店主动判断无法继续推进。保存后当前轮次立即失效，不可再编辑，不可查看或复制完整手机号。

`跟进成功` 表示门店已成功推进客户。后续等待订单核销结果。

### 只读状态

不可操作状态下，当前跟进操作区替换为原因说明：

- `超期失效`：显示 `已失效不可跟进`。
- `主动战败`：显示 `线索已战败，不可继续跟进`。
- `已核销`：显示 `订单已完成`。
- `已退款`：显示 `订单已退款，不可跟进`。

已失效轮次不能编辑跟进状态。

### 跟进历史区

跟进历史区标题为 `跟进历史`。

每条记录显示：

- `分配轮次`
- `线索状态`
- `本轮跟进时间`
- `跟进结果`
- `本次跟进结论/备注`
- `本轮失效时间`
- `失效原因`

店端不显示具体其他门店名称。需要表达归属时：

- 当前门店自己的记录显示 `本店`。
- 其他门店历史显示 `其他门店`。
- 管理后台可显示具体门店名称、门店 ID、分配轮次 ID、后台订单状态和后台轮次状态。

## 操作权限矩阵

| 线索状态 | 查看完整手机号 | 复制完整手机号 | 编辑跟进结果 | 操作文案 |
| --- | --- | --- | --- | --- |
| 待跟进 | 是 | 是 | 是 | 登记跟进 |
| 已跟进 | 是 | 是 | 是 | 更新跟进 |
| 超期失效 | 否 | 否 | 否 | 已失效不可跟进 |
| 主动战败 | 否 | 否 | 否 | 已主动战败，不可跟进 |
| 已核销 | 否 | 否 | 否 | 订单已完成 |
| 已退款 | 否 | 否 | 否 | 不可跟进 |

## 后续实现提示

- API 可以继续返回后台字段，但店端页面优先使用 `store_display_status` 做展示和权限控制。
- 前端明细和详情都应基于相同的联系方式权限判断，避免列表可复制但详情不可复制的状态不一致。
- `未接通` 后的超期失效由 SLA 规则计算，不由店端手动选择。
- `线索战败` 保存后应立即让当前轮次进入不可操作状态。
- 内部排查字段可放在管理后台或详情底部折叠区，不进入店端主视觉。
