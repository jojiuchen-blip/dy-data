# DYDATA-23 门店结算看板视觉规范

日期：2026-07-15
状态：已确认，待正式功能实现
关联：DYDATA-19、DYDATA-23、DYDATA-24、DYDATA-4
视觉样例：`docs/commission-dashboard-navigation-mock.html`

## 1. 目标

为门店账号定义一套可评审、可实现的结算看板信息架构与 V0.2 视觉规范。当前阶段只确认门店可见页面、筛选、指标、费用明细跳转与开票流程展示，不替代后续真实数据、权限和财务流程设计。

## 2. 页面信息架构

门店登录后默认进入“全国门店榜单”。门店结算区域使用四个同级稳定页面：

1. 全国门店榜单
2. 单店分账
3. 订单费用明细
4. 开票确认

不设置父级“数据看板”入口。四个页面使用 V0.2 `TertiaryNav` 样式：真实链接、`aria-current="page"`、深橙文字与 2px 品牌橙下划线；移动端允许导航容器横向滚动。

## 3. 全国门店榜单

### 筛选

- 日期范围：默认当前月，格式为 `YYYY/MM`；可选择历史月份或“全部累计数据”。
- 产品范围。
- 商品类型。
- 门店搜索。
- 排名依据：销售订单数量、销售总金额、核销收入、厂端激励推广费。

### 汇总指标

以下四项均保留等宽的“当月 / 累计”位置：

- 销售订单数量。
- 销售总金额。
- 核销收入。
- 厂端激励推广费。

正式累计从 2026-08 账期开始。2026-07 是测试版，不展示伪造累计值，只显示未启用状态。

### 跨店权限

全国榜单可展示其他门店的汇总排名，但不能进入其他门店的分账和订单明细。当前所属门店可以进入单店分账。

## 4. 单店分账

真实环境按门店账号隔离，不显示额外的“当前门店 / 仅可查看本门店”说明栏。

筛选包含月份、门店、产品范围和商品类型，不显示账单版本。

保留五项指标：

1. 销售总金额。
2. 本店卖出订单已核销金额。
3. 应收推广费。
4. 本店核销金额。
5. 应扣管理服务费。

费用表说明：

- 应收推广费：`本店通过抖音直播、短视频等卖出的订单，在当月已被核销`
- 应扣管理服务费：`本店在当月核销所有抖音渠道的订单`

费用表末列表头使用“点击跳转”。“查看订单”直接进入订单费用明细的筛选器和表格，不落在页面顶部。

## 5. 订单费用明细

页面不显示从单店分账带入的重复上下文汇总，也不使用“费用方向”下拉筛选。

在页面内使用两个费用明细切换按钮：

- 推广费订单明细。
- 管理服务费订单明细。

从单店分账跳转时携带月份、门店、商品范围/类型、费用方向、配置比例和账单版本等可追溯上下文；视觉层只展示当前基础版本需要的筛选与订单表。

## 6. 开票确认

开票确认为独立同级页面。当前标题状态统一为“当前功能暂未开放”。

基础版本只展示五个流程节点：

1. 月度账单待确认。
2. 门店确认账单。
3. 门店开具推广费发票。
4. 发票提交财务审核。
5. 审核完成并进入结算。

## 7. 不做

- 不实现 2026-07 累计数据。
- 不定义累计口径的后端计算、历史回填、退款调整或跨月去重。
- 不新增生产 API、数据库迁移或真实门店权限。
- 不展开发票字段、号码/税额校验、红冲、替换或重开。
- 不提供财务审核操作、驳回原因、结算导入、财务角色看板或全量开票台账。
- 不在本轮改造现有生产 React 路由；生产实现需在数据契约与权限方案确认后单独进入开发。

## 8. 响应式与可访问性

- 390px、768px、1440px 视口不得出现页面级横向溢出。
- 宽表使用自身滚动容器。
- 每页只能有一个 `h1`。
- 表单控件必须有可访问标签。
- 页面链接必须支持刷新、深链和浏览器历史。
- 移动端交互目标最小高度为 44px。
