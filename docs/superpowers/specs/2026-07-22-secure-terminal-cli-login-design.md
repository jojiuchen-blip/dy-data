# dydata 安全终端登录设计修订

- 日期：2026-07-22
- 状态：已确认并获准实施
- 需求来源：Linear `DYDATA-40`
- 修订对象：`docs/superpowers/specs/2026-07-21-agent-first-read-only-cli-design.md` §7.1、§10.1
- 不变范围：CLI 仍为严格只读；业务命令、门店权限和线索指标口径不变

## 1. 目标

让 Agent 可以在自身带有安全交互 TTY 的命令行工具中启动登录命令，再把输入权交给用户。用户亲自输入账号和密码；密码由终端隐藏输入，不进入命令参数、环境变量、配置文件、标准输出、日志或 Agent 上下文。

默认入口调整为：

```powershell
dydata auth login
```

原有浏览器设备授权保留为显式回退：

```powershell
dydata auth login --browser
```

## 2. 核心决策

| 决策项 | 结论 |
| --- | --- |
| 默认登录方式 | 本地交互式 TTY 输入账号和隐藏密码 |
| Agent 边界 | Agent 可以在用户明确要求后启动命令，但不得代填、索取、读取或转述凭据 |
| 密码输入 | 使用终端隐藏输入；不接受参数、stdin 管道、环境变量或配置文件 |
| 服务端链路 | 复用现有 `/auth/login` Web 登录和 `/auth/cli/device/approve` 设备授权，不新增密码换 Token 接口 |
| 令牌范围 | 仍为 `cli:read`，业务端点仍由服务端白名单限制为只读 |
| 身份确认 | 登录成功后显示服务端返回的用户名、角色和门店范围，用户确认后才批准设备授权 |
| 旧凭据保护 | 检测到本机已有 CLI 凭据时不覆盖；切换账号必须先执行 `dydata auth logout` |
| 回退路径 | 不具备安全交互 TTY 时，明确失败并提示改用 `--browser` |
| 自动重试 | 用户名/密码登录、设备批准和其他认证 POST 均不自动重试 |

## 3. 终端登录时序

```text
Agent / 用户
    │ 启动 dydata auth login
    ▼
CLI 检查：没有已有凭据 + 当前输入输出是安全交互 TTY
    │
    ├─ Username: 普通终端输入
    ├─ Password: 隐藏终端输入
    │     └─ 无法关闭回显时硬失败，不允许 getpass 回退明文输入
    │
    ├─ POST /auth/login（独立、短生命周期的内存 HTTP 会话）
    │     └─ 服务端返回现有 Web Session Cookie 和身份元数据
    │
    ├─ CLI 显示规范化身份：username / role / store scope
    ├─ 用户确认 y/N
    │
    ├─ POST /auth/cli/device/start（确认后才创建匿名、短期授权请求）
    ├─ POST /auth/cli/device/approve（携带刚取得的 Web Cookie）
    ├─ 清除 Cookie 引用并关闭临时 HTTP 会话
    │
    ├─ POST /auth/cli/device/token（一次性设备码换取 cli:read 凭据）
    └─ 以 compare-and-swap 写入操作系统凭据库
```

密码只在调用 `/auth/login` 所需的最小局部作用域内存在。Python 字符串无法承诺物理内存擦除，因此实现承诺是“不持久化、不记录、不回显、不复用，并尽快释放引用”，而不是不可验证的内存清零承诺。

## 4. 信任与隔离边界

### 4.1 Agent 与人类输入边界

- Agent 只能启动命令和解释下一步，不得让用户把账号密码粘贴进对话。
- 密码必须由用户在隐藏输入提示处直接键入。
- 如果 Agent 的命令工具不能把真实交互 TTY 交给用户，Agent 必须停止终端登录并建议 `dydata auth login --browser`。
- `agent_callable` 继续为 `false`：它不是 Agent 可自主完成的命令；文档会单独说明 Agent 可在获得用户指示后“启动并交接”。

### 4.2 HTTP 会话边界

- 终端密码登录使用独立 `httpx.Client`，不把 Web Cookie放入业务查询客户端。
- 临时客户端只暴露“登录”和“批准设备码”两个远端动作。
- 无论成功、取消还是异常，都必须清 Cookie 并关闭客户端。
- 服务端响应采用字段白名单重建，额外字段、令牌字段或结构漂移一律视为 `SCHEMA_MISMATCH`。

### 4.3 凭据写入边界

- 登录开始前检查现有 OS keyring 状态；存在时返回无覆盖提示。
- 最终保存使用 `expected=None` 的 compare-and-swap，防止两个登录进程竞态覆盖。
- 若并发进程先写入凭据，本次新签发刷新凭据应尽力撤销，并保留先写入的状态。

## 5. 命令与输出契约

`auth.login` 新增参数：

| 参数 | 默认值 | 语义 |
| --- | --- | --- |
| `--browser` | `false` | 使用原有浏览器设备授权；不提供时使用安全终端登录 |

终端成功输出只包含非秘密身份摘要和完成提示；不会打印密码、Cookie、设备码、访问令牌或刷新令牌。浏览器模式可以继续显示验证 URL 和一次性用户码，但不得显示内部 `device_code`。

检测到已有凭据时，命令作为安全幂等 no-op 返回，不发起网络请求，并提示先登出才能切换账号。终端模式在非 TTY 环境下使用稳定参数错误退出；无效凭据使用稳定认证失败错误退出。隐藏输入失败、无效凭据或取消确认时不会创建 device grant、批准设备码或保存凭据。

## 6. 安全负向要求

- `dydata auth login --username ...`、`--password ...`、`--token ...` 必须解析失败，且错误输出不得回显参数值。
- 管道输入和重定向 stdin 不得绕过 TTY 检查。
- 密码不得出现在异常字符串、对象 `repr`、测试失败输出或生成文档示例中。
- 登录响应中的意外字段、秘密字段、非预期角色或非法门店范围必须拒绝。
- 401、429、5xx 和网络异常只能映射到本地稳定错误，不转发服务端 `detail`。
- 认证 POST 请求保持单次提交，不因网络断开自动重放。
- 该变更不扩大 `/auth/login` 的服务端能力；登录防爆破/限流属于该既有 Web 入口的共享平台控制，不在 CLI 内以客户端重试替代。

## 7. 验收标准

1. 默认命令在真实 TTY 中提示用户名、隐藏提示密码、展示身份摘要并要求确认。
2. 确认后复用现有 Web Cookie 批准设备授权，最终只把 `cli:read` 凭据写入 OS keyring。
3. `--browser` 仍能执行原有设备授权流程。
4. 非 TTY、无效凭据、用户取消、响应漂移和并发保存均不泄露秘密或覆盖已有凭据。
5. `dydata commands --json`、Agent 指南、生成命令参考和 README 同步说明两种登录模式与人机边界。
6. 既有门店范围、线索统计、API 授权和全量测试继续通过。
