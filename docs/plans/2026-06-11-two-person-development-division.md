# 两人协作开发分工 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 第一阶段由两名开发者在现有 Python 脚本、CSV/JSON 中间文件和静态 HTML 看板基础上，交付页面 1、页面 2、页面 3 及其数据口径、下钻、导出和复核能力。

**Architecture:** 当前不先重构成完整 Web 服务，继续沿用本地数据采集脚本生成分账基础表，再由静态看板读取汇总数据和明细数据。开发者 A 负责数据采集、基础表、计算口径和校验；开发者 B 负责页面结构、筛选交互、提示口径、明细表和导出体验。两人以“分账基础表字段契约”和“页面汇总 JSON/CSV 输出”为接口，避免同时改同一批核心文件。

**Tech Stack:** Python, CSV/JSON, static HTML/CSS/JavaScript, PowerShell, Markdown.

## 1. 项目边界

第一阶段交付的是“抖音订单分账数据看板”，不是正式财务结算系统。

必须交付：

- 页面 1：全国门店销售情况榜单，默认展示前 20 门店，支持全部产品/特定服务产品筛选。
- 页面 2：单店月度分账看板，展示当期应收分佣、可分佣总金额、本店预计分出分佣参考额，以及应收、应付、不参与分佣三块表。
- 页面 3：门店月度数据明细表，默认呈现后端完整分账基础表，支持筛选、排序、列显隐、从页面 2 带条件下钻和导出；页面 3 不直接展示页面 2 的透视关系字段，而是展示销售归属门店、实际核销门店、是否核销、是否分佣等底层字段。
- 页面 1 和页面 2 底部必须列出本页全部展示数据的计算口径。
- 核心数字右上角必须有提示 icon，点击或悬停展示数据口径。

第一阶段明确不做：

- 不做角色权限区分，能访问系统的人都能看全部数据。
- 不做真实交易、核销、退款、付款、打款或资金划拨。
- 不做发票上传、OCR、发票审核流程；页面只读取财务已维护或导入的到票时间、发票审核状态、审核通过时间。
- 不把飞书多维表格作为主数据库；它只可作为人工校对、异常处理和财务确认的协作界面。

## 2. 人员分工

| 角色 | 建议负责人 | 主责 | 不主责 |
| --- | --- | --- | --- |
| 开发者 A | 数据/后端计算负责人 | 数据采集、字段契约、分账基础表、页面汇总数据、异常校验、导出数据准确性 | 页面样式、交互细节、视觉布局 |
| 开发者 B | 前端/看板负责人 | 页面 1/2/3 静态看板、筛选、下钻、提示 icon、口径展示、列显隐、导出入口、产品文档同步 | 抖音接口采集、底层数据匹配、金额口径计算 |

协作原则：

- A 先稳定字段和汇总输出，B 可以先用 A 提供的样例 JSON/CSV 开发页面。
- B 不直接改 A 的采集和计算脚本；如发现字段缺失，通过字段契约提需求。
- A 不直接改 B 的页面布局；如发现口径展示不准确，通过页面指标清单提需求。
- 两人都可以改文档，但同一时间只由一人改 `docs/项目产品介绍书.md`，避免文档冲突。

## 4. 字段契约

A 需要保证分账基础表至少提供以下字段，B 只能基于这些字段做展示、筛选、排序和导出。

| 字段组 | 必要字段 |
| --- | --- |
| 订单信息 | 订单 ID、下单时间、销售月份、商品类型、SKU ID、SKU 名称、订单实收金额、订单状态 |
| 券与核销 | 券 ID、券状态、是否核销、核销时间、核销月份、核销状态、核销类型 |
| 门店归属 | 销售归属门店、销售归属门店 poi_id、实际核销门店、实际核销门店 poi_id、认证主体 |
| 抖音号/职人 | 抖音号、职人、绑定门店、绑定状态 |
| 分佣信息 | 分佣比例、可分佣总金额、当期应收分佣、应付分佣、是否分佣 |
| 发票/到票 | 到票状态、到票时间、发票审核状态、审核通过时间 |
| 退款/复核 | 退款状态、退款金额、退款完成时间、异常状态、异常原因、复核备注 |

关键枚举必须统一：

- 是否核销：`是`、`否`
- 是否分佣：`是`、`否`、`不适用`
- 到票状态：`未到票`、`已到票`、`审核通过`、`审核未通过`
- 退款状态：`未退款`、`退款中`、`已退款`
- 异常状态：`正常`、`待复核`、`异常`

页面 2 可以使用“本店卖出、他店核销”“他店卖出、本店核销”“本店卖出、本店核销”等透视关系来组织展示，但页面 3 不把这些关系作为直接展示字段或下钻参数。页面 3 只接收底层筛选条件，例如当前门店、销售归属门店、实际核销门店、是否核销、是否分佣、到票状态和月份归属。

## 8. Git 协作建议

- A 使用分支：`feature/settlement-data-contract`
- B 使用分支：`feature/settlement-dashboard-pages`
- 合并顺序：先合并 A 的字段契约和基础表，再合并 B 的页面实现。
- 每次合并前运行：

```powershell
git status --short --branch
git diff --check
python validate_settlement_data_availability.py
python build_monthly_settlement_dashboard_from_base.py
```

如果 B 需要新增字段，先在字段契约中提需求，不直接在页面代码里写临时推断逻辑。如果 A 需要调整字段名，必须先更新契约并通知 B 同步。

## 9. 第一轮建议执行顺序

1. A 完成 `docs/分账基础表字段契约.md`，把当前已有字段和缺失字段标出来。

## 10. 完成标准

第一阶段完成时，必须同时满足：

- 页面 1 的每个榜单数字都能在页面底部找到计算口径。
- 页面 2 的三个核心数字都能通过 tooltip 看到口径，并且能下钻到页面 3。
- 页面 3 可以解释页面 1 和页面 2 的汇总数字来源，且页面 3 使用底层字段表达筛选条件，不直接展示页面 2 的透视关系字段。
- 当前筛选条件下的明细可以导出。
- 发票相关只读取已导入状态和时间，不出现上传、识别、审核功能入口。
- `validate_settlement_data_availability.py` 能提示页面必需字段是否缺失。
- `docs/项目产品介绍书.md`、字段契约、页面验收清单三者口径一致。
