## 1. 项目背景与机会判断

dy-data 是汽车经销商集团抖音经营业务的数字化承接平台，也是抖音分账新业务能够成立和持续运行所依赖的基础设施。项目服务的不是单一结算页面，而是由经营与结算、线索运营、经营数据和后台管理共同组成的业务体系。

财务侧面对订单、核销、跨店关系、分佣规则和门店范围形成的组合复杂度。该业务并不存在一套能够被简单线上化的成熟人工流程；当数据达到实际规模后，脱离系统几乎无法稳定完成分账计算和复核。线索侧则缺少统一分配、规范跟进和可追溯留痕，导致服务行为容易重复、遗漏或延迟，管理人员也难以判断服务交付是否真实发生。

当前系统已经覆盖结算、线索、经营数据和后台管理的部分核心流程，能够缓解部分分账计算、数据查看、线索分配和跟进留痕问题。因此，本期建设是在现有系统上补齐业务闭环、加强规则和权限治理并完成业务验收，不进行推倒重建。

本项目提供经营分析、线索运营、后台配置和分账参考，不执行真实资金划拨。代码、路由、接口、模型或页面存在，只能证明能力已经进入实现层，不能自动视为业务口径正确或已经完成用户验收。

需求判断主要来自对业务流程的分析，并由亲自观察和使用者直接反馈提供补充证据。系统对线索流程的直接承诺是分配、跟进和留痕可执行、可查询、可追溯；核销率、服务质量和服务交付比例的提升属于预期业务结果，需要在系统运行后继续用经营数据验证。

## 2. 效率目标

### 2.1 目标维度

本期以“业务可运行性与流程留痕完整性”为核心验收维度，不用页面数量或功能入口数量代替业务结果。

| 业务域 | 当前基线 | 2026 年 7 月 22 日目标 | 验收口径 |
|---|---|---|---|
| 财务分账 | 没有系统时，新分账业务无法在实际数据规模和规则复杂度下稳定运行 | 全部当前已知且持续发生的标准分账场景由系统支持 | 对标准分账场景逐项验证计算、复核、来源追溯与结果留痕；仅源数据缺失、归属争议或首次出现的新规则允许线下处理 |
| 线索运营 | 分配与跟进记录分散，过程和完成情况不能可靠追溯 | 所有权限内合法跟进均能成功保存、查询和追溯 | 每条记录能够关联订单或线索、分配轮次、门店、操作人和时间；静默丢失为 0 |
| 业务验收 | 当前代码存在不等于业务场景完成验收 | 除 CLI / Agent 外的本期能力完成技术验证、生产部署与对应角色验收 | 分别由财务、运营、客服和管理角色按业务域确认，不以单一技术测试替代业务验收 |

联系结果、首次联系时长和核销转化率不作为本轮基础流程是否跑通的硬门槛。这些指标用于后续评估服务策略的实际效果，不反向掩盖记录无法保存、过程无法追溯等基础缺陷。

## 3. 内部用户角色与当前工作流

### 3.1 利益相关角色

| 角色 | 与需求的关系 | 核心诉求与痛点 | 利益冲突情况 |
|---|---|---|---|
| 系统开发统筹负责人 | 需求提出者和业务协调者，不是主要日常使用者 | 统一业务口径、协调需求并保证系统能够按整体目标交付 | 当前未发现实质冲突 |
| 集团财务 | 直接使用者 | 需要系统化计算和复核订单、核销、跨店关系、扣减与结算结果，并获得可追溯依据 | 与门店财务关注粒度不同，但当前无实质冲突 |
| 经销商财务 | 直接使用者 | 需要清晰、可复核的本店结算依据，并能识别金额差异和责任归属 | 当前无实质冲突 |
| 经销商抖音负责人 | 直接使用者 | 需要统一查看销售、商品、核销和线索表现，并识别需要跟进的异常 | 当前无实质冲突 |
| 经销商客服 | 直接使用者 | 需要接收分配线索、联系客户并完整记录跟进，避免重复、遗漏和延迟 | 当前无实质冲突 |
| 经销商侧多店管理人员 | 管理使用者 | 需要在授权的多个门店范围内横向比较经营和服务交付情况 | 与其他管理层的差异是数据范围和汇总粒度 |
| 区域管理人员 | 管理使用者 | 需要查看所辖片区门店、下钻异常并督促改进 | 与其他管理层的差异是数据范围和汇总粒度 |
| 总部管理人员 | 管理使用者 | 需要查看全国经营表现、服务交付和异常分布 | 与其他管理层的差异是数据范围和汇总粒度 |
| 授权后台管理员 | 配置和运行维护者 | 需要维护账号权限、业务规则、商品可见性、线索配置和数据运行，并定位异常影响范围 | 当前无实质冲突 |
| 购买服务的客户 | 被服务业务对象，不是系统直接使用者 | 需要获得及时、规范的服务联系与后续交付 | 不参与系统权限和页面操作 |

### 3.2 当前工作流

抖音订单、核销、销售和线索数据进入系统后，与门店、商品及业务规则建立关联。财务人员依据订单和券明细复核销售归属、核销门店、跨店关系及结算参考结果；线索经规则分配到对应门店，由抖音负责人或客服联系客户并记录跟进情况；管理人员按单店、多店、区域或全国权限查看经营表现、服务交付和异常；后台管理员维护账号权限、SKU、归属规则、商品可见性、线索分配配置和数据同步任务，并处理运行异常。

系统当前只提供经营和分账依据，不执行真实资金动作。对账、开票信息获取和 CLI / Agent 接入属于目标能力，不计入当前已经运行的流程。

## 4. 核心价值模型

项目由两个并列的业务闭环构成：准确分佣和有效服务交付。

| 价值闭环 | 输入 | 系统形成的能力 | 直接业务结果 | 需要持续验证的结果 |
|---|---|---|---|---|
| 准确分佣 | 订单、核销、跨店关系、商品和分佣规则 | 统一计算、来源追溯、结果复核、异常识别和规则留痕 | 分账新业务能够稳定运行，集团与门店获得可复核依据 | 不同规则和异常场景下的持续准确性与处理成本 |
| 有效服务交付 | 线索、门店、分配规则和跟进记录 | 统一分配、行为约束、关键节点留痕和分层可见 | 重复、遗漏、延迟和过程不可见问题得到直接控制 | 核销率、服务质量和服务交付比例的实际提升幅度 |

经营数据为两个闭环提供横向比较、异常下钻和管理反馈；后台管理为两个闭环提供账号权限、业务规则、商品可见性、线索配置和运行保障。二者是支撑能力，不作为独立首要价值。

该价值模型不承诺系统上线必然提高核销转化率。系统首先保证业务任务可执行、过程可追溯和结果可复核，再通过后续经营数据验证服务策略是否真正带来转化改善。

## 5. MVP 范围

### 5.1 In Scope

| 业务域 | 范围定义 | 当前与目标能力边界 |
|---|---|---|
| 经营与结算 | 销售看板、门店排名、月度结算、订单明细、跨店核销识别、应收应扣净额参考和异常复核 | 在现有能力上补齐标准场景并完成业务验收；不执行资金划拨 |
| 线索运营 | 线索总览与详情、跟进记录、分配轮次、规则版本、试运行、总部线索池和门店评分 | 保证合法跟进可保存、查询、追溯，管理过程可见 |
| 后台管理 | 账号与门店权限、SKU 规则、非佣金归属账号、商品类型可见性、反馈、同步和线索分配配置 | 配置变更需要边界清晰、影响可追溯、异常可定位 |
| 数据平台与生产运行 | 抖音数据采集、API、数据库、Web、worker、浏览器导出、迁移、部署和运行可靠性 | 作为业务数据和生产运行底座，不作为独立用户端 |
| 专业财务扩展 | 对账、开票信息获取，以及支撑专业财务板块所需的业务概念和治理能力 | 属于待建设目标域；当前分佣估算能力不能被表述为已经形成专业财务闭环 |
| CLI / Agent 只读接入 | 所有授权用户通过系统账号登录，在自身角色、组织和数据权限内查询、筛选、汇总和导出数据 | 纳入总范围但独立排期；无论 Web 账号是否有写权限，该渠道始终严格只读 |

### 5.2 Out Scope

| 范围 | 排除理由 |
|---|---|
| 真实资金划拨或支付执行 | 本系统提供经营与分账依据，不承担资金执行责任 |
| CLI / Agent 的新增、修改、分配、提交、审批、同步触发或其他状态变更 | 当前只读边界已经锁定；写入能力需要重新评估风险和授权模型 |
| 面向购买服务客户的直接使用端 | 客户是被服务对象，不是本系统直接使用者 |
| 把核销率提升作为基础流程跑通的硬门槛 | 转化结果受服务策略、客户意愿等多因素影响，不能替代流程完整性验收 |

## 6. 备选方案对比

| 方案 | 描述 | 优势 | 劣势 | 取舍理由 |
|---|---|---|---|---|
| 在现有系统上补齐闭环 | 保留已经运行的结算、线索、数据和后台能力，补齐业务缺口、治理能力和验收证据 | 复用现有数据、代码和用户实践，增量价值清晰，迁移风险较低 | 需要严格识别历史实现中的过期口径和隐性缺陷 | 采用。当前系统已经部分解决核心问题，本期目标是补缺口而不是重建 |
| 推倒重建 | 重新设计全部业务域和技术实现 | 可以从统一模型重新开始 | 成本和迁移风险高，容易丢失已验证实践，也不符合 2026 年 7 月 22 日目标 | 不采用。没有证据证明重建比增量治理更能解决业务问题 |
| 继续依赖人工和分散工具 | 以表格、人工核对、逐店沟通和分散记录维持业务 | 初期系统投入较低 | 分账无法在实际规模下稳定运行，线索过程无法可靠追溯，管理成本持续增加 | 不采用。不能承接新分账业务和统一服务交付管理 |

| 前提 | 状态 | 判断依据与后续处理 |

| 如果不继续建设，本期业务会受到实质影响 | 已验证 | 财务分账将退回高风险人工核对且难以持续；线索分配、服务交付和核销结果将继续缺少统一追溯 |
| 现有系统只覆盖了部分问题，本期仍有明确增量价值 | 已验证 | 当前已经存在结算、线索、经营数据和后台管理能力；专业财务闭环、规则治理和完整业务验收仍需补齐，CLI / Agent 另行排期 |
| 统一数据、规则、留痕与分层管理能够直接缓解核心痛点 | 已验证但保留效果边界 | 现有部分实践已经验证流程可运行性；系统能够保证任务可执行和记录可追溯，但核销率提升仍需上线后用数据验证 |

## 8. 关键风险与兜底策略

| 风险 | 影响 | 兜底动作 |
|---|---|---|
| 源数据缺失、延迟或口径不一致 | 经营与结算结果失真 | 数据校验、异常标记、来源追溯和人工复核；异常结果不得作为最终依据 |
| 财务规则配置错误或变更失控 | 分佣、对账和结算结果错误 | 规则版本化、试运行、变更留痕和关键结果人工确认 |
| 多店、区域和全国权限边界错误 | 用户越权访问非授权业务数据 | 统一鉴权、服务端数据范围过滤、默认拒绝、权限测试和访问审计；全国门店排行榜前 20 名按明确业务例外处理 |
| CLI 只读边界失效 | 数据或业务状态被非预期修改 | 只读命令白名单、最小权限凭证、写接口阻断和全量调用审计 |
| 线索记录不完整或失真 | 管理指标不能代表真实服务交付 | 关键节点留痕、状态约束、异常抽查和可追溯修正 |
| 完整目标范围过大或把代码存在误判为验收通过 | 交付失控，业务缺口被掩盖 | 按业务域分阶段验收，每项能力独立验证；代码存在不等于业务验收通过 |

## 10. 页面定位

### 10.1 项目覆盖对象

项目覆盖两个用户接入端：Web 业务端和 CLI / Agent 严格只读接入端。不建设面向购买服务客户的直接使用端。

单店、多店、区域和全国是数据权限范围，不是独立端；结算、线索、数据表现和后台管理是业务模块，不是独立端；API、数据库和 worker 是技术组件，不是用户端。

### 10.2 页面与入口定位

| 页面或入口 | 目标使用者 | 主要用途 | 定位 |
|---|---|---|---|
| Web 查看型页面 | 财务、抖音负责人、多店/区域/总部管理人员及其他授权用户 | 查看经营看板、门店排名、结算结果、订单明细、线索表现和分层管理汇总 | 查看 |
| Web 操作型页面 | 财务、抖音负责人、客服及授权运行人员 | 财务复核与异常处理、对账处理、线索分配与跟进、反馈和数据同步运行 | 操作 |
| Web 配置型页面 | 授权后台管理员 | 维护账号与门店权限、SKU 与归属规则、商品可见性和线索分配规则 | 配置 |
| CLI / Agent 只读入口 | 所有已授权系统用户 | 在账号权限范围内查询、筛选、汇总和导出业务数据 | 非页面只读渠道，不适用页面定位分类 |

单个 Web 页面可以包含辅助查看或辅助操作，但主定位必须由用户进入页面时的核心任务决定，不能仅根据是否存在按钮判断。

### 10.3 权限与业务例外

Web 和 CLI / Agent 默认按账号、组织和数据权限范围控制业务数据。全国门店排行榜向已授权 Web 用户展示前 20 名属于明确的横向对标业务例外，不判定为越权；该例外不扩展为对其他全国明细或汇总数据的默认访问权。

CLI / Agent 使用同一系统账号登录并继承角色、组织范围与数据权限，但渠道能力始终严格只读。即使账号在 Web 端拥有写权限，也不得通过 CLI / Agent 改变业务状态。

### 10.4 下游待确认边界

本 BRD 锁定业务目标、范围、角色、成功标准，以及权限、财务、线索状态、当前能力与目标能力区分等不可由下游自行推断的跨模块业务不变量。

页面信息架构、任务路径、字段结构、数据模型、接口协议、状态校验、异常处理细节、数值阈值和具体文案由后续 page-designer、foundation-builder 和 prd-writer 在本 BRD 约束内确定，不得创造新的业务语义或绕过权限边界。

移动端规范仍是待推进需求，本阶段不据此修改现有实现。设计系统仅以标记为 `runtime-active` 的视觉语言和复用交互契约形成当前强约束；`future` 标记与实际运行状态冲突时，应先补齐或修订规范，再作为下游设计依据。
