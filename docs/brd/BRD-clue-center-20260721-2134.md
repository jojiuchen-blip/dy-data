# BRD: 线索中心业务模型 V1.0

> 项目类型: 运营型
> 生成时间: 2026-07-21 21:31
> Skill: brd-writer

## 1. 项目背景与机会判断

### 1.1 背景

`dy-data` 已具备抖音线索采集、线索主池、分配轮次、门店跟进、总部池、规则配置和业务页面等能力，但历史规格、Linear 需求、当前代码和试跑验收之间存在口径分散及局部冲突。相同概念曾被不同材料解释为不同含义，主要集中在线索本体、订单终态、分配轮次、总部池、跟进动作、SLA、保护期、手机号权限、经营指标以及旧新分配引擎边界。

这种分散已经不只是文档问题。它会直接造成以下业务风险：

- 门店收到重复、遗漏或已经失效的客户任务，甚至在失去当前归属后继续联系客户。
- 总部无法准确判断线索库存、当前责任门店、跟进质量和经营结果。
- 开发与测试使用不同口径判断完成度，导致局部功能存在但端到端闭环仍未成立。
- 新旧分配逻辑同时保留时，同一订单可能生成冲突轮次、错误权限或污染指标。

### 1.2 本文档的权威范围

本 BRD 是线索中心当前业务语义的单一权威源，负责冻结业务对象、生命周期、池位置、分配与再分配规则、跟进动作、时限、权限原则、经营指标和页面职责。最终审核与业务解释权归总部业务侧。

本文档不直接规定数据库表、接口字段、任务调度、页面控件或技术实现。PAGE_EXPLAINER、FOUNDATION、PRD、开发计划和测试方案必须引用本 BRD；若下游发现需要改变业务语义，必须返回总部业务侧评审并形成新版本，不能静默改口径。

### 1.3 机会判断

本项目的机会不是把抖音现有分配方式搬进内部系统，而是建立可持续优化的自有线索经营闭环：系统按门店能力和服务资格择优分配，总部能够观察完整库存与结果，优秀门店获得更多优质线索，一般门店被统一的时限和动作规范约束，最终以总体核销率是否提升判断业务价值。

当前尚无真实门店运营样本证明新规则一定提升核销率。因此，本 BRD 将方案定义为可检验的业务假设，而不是已经成立的效果结论。

## 2. 效率目标

### 2.1 北极星指标

北极星指标为 **总体核销率**，统计粒度为唯一 `order_id`。

| 项目 | 统一口径 |
|------|----------|
| 分母 | 订单下单时间所属自然月内，进入完整线索主池的全部唯一订单；包括首次采集时已经核销、已经退款，以及后续进入门店跟进池、总部池或关闭状态的订单 |
| 分子 | 分母订单中，历史上曾达到已核销状态的唯一订单 |
| 去重 | 同一订单的多条原始线索、多个分配轮次和多次跟进只计算一次 |
| 月份归属 | 按用户下单时间归入自然月，不按线索采集时间、分配时间或跟进时间归月 |
| 回补 | 订单状态后续变化时允许回补其所属月份；月末满 30 天后标记为“已成熟”，此前标记为“观察中” |

计算公式：

`总体核销率 = 曾达到已核销状态的唯一订单数 / 完整线索主池唯一订单数`

### 2.2 基线与首期目标

- **基线周期**：2026-01-01 至 2026-06-30。
- **基线算法**：六个月全部成熟订单合并后计算一次总体核销率，不取六个月月度百分比的简单平均。
- **过渡月**：正式运营启动所在自然月作为过渡月，不进入上线前后正式比较。
- **首期观察周期**：过渡月之后连续 3 个已成熟完整自然月。
- **首期成功标准**：上述 3 个成熟月合并后的总体核销率高于基线。
- **目标解释**：首期只验证提升方向。完成首个周期后，再用真实运营数据制定下一阶段的提升幅度目标。

若首期未高于基线，应复核策略参数、门店执行质量、数据完整性和样本构成，不得通过缩小分母、排除总部池或删除关闭订单改变口径。

### 2.3 诊断指标

以下指标用于解释北极星指标变化，不替代总体核销率：

| 指标 | 业务定义 | 主要用途 |
|------|----------|----------|
| 完整主池订单数 | 统计期内完整主池的唯一订单数 | 判断业务规模和分母稳定性 |
| 门店跟进池库存 | 当前存在有效门店轮次的唯一线索数 | 判断门店待处理压力 |
| 总部池库存 | 当前处于总部池的唯一线索数 | 判断数据与候选覆盖缺口 |
| 自动分配成功率 | 成功进入门店有效轮次的活跃线索数 / 进入分配判断的活跃线索数 | 诊断锚点、地理、规则和候选质量 |
| 24 小时有效跟进率 | 在当前轮分配后 24 小时内产生任一有效跟进动作的轮次数 / 到达 24 小时观察点的轮次数 | 作为门店综合评分的默认组成项 |
| 门店核销转化能力 | 分配给该门店并进入可比较成熟窗口的唯一订单中，最终核销的比例 | 作为门店综合评分的默认组成项 |
| 保护期到期率 | 保护期届满且仍未核销的轮次数 / 已进入保护期的成熟轮次数 | 诊断跟进质量，而非只看是否点击过跟进 |

门店指标只统计实际分配给门店的轮次和订单。总部池线索不进入任何门店的线索数、跟进率或核销转化能力分母，但进入总部整体线索量和总部池库存。

## 3. 内部用户角色与当前工作流

### 3.1 利益相关角色

| 角色 | 业务职责 | 核心诉求 | 权限边界 |
|------|----------|----------|----------|
| 总部业务侧 | 最终审核业务规则和指标口径 | 统一定义、可比较结果、可追责过程 | 拥有业务解释权；业务口径变更必须由其确认 |
| 最高管理员 | 规则配置、发布、试运行、重建、审计和高风险管理 | 安全地管理规则与运行状态 | 不默认作为门店跟进执行者；高风险操作必须留痕 |
| 普通管理员 | 查看配置与运行结果 | 了解当前状态并协助排查 | 按 DYDATA-32 目标保持只读 |
| 总部运营/总部管理 | 查看全国经营、库存和总部池 | 识别异常、督导执行、评估策略 | 默认不能代替门店跟进，也不能因全国可见而获取完整手机号 |
| 多店/区域管理人员 | 查看授权范围内门店表现 | 督导门店及时、规范跟进 | 默认只读监督，不代替门店登记跟进 |
| 门店客服/抖音负责人 | 处理当前分配给本店的有效线索 | 快速识别可联系任务、联系客户、登记结果 | 仅可操作当前有效轮次；失效后不可查看完整手机号或继续跟进 |
| 客户及其订单/线索 | 被服务业务对象 | 获得及时且不过度打扰的门店服务 | 不是系统直接使用者 |
| 开发、测试、运维 | 实现、验证和运行系统 | 获得稳定、可追踪的验收基线 | 不拥有业务语义修改权 |

角色之间的主要冲突是“总部希望提高总体转化”与“门店可能选择性处理线索”，以及“管理范围可见”与“客户隐私最小授权”。解决原则是按实际结果择优分配、按统一规则记录门店行为，并将完整联系方式严格限制给当前有效轮次所属门店。

### 3.2 核心业务对象与状态

| 对象 | 业务定义 | 关键约束 |
|------|----------|----------|
| 原始线索记录 | 抖音接口返回的原始记录 | 长期保留，可追溯，不因订单终态或去重而删除 |
| 线索主记录 | 系统对订单及其线索生命周期的统一业务视图 | 完整主池保留全部订单；经营统计按唯一 `order_id` 去重 |
| 门店跟进池 | 当前存在有效门店分配轮次的线索集合 | 同一线索任一时刻最多一个当前有效轮次 |
| 总部池 | 当前没有门店责任归属、需要总部治理的活跃线索集合 | 不属于任何门店，不进入门店指标；当前版本不再次投放 |
| 关闭线索 | 已核销或已退款，不再需要门店促核销跟进的线索 | 保留主记录，但不创建或继续分配轮次 |
| 分配策略步骤 | 系统依次执行的固定策略模块 | 步骤可以跳过；策略步骤序号不等于业务轮次号 |
| 分配轮次 | 一次真实分配给具体门店的责任周期 | 只有实际分配成功才创建，`round_no` 连续递增，不创建空轮次 |
| 跟进记录 | 门店在某一有效轮次内登记的一次跟进行为 | 每次动作独立留痕，并同步该轮最新摘要 |

店端统一显示业务化的“线索状态”，后台同时保留订单状态、轮次状态和池位置。三者不得混成一个字段：

- 订单已核销时显示“已核销”。
- 订单已退款时显示“已退款”。
- 当前轮仍有效且无跟进时显示“待跟进”。
- 当前轮仍有效且已有跟进时显示“已跟进”。
- 历史轮因 SLA 到期关闭时显示“超期失效”。
- 历史轮因门店主动放弃关闭时显示“主动战败”。
- 失效轮次仍可查看脱敏手机号和历史，但不可查看、复制完整手机号或继续编辑跟进。

### 3.3 当前权威业务工作流

1. 抖音线索进入完整主池。首次采集时订单已经核销或已经退款，仍保留主记录，但直接关闭且不创建分配轮次。
2. 仍处于可促核销阶段的线索，仅使用 `follow_poi_id` 作为无法取得客户精准位置时的位置代理锚点，不回退使用其他意向 POI 字段。
3. `follow_poi_id` 缺失、无法映射门店，或锚点门店的城市、经纬度不可用时，线索进入总部池。
4. 锚点有效时，系统按作用范围优先级匹配已发布规则：锚点门店专属规则 > 门店组规则 > 城市规则 > 全局规则。线索在首次分配时锁定规则版本，后台后续修改不切换其既有版本。
5. 规则固定沿用三类策略模块：销售店优先、城市半径优选、城市兜底。模块类型固定，但允许启停、排序和配置参数；10 公里和 15 公里只是默认参数，不得硬编码为不可调整常量。
6. 系统按启用模块顺序执行。模块无候选时只记录跳过原因，不创建空轮次；只有实际选中门店才创建连续编号的业务轮次。新轮次使用创建当时有效的评分证据，历史轮次、评分快照和决策依据不可改写。
7. 门店只可在当前有效轮次执行跟进。订单核销或退款优先于 SLA、保护期和跟进动作，立即结束线索并关闭当前轮次。
8. 战败、客户要求换门店、SLA 到期或保护期届满会关闭当前轮，并进入下一个启用策略模块。最后一个启用模块结束或所有模块都没有候选时，线索进入总部池。
9. 门店的跟进与核销结果进入后续评分数据，影响新线索和新轮次；历史已创建轮次不重新计算。

### 3.4 固定策略模块

| 策略模块 | 候选范围 | 排序/选择规则 | 轮次规则 |
|----------|----------|---------------|----------|
| 销售店优先 | 仅订单关联的销售店；销售店距 `follow_poi_id` 锚点在配置半径内，默认 10 公里 | 不参与综合评分，满足资格即选中 | 这是逻辑上的优先步骤；若实际选中，创建该线索第 1 个业务轮次 |
| 城市半径优选 | 锚点所在城市、配置半径内的可服务门店，默认 15 公里；排除优先步骤销售店和历史已分配门店 | 按综合评分降序，评分相同时按距锚点距离升序兜底 | 若此前没有实际分配，仍创建第 1 轮；详情应显示“第1轮 · 15公里城市优选”等实际策略名 |
| 城市兜底 | 锚点所在城市全部可服务门店；排除历史已分配门店 | 使用相同综合评分，分配给排名第一的可用门店 | 只有真实选中门店才创建下一连续轮次；无候选则进入总部池 |

默认综合评分：

`综合评分 =（核销转化能力 × 70% + 24 小时有效跟进率 × 30%）× 门店权重`

门店权重默认值为 1。当前版本保留权重字段及默认值，不要求实现门店个性化权重编辑模型。评分所用数据、公式参数、门店权重和计算时间必须形成可追溯快照。

### 3.5 跟进动作、SLA 与保护期

| 跟进动作 | 是否记为已产生跟进 | 是否启动保护期 | 当前轮处理 |
|----------|--------------------|----------------|------------|
| 已预约 | 是 | 是 | 当前轮继续有效，等待核销或保护期结束 |
| 待进一步跟进 | 是 | 是 | 当前轮继续有效，等待后续跟进、核销或保护期结束 |
| 未联系上 | 是 | 是 | 当前轮继续有效；即使未接通，也属于已产生跟进行为 |
| 暂不需要（线索战败） | 是 | 否 | 当前轮立即失效，进入下一启用策略 |
| 客户要求换门店 | 是 | 否 | 当前轮立即失效，排除原门店后进入下一启用策略 |

- 默认首次跟进 SLA 为分配后 24 小时，后台允许配置；配置为无限时不产生 SLA 到期。
- 默认核销保护期为首次有效跟进动作后 7 天，后台允许配置。
- 同一轮保护期只由首次进入保护的跟进动作确定，后续动作不重新计算开始时间或结束时间。
- SLA 只约束尚未产生任何跟进行为的轮次；产生可启动保护期的动作后，改由保护期约束。
- 每轮使用同一套动作与时限规则，不因轮次序号改变动作含义。

## 4. 核心价值模型

线索中心形成以下业务闭环：

`完整线索进入主池 → 按资格与能力择优分配 → 门店在明确时限内规范跟进 → 订单核销或按规则流转 → 行为与结果沉淀为门店评分 → 后续线索继续择优分配`

价值分为三层：

1. **结果价值**：通过更合理的门店选择和持续轮转，提高完整主池订单的总体核销率。
2. **组织价值**：优秀门店因实际转化和及时跟进获得更高评分与更多线索，形成正向激励；一般门店的动作、时限和跟进质量进入统一标准。
3. **运行底线**：完整主池、唯一有效轮次、明确权限、不可变历史和总部池兜底，保证系统不会以重复联系、越权查看或指标失真换取表面增长。

其中，唯一轮次和可追溯性是实现结果价值的必要条件，但不是最终成功标准。最终仍以同口径总体核销率是否高于基线判断方案有效性。

## 5. MVP 范围

### 5.1 当前权威业务模型（In Scope）

| 能力 | 当前必须成立的业务定义 |
|------|------------------------|
| 完整线索主池 | 保留全部原始线索与唯一订单视图，订单终态不删除历史 |
| 订单终态 | 已核销、已退款优先关闭线索；首次采集已终态不创建轮次 |
| 池位置 | 活跃线索必须可明确解释为门店跟进池、总部池或待执行中的受控状态，最终唯一落池 |
| 锚点与地理资格 | 仅以 `follow_poi_id` 为锚点；地理数据异常进入总部池并保留原因 |
| 门店评分与快照 | 默认评分公式、默认权重、冷启动口径及历史快照可追溯 |
| 规则版本 | 固定策略模块可启停、排序和配置参数；规则按作用范围命中，发布后不可静默改写 |
| 分配与再分配 | 三类策略按顺序执行，排除历史门店，仅真实分配创建连续轮次 |
| 跟进状态机 | 五类动作、SLA、保护期、订单终态优先级及多轮流转语义统一 |
| 总部池 | 承接锚点异常、无候选及末轮结束线索，不计门店指标，当前不再投放 |
| 联系方式权限 | 默认脱敏；仅当前有效轮次所属门店获授权用户可查看或复制完整手机号 |
| 经营指标 | 使用完整主池、唯一订单、自然月和成熟窗口计算总体核销率及诊断指标 |
| 管理与审计 | 支持规则配置、发布、试运行、决策追溯、总部池治理、线索采集和受控重建 |
| 切换与旧逻辑处理 | 在未投入真实生产的前提下直接删除冲突的旧分配引擎，不建立长期兼容层 |
| 验收追踪 | 所有条款必须映射到需求、实现位置、自动化测试和浏览器验收证据 |

### 5.2 远期业务蓝图（Out Scope）

| 能力 | 本版不纳入 Definition of Done 的原因 |
|------|--------------------------------------|

## 6. 备选方案对比

| 方案 | 描述 | 优势 | 劣势 | 取舍理由 |

## 8. 关键风险与兜底策略

| 风险 | 业务影响 | 兜底动作 |
|------|----------|----------|
| 源数据缺失、终态延迟或状态冲突 | 错误创建轮次、已核销/退款客户仍被联系、指标回补异常 | 建立字段完整性和终态一致性检查；终态事件优先；异常保留原始证据并进入数据治理，不静默猜测 |
| `follow_poi_id` 或门店经纬度缺失/错误 | 候选范围错误或大量线索无法分配 | 严格资格校验；异常进入总部池并记录原因；持续维护门店地理质量需求，不回退到未经确认的定位字段 |
| 重复分配、历史门店未排除或旧新引擎并行 | 同一客户被多店同时联系，责任与指标冲突 | 唯一有效轮次、幂等分配、历史门店排除、决策日志；上线前全面删除旧引擎和双写入口 |
| 评分样本不足、冷启动或强店正反馈 | 分数看似精确但不能代表真实能力，弱店长期失去机会 | 明确冷启动口径、保留评分快照和数据来源；定期复核分布；真实样本形成前不承诺固定提升幅度 |
| 订单终态、跟进动作、SLA 和保护期并发 | 同一轮次被重复关闭或错误创建下一轮 | 订单终态最高优先；状态迁移原子化；事件幂等；保存触发事件与前后状态用于审计 |
| 完整手机号越权查看、复制或导出 | 客户隐私泄露和责任不清 | 服务端按当前有效轮次授权；默认脱敏；管理范围可见不等于联系方式可见；查看、复制、导出留审计 |
| 试运行、旧轮次、总部池或重复订单污染指标 | 门店考核与总体效果结论失真 | 定义正式数据集；试运行单独隔离；门店指标只使用真实门店轮次；总体指标按完整主池唯一订单计算 |
| 错误发布、重建范围错误或版本不可追溯 | 大批线索错误流转，且难以恢复原因 | 发布版本冻结；先试运行再正式；高风险操作限权并审计；重建保留批次、规则版本和可回退证据 |

## 9. 项目周期

DYDATA-36 不设置独立的日历硬截止日期，以阶段门禁推进：

1. **BRD 冻结**：P0 字段 100% 锁定、冲突清零、四项前提挑战通过、七项质量门通过并获得总部业务侧终稿确认。
2. **下游规格**：PAGE_EXPLAINER 按本 BRD 校正，FOUNDATION 定义模型、接口、状态迁移、权限与运行方案，PRD 定义页面和验收细则。
3. **交付计划**：将 §5.3 的每个缺口映射到 Linear、实现位置、自动化测试和浏览器验收。
4. **旧引擎下线**：完成必要规格门禁后执行 DYDATA-34，删除旧分配、旧重建和兼容路径，不长期双写。
5. **受控试跑与重建**：使用完整数据验证唯一落池、规则版本、轮次、权限、总部池和指标数据集。
6. **正式运营与效果验收**：正式运营启动月作为过渡月，随后累计三个成熟完整自然月，按 §2 判断首期方向性目标。

远期蓝图不进入上述当前完成门槛，其实现周期由后续专项 BRD、FOUNDATION、PRD 和交付计划单独确定。

## 10. 页面定位

### 10.1 项目覆盖对象

本 BRD 覆盖：

- 线索中心：`/clues`、`/clues/details` 及线索跟进详情浮层。
- 分配后台：`/admin/clue-allocation/rules`、`/trial`、`/records`、`/headquarters`。
- 数据同步页面中的线索自动采集、手动补拉、线索主池物化/重建和相应任务状态区域。

数据同步页面内属于订单、结算和其他数据域的能力不纳入本专项 BRD。结算中心及其他协作者正在改造的页面，不因引用线索摘要或商品口径而自动进入本项目范围。

### 10.2 页面 / 入口定位

| 页面或入口 | 目标使用者 | 主要用途 | 定位 |
|------------|------------|----------|------|
| 线索看板 | 总部、区域/多店管理人员及按范围授权的门店用户 | 查看总体核销、库存、跟进表现和门店表现 | 查看型 |
| 线索明细 | 当前门店执行人员；管理角色按范围只读 | 定位当前任务、筛选线索、进入详情；管理角色监督存量与状态 | 操作型 |
| 线索跟进详情 | 当前有效轮次所属门店执行人员；历史相关角色只读 | 查看联系方式、商品订单、轮次与跟进历史，并在有权限时登记当前轮跟进 | 操作型 |
| 分配规则 | 最高管理员；普通管理员只读 | 维护、发布和审阅规则版本、作用范围、策略顺序与参数 | 配置型 |
| 分配试运行 | 最高管理员 | 在正式生效、切换或重建前验证候选、评分、跳过、分配及总部池结果 | 操作型 |
| 分配记录 | 最高管理员、总部运营和授权审计角色 | 追溯规则版本、参数、候选、评分、决策、批次与实际轮次 | 查看型 |
| 总部线索池 | 总部运营、总部管理和授权管理员 | 查看未进入或已退出门店跟进池的库存及原因 | 查看型 |
| 数据同步中的线索区域 | 最高管理员或 DYDATA-32 后续明确授权的运行角色 | 管理线索采集、手动补拉、物化/重建和任务反馈 | 操作型，含必要辅助配置 |

### 10.3 页面业务边界

- 店端的核心任务是处理当前有效线索，不展示抖音 `follow_life_account_name` 作为内部归属依据。
- 管理范围可见不自动授予完整手机号或代替门店跟进的权限。
- 总部池当前只用于查看库存和原因，不提供再次投放；若未来增加再投放，应重新评审其页面定位和指标影响。
- 页面设计可以优化信息架构、视觉和操作效率，但不得改变订单终态优先、唯一有效轮次、策略与轮次分离、联系方式授权和指标口径。

### 10.4 下游待确认边界

PAGE_EXPLAINER 与页面设计负责信息架构、页面流程、交互语义和视觉呈现；FOUNDATION 负责数据模型、接口、事件与状态迁移、定时任务、权限安全、迁移和运行方案；PRD 负责页面字段、控件、反馈、异常与空状态、角色交互和逐项验收条件；开发计划与 Linear 负责实现拆分、迁移、测试、上线和旧引擎删除。任何下游产物都不得以技术限制或页面便利为由改变本 BRD 已冻结的业务语义。
