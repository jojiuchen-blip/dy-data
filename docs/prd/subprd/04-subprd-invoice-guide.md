## §1 文档范围

本文档覆盖**开票确认页的只读开票流程引导**（当前能力边界 + 五节点流程说明 + 开票准备材料 + 预计开票范围 + 支持渠道与常见问题）。

本区块只定义 `#/invoice` 的说明与引导能力。当前版本不提供账单确认、锁账、解锁、发票开具或提交、财务审核、撤回、修改、删除和资金结算操作，也不调用发票接口。页面名称“开票确认”不表示系统已经具备在线开票能力。

### 需求清单

| # | 需求 | 需求简述 | 对应章节 |
|---|------|---------|---------|
| R1 | 未开放边界与五节点流程 | 持续标明功能尚未开放，并按顺序解释月度账单生成、门店确认、开具推广服务费发票、财务审核和进入结算。 | §3 |
| R2 | 开票准备信息 | 说明正式账期边界、核对前置条件、需准备材料和预计开票范围。 | §4 |
| R3 | 支持渠道与 FAQ | 说明待上线支持入口、常见问题和账单差异核对路径，不伪造联系方式。 | §5 |
| R4 | 外部依赖保留 | 发票抬头、税号、类目、税率、接收方式和正式支持入口继续标记为待财务确认或待上线通知。 | §4、§5 |

## §3 当前能力边界与五节点流程

### 3.1 用户体验

**数据来源**：前端静态渲染已确认的能力边界和五节点流程文案；本区域不请求业务数据、不调用发票接口。账单与接口写入边界见[双费用结算与报表 API §5](../foundation/foundation-api-dy-data/settlement-reporting.md#5-开票确认页与账单写入边界)。

**交互语义引用**：`invoice-guide.readonly-process.1`

**布局**：

```
[账单确认与开票]                              [当前功能暂未开放]

[01 月度账单待确认] → [02 门店确认账单] → [03 门店开具推广服务费发票]
                                  → [04 发票提交财务审核]
                                  → [05 审核完成并进入结算]
```

**展示规则**：

- 页面标题、首屏说明和流程区必须持续展示“当前功能暂未开放”以及“当前不提供在线开票或财务操作”。
- 五个节点只表达未来业务顺序，不表达当前门店已经到达某一步，也不展示完成、处理中、驳回或审核通过等实时状态。
- 每个节点统一标记为“流程节点”；视觉强调只能帮助理解阅读顺序，不得解释为真实进度。
- 页面不得出现“确认账单”“立即开票”“提交发票”“发起审核”“审核通过”“进入结算”等会被误认为可执行写操作的主按钮、表单或上传入口。
- 当前版本不读取 `settlement_statement.statement_status` 来驱动节点状态，也不把单店分账返回的账单状态解释为已经完成在线确认或开票。
- “未来开放”仅是流程说明，不属于本轮实现范围；后续开放任何写能力前，必须新建或更新 Linear issue，补齐角色、幂等、撤回、审计和异常验收并扩展 Foundation API。

**前端职责**：仅渲染已确认的静态流程顺序和能力边界；不请求或推导账单进度，不调用确认、锁账或发票接口，不维护可提交的表单状态。

| 02 门店确认账单 | — | 五节点顺序中的第 2 个说明节点；当前不可操作 | — | — |
| 03 门店开具推广服务费发票 | — | 五节点顺序中的第 3 个说明节点；当前不可操作 | — | — |

### 3.4 异常与兜底

**服务端兜底**：

| 场景 | 处理 |
|------|------|
| 用户直接访问 `#/invoice` | 只返回承载静态页面所需的应用资源；不触发账单、发票或结算写入。 |
| 未来账单或发票服务不可用 | 不影响本页静态流程说明；页面不得伪装成可执行入口。 |
| 出现未经定义的确认、锁账或发票请求 | 当前契约不提供对应公开路由；按未定义接口拒绝，不对业务状态产生副作用。 |

**前端渲染兜底**：

| 场景 | 处理 |
|------|------|
| 后续节点或业务口径尚未确定 | 保留流程顺序并将未确定内容标记“待财务确认”或“待上线通知”，不编造状态。 |
| 静态资源局部加载失败 | 保留页面标题、未开放状态和无操作边界；不显示可提交控件。 |
| 用户误以为高亮节点是当前进度 | 节点统一显示“流程节点”，并在流程区补充“仅说明顺序，不代表实时进度”。 |

## §4 开票准备与预计范围

### 4.1 用户体验

**数据来源**：前端静态渲染正式账期边界、核对路径、材料类别和预计开票范围定义；不读取实际账单金额，不调用发票接口。术语口径见[Foundation 术语表](../foundation/foundation-glossary-dy-data.md)，接口边界见[双费用结算与报表 API §5](../foundation/foundation-api-dy-data/settlement-reporting.md#5-开票确认页与账单写入边界)。

**交互语义引用**：`invoice-guide.preparation.2`

**布局**：

```
┌────────────────────┬────────────────────┬────────────────────┐
│ 前置条件           │ 需准备材料         │ 预计开票范围       │
│ · 正式账期边界     │ · 开票主体信息     │ · 已确认账单       │
│ · 先核对费用依据   │ · 已确认月度账单   │ · 推广服务费       │
│ · 等待财务通知     │ · 其他证明材料     │ · 排除管理服务费   │
└────────────────────┴────────────────────┴────────────────────┘
```

**展示规则**：

- 前置条件必须说明：正式账期从 `2026-08` 开始，`2026-07` 是测试账期，不进入正式开票准备范围。
- 用户应先在单店分账和订单费用明细中核对推广服务费金额、订单依据、费率和规则版本；存在差异时先完成追溯，不在本页调整金额。
- 准备材料只列材料类别：门店开票主体与纳税人识别信息、已确认月度账单、推广服务费订单明细，以及财务后续要求的其他资质或证明材料。
- 发票抬头、税号、类目、税率、接收方式尚未形成正式口径，必须整体标记“待财务确认”或“以上线通知为准”，不得展示示例值为真实值。
- “预计开票范围”只定义业务范围，不展示实际可开票金额：仅包含正式账期已确认账单中的推广服务费；管理服务费属于门店应扣费用，不计入门店推广服务费开票金额。
- 未确认账单、`2026-07` 测试数据和管理服务费均不得进入本页所述的预计开票范围。
- 页面不得自行用推广服务费汇总、订单明细或当前规则计算发票金额，也不得将“预计范围”写成最终税务或财务结论。

**前端职责**：仅渲染静态准备指引和待确认标签；不读取或汇总实际账单金额，不校验税务信息，不计算含税/不含税金额，不生成发票材料或提交数据。

> 下表为端到端追踪：UI 元素 ← API 字段 ← 服务端计算 ← 数据源/配置源。**本区域只解释范围，不读取账单行或发票数据**。

| UI 元素 | API 字段 | 计算规则 | 数据源（服务端读取） | 配置源（服务端读取） |

| 核对路径 | — | 引导用户通过固定四页导航返回单店分账和订单费用明细 | — | — |
| 材料类别 | — | 展示已确认的材料类别，不展示未确认具体值 | — | — |
| 待财务确认项 | — | 发票抬头、税号、类目、税率、接收方式统一标记待确认 | — | — |
| 预计开票范围 | — | 静态说明“正式账期已确认账单中的推广服务费”，不计算金额 | — | — |
| 管理服务费排除说明 | — | 固定说明管理服务费为应扣费用，不计入推广服务费开票金额 | — | — |

### 4.4 异常与兜底

**服务端兜底**：

| 场景 | 处理 |
|------|------|
| 账单或费用查询服务异常 | 本页仍只展示静态准备口径，不返回或猜测实际账单金额。 |
| 发票材料口径尚未确认 | 不提供材料配置或提交接口；继续作为 Foundation 外部依赖跟踪。 |
| 用户尝试构造金额或税务参数 | 当前页面无对应请求参数和处理接口，不采纳为业务事实。 |

**前端渲染兜底**：

| 场景 | 处理 |
|------|------|
| 发票具体字段未确认 | 显示“待财务确认 / 以上线通知为准”，不填充虚构示例。 |
| 用户需要核对实际金额 | 引导返回单店分账和订单费用明细，不在本页展示推算值。 |
| 用户查看测试账期 | 明确提示 `2026-07` 不纳入正式开票准备，不显示可开票暗示。 |
| 用户混淆推广费与管理费 | 在预计范围卡片中同时显示纳入项与排除项。 |

### 4.6 验收

## §5 支持渠道与常见问题

### 5.1 用户体验

**数据来源**：前端静态渲染支持渠道边界与 FAQ；不读取联系人、工单、聊天或发票服务数据，不调用外部支持接口。

**交互语义引用**：`invoice-guide.preparation.2`

**布局**：

```
┌──────────────────────────────────────────────────────────────┐
│ 支持渠道                                                     │
│ 当前页面不提供在线提交；正式支持入口待上线通知。             │
├──────────────────────────────────────────────────────────────┤
│ 常见问题                                                     │
│ · 现在能在系统里开票吗？                                    │
│ · 2026-07 测试账单需要开票吗？                              │
│ · 账单金额有疑问怎么办？                                    │
└──────────────────────────────────────────────────────────────┘
```

**展示规则**：

- 支持渠道明确说明当前页面不提供在线提交，正式支持入口标记“待上线通知”。
- 可以提示用户联系所属区域运营或既有财务对接人，但不得展示未经确认的姓名、电话、邮箱、群聊、二维码或外部链接。
- FAQ 至少回答：当前能否在线开票、`2026-07` 测试账单是否需要开票、账单金额或订单归属有疑问时如何核对。
- “现在能在系统里开票吗”的答案固定为不能；当前仅提供流程和准备指引。
- 账单差异处理路径为：先返回单店分账核对汇总，再下钻订单费用明细核对业务 ID、费用方向、费率、调整和规则版本，最后联系既有运营或财务对接人。
- 本页不创建工单、不发消息、不上传附件、不保存用户咨询内容。

**前端职责**：仅渲染静态支持说明和 FAQ，并使用已有四页导航帮助用户回到核对页面；不构造联系人、不接入客服或工单系统、不保存咨询记录。

### 5.3 数据链路

> 下表为端到端追踪：UI 元素 ← API 字段 ← 服务端计算 ← 数据源/配置源。**本区域没有在线支持或工单数据链路**。

| UI 元素 | API 字段 | 计算规则 | 数据源（服务端读取） | 配置源（服务端读取） |
|---------|----------|---------|--------|--------|
| 当前无在线提交说明 | — | 固定能力边界文案 | — | — |
| 正式支持入口 | — | 固定标记“待上线通知”，不生成联系方式 | — | — |
| 在线开票 FAQ | — | 固定回答当前不能在线开票 | — | — |
| 测试账期 FAQ | — | 固定回答 `2026-07` 不需要进入正式开票准备 | — | — |
| 账单差异 FAQ | — | 引导使用已有单店分账和订单费用明细页面核对 | — | — |

### 5.4 异常与兜底

**服务端兜底**：

| 场景 | 处理 |
|------|------|
| 正式支持入口尚未确定 | 不返回占位联系人或外部链接，本页保持静态待上线说明。 |
| 后续支持系统不可用 | 不影响本页静态 FAQ；不得降级为未授权的个人联系方式。 |
| 用户尝试提交咨询内容 | 当前页面无写接口，不保存表单、附件或消息。 |

**前端渲染兜底**：

| 场景 | 处理 |
|------|------|
| 联系方式未确认 | 只显示“请联系既有区域运营或财务对接人；正式入口待上线通知”。 |
| 用户无法从 FAQ 解决差异 | 提示先完成两级核对，再通过既有组织渠道联系，不显示虚构入口。 |
| 固定导航不可用 | 保留核对步骤文字，不生成带未授权门店或账单参数的深链。 |

| 1 | 业务规则 | 在线开票 FAQ | 用户查看“现在能在系统里开票吗” | 明确回答不能，并说明当前仅提供流程和准备指引。 |
| 2 | 业务规则 | 测试账期 FAQ | 用户查看 `2026-07` 是否需要开票 | 明确回答不进入正式开票准备。 |
| 3 | UX 交互 | 账单差异核对 | 用户按 FAQ 处理金额疑问 | 能理解先查单店汇总、再查订单依据、最后联系既有对接人的路径。 |
| 4 | 异常兜底 | 正式入口未上线 | 页面没有已确认支持入口 | 显示待上线通知，不出现虚构电话、邮箱、二维码或链接。 |
| 5 | 业务规则 | 无咨询写入 | 用户浏览支持与 FAQ 区 | 页面不创建工单、不上传附件、不发送或保存咨询信息。 |

## §6 异常与兜底策略（全局）

### 6.1 静态页面兜底

| 场景 | 处理 |
|------|------|
| 应用资源加载缓慢 | 优先展示页面标题、未开放状态和只读边界，再加载流程与指引内容。 |
| 局部指引内容异常 | 不隐藏“功能暂未开放”，不显示任何写操作控件。 |
| 浏览器刷新或直接打开路由 | 恢复同一静态说明页，不提交账单、发票或咨询数据。 |

### 6.2 外部依赖兜底

| 外部依赖 | 当前影响 | 兜底策略 |
|---------|---------|---------|
| 正式开票材料口径 | 无法给出抬头、税号、类目、税率和接收方式 | 统一标记待财务确认，不展示示例为真实值。 |
| 正式支持入口 | 无法提供在线咨询或提交渠道 | 标记待上线通知，只引用既有区域运营或财务对接关系。 |
| 未来发票 API | 当前不能执行开票、提交或审核 | 本页不调用接口、不显示操作按钮；未来需求重新进入 Linear 和 Foundation 流程。 |

## §7 接口契约与写入边界

### 7.1 当前页面接口

`#/invoice` 当前没有业务 API 调用。五节点流程、准备材料类别、预计范围定义、支持渠道边界和 FAQ 均为前端静态只读内容。

> 权威边界见[双费用结算与报表 API §5](../foundation/foundation-api-dy-data/settlement-reporting.md#5-开票确认页与账单写入边界)。

**当前版本约束**：

- 不提供发票查询、创建、上传、提交、撤回、修改、删除或审核接口。
- 不提供公开的账单 `confirm/lock/unlock/update/delete` 路由。
- 单店分账即使返回账单状态和推广服务费汇总，也不得据此声称本页已经完成在线确认或形成可开票金额。
- 内部账单锁定事务与本页无直接调用关系；页面不触发 `settlement_statement`、`settlement_statement_line` 或 `settlement_statement_entry` 写入。
- 若未来开放门店确认、发票提交或财务审核，必须先在 Linear 中定义范围和验收，再扩展 Foundation 术语、Schema、API 与权限审计契约；不得直接在本 PRD 上追加按钮绕过治理流程。

### 7.2 页面与数据边界

| 页面展示内容 | 当前来源 | 是否读取业务数据 | 是否允许写入 |
|-------------|---------|:----------------:|:------------:|
| 未开放状态与五节点流程 | 已确认静态文案 | 否 | 否 |
| 前置条件与材料类别 | 已确认静态文案 | 否 | 否 |
| 预计开票范围定义 | 已确认业务口径 | 否；不读取实际金额 | 否 |
| 支持渠道与 FAQ | 已确认静态文案及待上线边界 | 否 | 否 |
