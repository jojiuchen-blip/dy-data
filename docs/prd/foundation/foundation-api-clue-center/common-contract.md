# 公共接口契约

> 返回 [API 索引](../foundation-api-clue-center.md)

## 1. 宿主项目约定

线索中心以当前 FastAPI + React 项目的已用契约为宿主约定。通用规范与宿主现状冲突时，本专项采用下表，并在后续实现中保持前后端一致。

| 项目 | 线索中心 V1.0 约定 | 说明 |
|------|-------------------|------|
| API 前缀 | `/api/v1` | 与现有路由和前端请求器一致 |
| JSON 字段 | `snake_case` | 避免线索专项引入全站双命名体系 |
| 枚举值 | 小写 `snake_case` 字符串 | API 不暴露数据库 `smallint` |
| 成功包络 | `{ "data": T, "meta": {...} }` | 文件下载除外 |
| 错误包络 | `{ "detail": { "code", "message", "request_id", "context" } }` | 替代不可机器判断的纯字符串 `detail` |
| 分页请求 | `page`、`page_size` | `page` 从 1 开始 |
| 分页响应 | `rows`、`pagination` | 与现有 Pydantic/TypeScript 模型一致 |
| 时间 | RFC 3339 UTC | 前端按用户时区显示 |
| 自然日筛选 | `YYYY-MM-DD` | 起止日均包含整日 |
| 业务 ID | 字符串 | 不向前端暴露数据库数字主键 |

该决定只冻结线索中心 V1.0。若未来全站统一迁移为 `camelCase` 或 `code/msg/data`，必须作为独立版本化改造，不在线索中心实现中混用两套字段。

## 2. 成功响应

### 2.1 普通响应

```json
{
  "data": {},
  "meta": {
    "generated_at": "2026-07-22T08:30:00Z",
    "source": "postgres",
    "request_id": "req_01J..."
  }
}
```

| 字段 | 类型 | 必返 | 说明 |
|------|------|------|------|
| `data` | object/array | 是 | 业务数据；空列表返回 `[]`，存在但无值的字段返回 `null` |
| `meta.generated_at` | datetime | 是 | 服务端生成时间 |
| `meta.source` | string | 是 | `postgres`、`demo` 或明确的降级源；生产写接口只允许 `postgres` |
| `meta.request_id` | string | 是 | 链路请求 ID，可关联审计与日志 |

### 2.2 分页响应

```json
{
  "data": {
    "rows": [],
    "pagination": {
      "page": 1,
      "page_size": 20,
      "total": 0,
      "total_pages": 0
    }
  },
  "meta": {
    "generated_at": "2026-07-22T08:30:00Z",
    "source": "postgres",
    "request_id": "req_01J..."
  }
}
```

- `page_size` 默认 20，普通列表最大 100；需要批量选择的试运行列表最大 200。
- `total_pages = 0` 时前端显示第 1/0 页但不得允许继续翻页。
- 不允许通过把 `page_size` 设置为超大值绕过导出接口。

### 2.3 文件下载响应

- HTTP 200，`Content-Type` 使用真实文件类型。
- 文件名通过 `Content-Disposition: attachment; filename*=UTF-8''...` 返回。
- CSV 必须包含 UTF-8 BOM；若实现 XLSX，则不得将文本伪装为 `.xlsx`。
- 文件内容不经过 JSON 包络；错误仍使用标准错误包络。

## 3. 错误契约

```json
{
  "detail": {
    "code": "CLUE_40901",
    "message": "当前分配轮次已失效，不能继续跟进",
    "request_id": "req_01J...",
    "context": {
      "order_id": "111...",
      "assignment_round_id": "round_..."
    }
  }
}
```

| HTTP | 业务码 | 适用场景 |
|------|--------|----------|
| 400 | `CLUE_40001` | 日期、分页、筛选或请求结构非法 |
| 400 | `CLUE_40002` | 枚举或策略参数非法 |
| 401 | `CLUE_40101` | 未登录、会话失效或内部服务认证失败 |
| 403 | `CLUE_40301` | 缺少页面权限 |
| 403 | `CLUE_40302` | 目标数据超出账号门店范围 |
| 403 | `CLUE_40303` | 不是当前有效轮次所属门店或无跟进授权 |
| 403 | `CLUE_40304` | 不允许读取完整手机号或明文导出 |
| 403 | `CLUE_40305` | 需要最高管理员或专门审计权限 |
| 404 | `CLUE_40401` | 线索/订单不存在或在当前范围不可见 |
| 404 | `CLUE_40402` | 分配轮次不存在 |
| 404 | `CLUE_40403` | 跟进记录不存在 |
| 404 | `CLUE_40404` | 规则、版本、门店组、批次或决策不存在 |
| 409 | `CLUE_40901` | 当前轮次已失效或订单已终态 |
| 409 | `CLUE_40902` | `state_version`/资源版本并发冲突 |
| 409 | `CLUE_40903` | 幂等键已被不同请求使用 |
| 409 | `CLUE_40904` | 预览过期或范围已变化 |
| 409 | `CLUE_40905` | 已发布规则版本不可修改 |
| 409 | `CLUE_40906` | 重复活动轮次、总部池条目或正式批次 |
| 422 | `CLUE_42201` | 锚点或门店地理数据不足，不能正式分配 |
| 422 | `CLUE_42202` | 没有可用的已发布规则 |
| 422 | `CLUE_42203` | 固定策略缺失、顺序重复或参数不完整 |
| 422 | `CLUE_42204` | 删除跟进记录会导致无法证明的状态回退 |
| 429 | `CLUE_42901` | 敏感读取、批量操作或外部接口限流 |
| 500 | `CLUE_50001` | 未分类服务异常；响应不得包含手机号、密钥或令牌 |

不存在与无权查看统一返回 404 的接口：线索详情、轮次详情、跟进记录详情，避免枚举越权对象。已登录但缺页面权限的路由仍返回 403。

## 4. 权限与数据范围

### 4.1 页面权限键

| 权限键 | 页面 | 接口范围 |
|--------|------|----------|
| A01 | 线索看板 | 筛选、概览、月度和门店指标 |
| A02 | 线索明细 | 明细、详情、导出、联系方式和跟进 |
| D05 | 线索分配规则 | 规则、版本和门店组 |
| D06 | 分配试运行 | 可试运行线索、预览、试运行和重建 |
| D07 | 分配记录 | 批次、决策、候选、评分、主池、质量和审计 |
| D08 | 总部线索池 | 总部池库存查询 |
| D10 | 数据同步 | 受控重建和同步状态 |

后端必须按请求路径映射权限键，不能只依赖前端隐藏入口。一个接口被多个页面消费时，满足任一明确列出的权限键即可；写权限仍需通过角色与动作级校验。

### 4.2 角色能力

| 角色/能力 | 默认数据范围 | 读 | 写 |
|-----------|--------------|----|----|
| 门店账号 | 当前账号绑定门店 | A01/A02 范围内脱敏读取 | 仅当前有效轮次；完整号读取需动作授权 |
| 多店/区域管理 | 授权门店集合 | 授权范围内汇总与明细 | 默认只读，不代替门店跟进 |
| 总部运营/管理 | 全局或授权范围 | 全国汇总、主池和总部池 | 默认只读，不因全局可见自动获得完整号 |
| 普通管理员 | 按 DYDATA-32 配置的页面与数据范围 | D05-D08 默认只读 | 线索分配管理写操作禁止 |
| 最高管理员 | 全局 | 全部管理读 | 规则、试运行、重建、评分刷新和跟进软删除 |
| 内部服务身份 | 固定任务范围 | 仅任务所需 | 仅明确的内部任务命令 |

### 4.3 当前有效轮次授权

完整手机号和跟进写入必须同时满足：

1. 订单仍为可促核销状态。
2. 主线索处于门店跟进池。
3. `assignment_round_id == current_assignment_round_id`。
4. 轮次状态为 `pending_follow_up` 或 `protected`。
5. 门店账号命中 `assigned_store_id`，或角色具备明确配置的跨店动作权限。

最高管理员的管理权限不自动等于客户联系权限。若未来允许总部代跟进或读取完整号，必须由 DYDATA-32 新增明确动作权限，不得通过 `is_highest_admin` 隐式放开。

## 5. 日期、筛选与排序

- 日期起止参数语义是自然日闭区间。例如 `2026-06-25` 至 `2026-06-26` 包含两天全部数据。
- 后端转换为 `[2026-06-25T00:00:00+08:00, 2026-06-27T00:00:00+08:00)` 后查询，禁止把结束日解释为当天零点。
- 线索筛选统一使用 `clue_created_date_start/end`；经营指标按 `order_created_at` 归月，二者不得混用。
- 列表默认稳定排序：主排序字段降序，业务 ID 降序兜底，避免翻页时重复或遗漏。
- 文本搜索 `q` 只能匹配订单号、脱敏手机号尾号或明确允许字段，不允许普通列表搜索手机号明文。

## 6. 幂等、并发与事务

### 6.1 幂等

- F01、R09、R10、A03、A04、A10、J01-J08 必须接收 `Idempotency-Key` 请求头。
- 服务端将键与调用方、路由、请求摘要绑定；相同请求重放返回原结果，不重复写入。
- 相同键对应不同请求返回 `CLUE_40903`。
- 外部状态事件以稳定 `event_key` 去重；同一源记录重复采集不得重复关闭轮次或重复计入指标。

### 6.2 乐观锁

- Schema 已定义 `state_version` 的可更新资源请求携带该版本；成功后返回递增版本。未定义版本列的门店组/成员替换在实现时使用行锁和成员集合摘要避免并发覆盖，不在 Phase 4 临时增加字段。
- 跟进保存同时校验主线索和当前轮次版本。
- 规则草稿、门店组、批次确认和状态任务均不得用“最后写入覆盖”掩盖冲突。

### 6.3 事务边界

- 跟进保存：跟进记录、轮次摘要、保护期、立即关闭结果和审计在一个事务中完成；需要下一轮时写可靠待办事件，由正式分配任务消费。
- 订单终态：状态事件、主线索关闭、当前轮次关闭、总部池关闭、指标事实和投影更新在一个事务或同一 outbox 事务中保证最终一致。
- 正式分配：批次项、决策、候选、规则绑定、真实轮次、主线索池位置和审计按单条线索事务提交；批次允许部分失败并可重试失败项。
- 试运行：只写试运行批次、批次项、决策和候选；不得写规则绑定、真实轮次、总部池活动条目或经营指标。
- 重建：先创建不可变快照与预览，再在新批次中重建；禁止原地改写历史决策、历史轮次或评分证据。

## 7. 审计与敏感数据

必须审计的事件：

- `phone_reveal`、`phone_copy`、`plaintext_export`
- `follow_up_delete`
- `rule_publish`、`rule_retire`
- `trial_run`、`rebuild_run`、`formal_allocation_run`
- 权限拒绝的联系方式读取与高风险管理请求

审计快照必须脱敏。日志、异常、追踪标签和审计 `detail_json` 均不得包含 `phone_plain`、`access_token`、`client_secret`、解密请求原文或导出文件内容。

## 8. 限流与批量上限

| 场景 | 默认限制 |
|------|----------|
| 单用户查看/复制完整号 | 60 次/分钟，并记录拒绝 |
| 同步导出 | 50,000 行；超出时返回 422 并要求缩小范围，后续另建异步导出需求 |
| 试运行选择 | 1-200 条/次 |
| 正式内部批次 | 由任务配置分片，单批默认 500 条 |
| 跟进保存 | 同订单串行化，幂等重试 |
| 规则发布/退役 | 同规则串行化 |

## 9. OpenAPI 要求

- 所有公开和管理接口必须进入 FastAPI OpenAPI，包含中文摘要、字段说明、枚举、示例和错误响应。
- 内部接口单独使用 `internal` tag，并在公网路由层不可达。
- Pydantic 响应模型中的业务字段必须必返；可空字段显式为 `null`。
- 文件下载接口必须在 OpenAPI 中声明真实媒体类型和错误包络。
