## 全局业务术语

| 术语 | 定义 |
|------|------|
| 线索 | 围绕一笔可识别订单开展促核销跟进的业务对象；多条抖音原始记录可以归并到同一线索，经营统计按唯一 `order_id` 去重。 |
| 原始线索记录 | 抖音线索接口返回并原样留存的一次源记录，用于审计、重放和字段追溯，不因去重、订单终态或业务关闭而删除。 |
| 线索主记录（主线索） | 系统将原始线索、订单状态、锚点、池位置和当前责任关系归一后形成的持久业务实体，以稳定的 `lead_key` 标识。 |
| 线索中心订单视图 | 面向列表、详情和订单级经营统计的物化视图，以 `order_id` 为主键汇总主线索、当前轮次、商品、联系方式和核销信息。 |
| 完整线索主池 | 包含所有已进入系统的唯一订单的逻辑全集，包括活跃、已核销、已退款、门店跟进池、总部池和已关闭订单，不等同于某一张物理表。 |
| 订单 ID | 抖音订单的稳定标识 `order_id`，是总体核销率、完整主池规模和订单级去重的统一统计键。 |
| 主线索 ID | 系统内部标识线索主记录的稳定键 `lead_key`，用于关联规则版本、轮次、决策和总部池记录，不替代经营统计中的 `order_id`。 |
| 代表线索 ID | 从同一订单关联的原始线索中选出的 `canonical_clue_id`，只用于追溯来源，不作为订单级统计或分配轮次的唯一键。 |
| 下单时间 | 用户创建订单的业务时间，用于总体核销率的自然月归属，不得用线索生成、采集、分配或跟进时间替代。 |
| 线索生成时间 | 抖音线索形成的时间，用于线索列表排序和运营追溯，不等同于用户下单时间。 |
| 分配时间 | 某一真实分配轮次将责任交给具体门店的时间，是该轮 SLA 的起算依据。 |
| 跟进时间 | 门店提交某一条跟进记录的时间；一轮可以有多条跟进记录，轮次上的最近跟进时间只是摘要。 |
| 核销时间 | 订单达到已核销状态的业务时间，用于关闭线索和结果追溯，不决定该订单归属的统计月份。 |
| 数据范围 | 当前账号按全国、区域、多店或单店授权能够查看的数据集合；可见不自动等于可操作或可查看完整联系方式。 |
| 当前责任门店 | 当前有效分配轮次实际指向的门店，是唯一可以在满足权限时联系客户和登记跟进的门店。 |
| 抖音跟进归属 | 抖音返回的 `follow_life_account_id/name` 信息，仅作为源数据保留，不作为本系统内部归属、首轮分配或店端展示依据。 |

## 生命周期与池位置术语

| 术语 | 定义 |
|------|------|
| 订单状态 | 订单交易事实的归一状态，当前业务只区分可促核销、已核销和已退款，并具有高于轮次时限及跟进动作的判定优先级。 |
| 订单终态 | 已核销或已退款；订单进入终态后线索立即关闭，不再创建或继续任何门店分配轮次。 |
| 线索生命周期状态 | 线索本体是否仍需促核销跟进的状态，必须与订单状态、池位置和轮次状态分别存储。 |
| 活跃线索 | 订单尚未核销或退款、仍可能通过门店跟进促成核销的线索。 |
| 关闭线索 | 因订单已核销或已退款而不再需要门店跟进的线索；主记录和历史仍保留，但不属于总部池或门店跟进池库存。 |
| 池位置 | 活跃线索当前处于待执行、门店跟进池或总部池中的业务位置，不用于表达订单是否核销或某轮是否失效。 |
| 待分配 | 活跃线索已进入主池但尚未完成本次正式分配判断的受控过渡状态，不能长期作为最终落点。 |
| 门店跟进池 | 当前存在一个有效门店分配轮次的活跃线索集合，同一线索任一时刻最多属于一个门店的跟进池。 |
| 总部池 | 当前没有门店责任归属、需要总部查看和治理的活跃线索集合，不计入任何门店的线索数或跟进率分母。 |
| 总部池库存 | 当前处于总部池的唯一活跃订单数，按 `order_id` 去重，不包含已核销、已退款或试运行结果。 |
| 总部池入池原因 | 解释活跃线索为何没有门店归属的标准原因，例如锚点缺失、锚点无法映射、地理数据不可用、规则无候选或全部策略耗尽。 |
| 唯一落池 | 一次正式处理结束后，活跃线索只能处于门店跟进池或总部池之一，关闭线索则不进入任何活跃池。 |
| 当前有效轮次 | 当前责任门店尚可依法执行跟进的唯一分配轮次；订单终态、超期、战败、换店或后续轮次创建都会使旧轮失效。 |
| 历史轮次 | 已结束且不再具有联系方式查看或跟进操作权的分配轮次，仅用于责任、动作和流转审计。 |
| 线索状态（店端显示） | 面向一线门店的合并显示状态，只允许使用待跟进、已跟进、超期失效、主动战败、已核销和已退款，不直接暴露内部状态码。 |
| 待跟进 | 当前轮有效且尚无任何跟进记录的店端状态。 |
| 已跟进 | 当前轮有效且已经产生至少一次跟进行为的店端状态，不表示订单已经核销。 |
| 超期失效 | 某轮在首次跟进 SLA 到期前没有产生跟进行为而失效的店端历史状态。 |
| 主动战败 | 门店选择“暂不需要（线索战败）”后当前轮立即失效的店端历史状态。 |
| 已核销 | 订单历史上已经达到核销状态、线索不再需要促核销跟进的店端状态。 |
| 已退款 | 订单已经退款、线索不再需要促核销跟进的店端状态。 |
| 可跟进 | 订单仍活跃、轮次是当前有效轮次且当前账号对责任门店具有操作权限的联合判断，不是单独依赖某一个状态字段。 |

## 分配规则与轮次术语

| 术语 | 定义 |
|------|------|
| 分配规则 | 在指定作用范围内定义固定策略模块的启停、执行顺序和参数的配置集合。 |
| 规则作用范围 | 规则可以绑定到锚点门店、门店组、城市或全局；命中优先级固定为锚点门店 > 门店组 > 城市 > 全局。 |
| 规则版本 | 分配规则的一份不可变版本；草稿可编辑，发布后冻结，线索首次分配时锁定并在后续轮转中固定沿用。 |
| 规则版本绑定 | 主线索首次进入正式分配时与一个已发布规则版本建立的持久关系，后台之后发布新版本不会改变既有线索。 |
| 固定策略模块 | 系统内置的销售店优先、城市半径优选和城市兜底三种算法类型；类型固定，但允许启停、排序和配置参数。 |
| 策略步骤 | 分配引擎对一个已启用策略模块的一次候选评估，步骤可以跳过且不等同于业务轮次。 |
| “第 0 轮” | 对销售店优先步骤的历史非正式称呼，不是有效业务轮次号，产品、接口和数据库均不得创建 round 0。 |
| 分配轮次 | 线索被真实分配给一个具体门店后形成的一段责任周期，只有选中门店时才创建，并按该线索实际分配次数从 1 连续递增。 |
| 分配轮次 ID | 唯一标识一次真实门店责任周期的内部技术编号，用于接口、审计和幂等控制，不作为店端主信息。 |
| 分配策略名 | 产生某轮真实分配的策略名称，详情以“第1轮 · 销售店优先”或“第1轮 · 15公里城市优选”等形式展示。 |
| 锚点 POI | 抖音线索字段 `follow_poi_id`，是无法取得客户精准位置时唯一允许使用的位置代理来源。 |
| 锚点门店 | 由 `follow_poi_id` 映射得到且城市、经纬度有效的门店，用于确定城市范围和计算候选距离，不自动成为责任门店。 |
| 锚点不可用 | `follow_poi_id` 缺失、无法映射门店或锚点门店城市/经纬度无效的状态，此时活跃线索直接进入总部池。 |
| 销售店 | 订单关联的实际销售门店，是销售店优先策略的唯一候选，不等同于锚点门店或抖音跟进归属门店。 |
| 可服务门店 | 账号、营业和必要地理信息均满足当前规则资格，可参与候选计算的门店。 |
| 候选门店 | 在某一策略步骤中通过作用范围、城市、距离、服务资格和历史排除条件的可选门店。 |
| 历史门店排除 | 新策略步骤必须排除该线索过去已经真实分配过的门店，避免同一线索再次回到同一责任门店。 |
| 销售店优先 | 仅判断销售店是否在锚点配置半径内，默认 10 公里；满足即分配且不参与综合评分。 |
| 城市半径优选 | 在锚点同城市和配置半径内选择候选门店，默认 15 公里，并在排除销售店及历史门店后按综合评分和距离排序。 |
| 城市兜底 | 将候选扩大到锚点城市全部可服务门店，在排除历史门店后按同一综合评分选择排名第一的门店。 |
| 配置半径 | 销售店优先或城市半径优选允许的最大锚点距离，10 公里和 15 公里只是默认值，不得硬编码为不可配置常量。 |
| 综合评分 | 按“（门店核销转化能力 × 70% + 24小时有效跟进率 × 30%）× 门店权重”计算的门店排序值。 |
| 门店权重 | 综合评分的可扩展乘数，当前默认值为 1；V1 保留字段和快照，但不提供门店个性化调权能力。 |
| 评分快照 | 某次分配决策实际使用的门店指标、权重、公式参数和计算时间的不可变证据，历史轮次不得随后重算。 |
| 分配决策 | 某一策略步骤对候选、排序、跳过或选中结果的追加式审计记录，需能解释为何分配给该门店或为何进入总部池。 |
| 跳过策略 | 策略被禁用、无资格候选或不满足前置条件时记录的决策结果，跳过不创建空轮次。 |
| 正式分配 | 会更新主线索池位置、创建真实分配轮次并改变门店责任的引擎执行模式。 |
| 分配试运行 | 使用真实规则和候选计算但不改变池位置、不创建正式轮次且不授予门店责任的预览执行模式。 |
| 分配批次 | 对一组线索执行试运行、正式分配或受控重建的可追溯运行单元，用于记录范围、规则版本、结果和执行状态。 |
| 幂等分配 | 同一线索、同一状态迁移和同一执行意图重复触发时不得重复创建轮次或重复改变池位置。 |

## 跟进、SLA 与保护期术语

| 术语 | 定义 |
|------|------|
| 跟进记录 | 门店在当前有效轮次内提交的一次动作、结论/备注、操作人和时间的独立流水，删除时也必须保留审计证据。 |
| 最近跟进摘要 | 轮次和线索中心订单视图上冗余保存的最新跟进时间及结果，只用于查询展示，不替代跟进记录流水。 |
| 有效跟进行为 | 五类允许动作中的任意一次成功提交，均计为已产生跟进；“未联系上”也属于有效跟进行为。 |
| 保护型跟进行为 | 已预约、待进一步跟进或未联系上中的首次动作，会结束首次跟进 SLA 并启动该轮唯一一次核销保护期。 |
| 已预约 | 门店已与客户形成到店或服务预约的动作，计为已跟进并启动保护期。 |
| 待进一步跟进 | 门店已联系或处理但仍需后续沟通的动作，计为已跟进并启动保护期。 |
| 未联系上 | 门店实施了联系但未接通客户的动作，计为已跟进并启动保护期。 |
| 暂不需要（线索战败） | 门店判断客户当前不再需要该线索服务的动作，提交后当前轮立即失效并进入下一启用策略。 |
| 客户要求换门店 | 客户明确要求由其他门店继续处理的动作，提交后当前轮立即失效、原门店进入历史排除并进入下一启用策略。 |
| “跟进成功” | V1 不使用的跟进动作或店端状态名称；是否已产生跟进、是否处于保护期和是否最终核销必须分别判断。 |
| 首次跟进 SLA | 从分配时间起允许门店在没有任何跟进行为时完成首次动作的时限，默认 24 小时，配置为无限时不产生 SLA 到期。 |
| SLA 到期 | 当前轮在首次跟进 SLA 结束前仍无任何跟进行为的事件，会关闭该轮并进入下一启用策略。 |
| 核销保护期 | 首次保护型跟进行为后为当前责任门店保留线索的时段，默认 7 天，期间订单仍未核销不触发普通 SLA 轮转。 |
| 保护期起点 | 当前轮首次保护型跟进行为的提交时间，后续跟进不会重新计算或延长保护期。 |
| 保护期到期 | 保护期结束且订单仍未核销或退款的事件，会关闭当前轮并进入下一启用策略。 |
| 再分配触发 | 战败、客户要求换店、SLA 到期或保护期到期使当前轮关闭并从下一启用策略继续分配的状态迁移。 |
| 末轮结束 | 最后一个启用策略完成后仍未形成可继续的门店轮次，活跃线索进入总部池。 |
| 状态迁移优先级 | 订单已核销/已退款最高，其次是即时关闭动作，再其次是 SLA 或保护期定时事件，以避免同一轮被并发重复处理。 |

## 联系方式与权限术语

| 术语 | 定义 |
|------|------|
| 脱敏手机号 | 默认展示为中间四位星号的手机号，例如 `138****5678`，可用于识别客户但不能直接拨打。 |
| 完整手机号 | 后台保存的可联系明文手机号，仅在服务端确认当前有效轮次、责任门店和账号权限后按次返回。 |
| 查看完整手机号 | 将当前线索的完整手机号临时展示给获授权门店用户的受审计操作，线索失效后必须立即回收前端缓存。 |
| 复制完整手机号 | 在与查看完整手机号相同的服务端授权下将明文写入剪贴板的受审计操作，不得复制脱敏号后误报成功。 |
| 联系方式操作权 | 当前有效轮次所属门店中获授权用户查看、复制或导出该线索完整手机号的权限，管理范围可见不自动授予该权限。 |
| 跟进操作权 | 当前有效轮次所属门店中获授权用户提交跟进动作和备注的权限，历史门店、普通管理员和只读管理角色不能代操作。 |
| 历史只读权 | 曾经负责该线索或具备管理数据范围的角色查看脱敏联系方式、轮次和跟进历史的权限，不包含完整号码或编辑能力。 |
| 最高管理员 | 可配置、发布、试运行、重建、审计和执行明确授予的高风险管理操作的角色，不默认作为门店跟进执行者。 |
| 普通管理员 | 可查看规则、运行状态和审计信息但默认不能写入配置或代替门店跟进的后台角色，最终权限由 DYDATA-32 统一配置。 |
| 操作审计 | 对手机号查看/复制/导出、规则发布、重建、跟进记录删除等敏感动作记录操作者、对象、时间、结果和必要原因的不可抵赖流水。 |

## 经营指标术语

| 术语 | 定义 |
|------|------|
| 总体核销率 | 指定下单周期完整主池中历史上曾达到已核销状态的唯一订单数除以同周期完整主池全部唯一订单数。 |
| 完整主池订单数 | 指定下单周期进入完整主池的唯一订单数，包括首次采集已核销、已退款、总部池、门店跟进池和关闭订单。 |
| 核销数 | 筛选范围内历史上曾达到已核销状态的唯一订单数，同一订单无论有多少轮次或跟进记录只计算一次。 |
| 线索跟进率 | 筛选范围内已产生至少一次有效跟进行为的实际分配轮次数除以同范围实际分配轮次数，不得称为“跟进成功率”。 |
| 24 小时有效跟进率 | 在分配后 24 小时内产生任一有效跟进行为的轮次数除以已经到达 24 小时观察点的轮次数，是门店综合评分的默认组成项。 |
| 门店核销转化能力 | 实际分配给该门店并进入可比较成熟窗口的唯一订单中最终核销订单的比例，是门店综合评分的默认组成项。 |
| 自动分配成功率 | 成功进入门店有效轮次的活跃线索数除以进入正式分配判断的活跃线索数。 |
| 保护期到期率 | 保护期届满且仍未核销的轮次数除以已经进入保护期且达到观察点的轮次数，用于诊断跟进质量。 |
| 成熟窗口 | 为等待订单后续核销或退款回补而设置的 30 天观察期，未满时指标标记为观察中，期满后标记为已成熟。 |
| 基线周期 | 2026-01-01 至 2026-06-30 的成熟订单合集，用于计算上线前总体核销率基准，不取各月百分比简单平均。 |
| 过渡月 | 正式运营启动所在自然月，不纳入上线前后正式效果比较。 |
| 首期观察周期 | 过渡月之后连续三个已成熟完整自然月，其合并总体核销率用于判断首期是否高于基线。 |
| 门店指标分母 | 仅包含实际分配给该门店的正式轮次和订单，总部池、试运行和其他门店历史不得计入。 |

## 页面与运行治理术语

| 术语 | 定义 |
|------|------|
| 线索看板 | 按账号数据范围展示完整主池、门店跟进池、总部池、跟进和核销表现的只读经营页面。 |
| 线索明细 | 展示账号可见线索及当前业务状态、分配轮次、商品和时间信息，并进入跟进详情的任务列表。 |
| 线索跟进详情 | 汇总联系方式、商品订单、真实分配轮次和每轮跟进历史，并仅向当前责任门店提供当前轮操作的详情工作台。 |
| 分配规则页面 | 供最高管理员维护、发布和审阅规则版本、作用范围、策略顺序及参数，普通管理员只读的后台页面。 |
| 分配记录页面 | 按批次和线索追溯规则版本、策略步骤、候选、评分快照、决策结果和实际轮次的审计页面。 |
| 总部线索池页面 | 供总部角色查看当前总部池库存、入池原因和来源证据的只读页面，V1 不提供再次投放。 |
| 线索重建 | 在明确范围和批次下依据原始数据重新生成主记录及派生视图的高风险维护动作，必须幂等、可审计并在执行前试运行。 |
| 正式数据集 | 仅包含正式引擎产生的主线索、池位置、分配轮次和状态事件的数据集合，排除试运行和旧引擎结果。 |
| 旧分配引擎 | 以抖音归属或旧物化逻辑生成 `execution_mode=legacy` 轮次的冲突路径，当前未投入生产，需由 DYDATA-34 全面删除且不保留双写。 |
| 新分配引擎（自有分配引擎） | 按已发布规则版本、固定策略模块、门店资格和评分快照执行正式分配并产生唯一真实轮次的权威引擎。 |

## 同义词与禁用表达

| 不采用表达 | 统一表达 | 原因 |
|------------|----------|------|
| 第 0 轮 | 销售店优先策略步骤；若选中则创建第 1 轮 | 策略步骤不等于业务轮次，轮次必须从 1 连续编号。 |
| 抖音分配门店 / 抖音归属门店 | 抖音跟进归属 | 该字段不是本系统当前责任门店的来源。 |
| 处理范围 | 跟进门店 | 轮次记录应展示真实门店名称，不使用“本店/其他店”等笼统称呼。 |
| 跟进成功率 | 线索跟进率或 24 小时有效跟进率 | 产生跟进行为不等于最终核销，也不等于业务成功。 |
| 跟进成功 | 已预约 / 待进一步跟进 / 未联系上等具体动作 | V1 使用五类明确动作，不保留含义模糊的成功枚举。 |
| 交易关闭 | 已核销或已退款 | 当前业务只按已确认的两类订单终态对店端解释。 |
| 线索有效 / 无效 | 可跟进、已关闭或具体轮次状态 | 必须说明是订单生命周期、池位置还是轮次权限，不能混成单一布尔值。 |
| 分配轮次明细 | 线索明细 | 主列表面向门店任务，不以内部轮次记录作为页面名称。 |
| 线索当前归属 | 当前责任门店 | 内部责任由自有分配轮次决定，不展示抖音归属。 |

## 待下一阶段落模的边界

以下内容不改变本术语表，但必须在 Schema/API 阶段给出唯一技术表达：

1. `lead_key` 与 `order_id` 的约束、缺失订单 ID 的保留与隔离方式。
2. 订单状态、线索生命周期状态、池位置、分配状态、轮次状态和店端显示状态的枚举及迁移矩阵。
3. 正式分配、试运行、重建和旧引擎数据的物理隔离与唯一性约束。
4. 五类跟进动作、SLA、保护期、终态事件和并发触发的幂等键。
5. 管理数据范围、当前责任门店、手机号权限和跟进操作权的服务端授权模型。
6. 完整主池、成熟窗口、总体核销率和门店评分的事实表及快照口径。
