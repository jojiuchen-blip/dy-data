# 数据库任务规则

## 适用范围

适用于 PostgreSQL、SQLAlchemy 模型、Alembic 迁移、查询优化、种子数据、数据回填和生产数据核对。

## 事实与环境

- 当前结构以 Alembic 迁移、SQLAlchemy 模型和目标数据库实际 schema 的交集为证据。
- 先明确本地、测试、开发或生产环境；不得把一个环境的行数和状态当成另一个环境的结论。
- 查询真实环境时只读取任务必要字段，输出必须脱敏。

## 结构变更

- 所有可持久化结构变更通过 Alembic 提交，模型与迁移在同一变更中保持一致。
- 迁移需评估旧数据、默认值、可空性、唯一约束、索引、锁表时间和回滚或前滚恢复方案。
- 删除列、收紧约束、大表回填和批量重算属于高风险操作，必须有独立步骤、备份判断和明确授权。
- 不修改已经在共享环境执行过的历史迁移来伪造新状态；追加新迁移修正。

## 查询与写入

- 查询必须体现门店范围、角色权限和业务时间口径，避免无界全表扫描和逐行重复查询。
- 新增筛选、排序或关联前检查索引；性能问题用 PostgreSQL 执行计划和真实规模证据判断。
- 金额继续使用现有整数最小货币单位或精确数值口径，不使用浮点数保存财务值。
- 回填和重建脚本应可分批、可观察、可重跑，并记录处理、跳过和失败数量。
- 配置、枚举和规则版本由数据库驱动时，接口和页面不得另建冲突常量。

## 最小验证

- 运行迁移升级并验证目标 schema；高风险变更还需验证恢复路径。
- 对新增约束、索引、回填和聚合口径补 pytest 或 SQL 证据。
- 写入后从数据库或 API 回读，确认事务提交后的值和页面消费口径一致。

## 禁止行为

- 提交真实数据库 URL、备份、导出数据或生产标识。
- 直接在生产执行未审阅的 DDL、全表更新或重建。
- 以“本地测试通过”替代目标环境 schema 与数据验证。
