# 前端开发规则

## 适用范围

适用于 `apps/web/` 的 React、TypeScript、Vite 页面、组件、请求封装、状态和交互。

## 开始前

1. 确认当前 Linear issue、页面角色、门店范围、验收场景和相关 API。
2. 读取 `docs/design-system/README.md`、`docs/design-system/tokens.json` 及现有同类组件。
3. 明确哪些值由后端计算、哪些只负责展示；禁止在页面复制结算、分配或权限规则。

## 数据与权限

- 默认使用真实 API；模拟数据只有在显式启用 `VITE_USE_MOCKS=true` 时可用，且不得静默覆盖真实错误。
- 请求沿用共享 client、`credentials: include` 和统一错误处理，不在页面散落 URL 或认证逻辑。
- 前端 `AuthGate` 和菜单可控制体验，但安全边界必须由后端角色与门店范围校验。
- 接口字段变化同步 TypeScript 类型、加载逻辑、空态和相关测试；不以 `any` 长期绕过契约。
- 可保存页面必须完成“加载 → 修改 → 提交 → 重新加载 → 回显一致”的真实链路验证。

## 组件与视觉

- 页面负责请求与编排，组件负责展示与交互，共享 hook 或 client 负责复用状态和请求。
- 优先复用现有组件、令牌和布局模式；不得另建一套颜色、字号、圆角、阴影或状态语义。
- 新页面必须覆盖加载、空数据、错误、无权限和正常状态；交互控件需有禁用、进行中和失败恢复反馈。
- 表格、图表和指标必须说明筛选、时间、单位和数据范围，金额与百分比格式遵循现有口径。
- 响应式改动不得破坏管理后台的信息密度和关键操作可达性。

## 最小验证

- `npm --prefix apps/web run build` 通过。
- 涉及 API 的页面至少用真实后端验证一次主要成功场景和一个失败或空数据场景。
- 涉及权限时验证至少两种角色或门店范围，确认不可见与后端拒绝一致。
- 涉及视觉改动时检查主要视口、长文本、窄宽度和关键交互状态。

## 禁止行为

- 在前端补服务端业务规则或权限真相。
- 以本地默认值掩盖接口字段缺失、失败或模拟回退。
- 只跑构建或只看到页面打开就宣称联调完成。
- 绕过项目视觉系统新增相互冲突的局部规范。
