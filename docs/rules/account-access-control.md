# 账号、页面权限与门店范围规则

本文件是 dy-data 账号授权模型的长期规则。具体开发范围、优先级、负责人、验收和状态以 Linear `DYDATA-32` 为准；本文件只维护后续页面、接口和账号管理都必须遵守的稳定边界。

本规则与 Linear `DYDATA-21` 的分佣比例、金额计算完全解耦。

## 1. 角色模型

系统业务角色仅保留三类：

| 角色 | 页面权限 | 门店范围 | 权限管理能力 |
|---|---|---|---|
| 最高管理员 | 固定拥有全部已注册页面，不可取消 | 固定全部门店，不可修改 | 配置管理员和门店账号的角色默认权限、单账号差异和门店范围；管理全部账号 |
| 管理员 | 角色默认权限加单账号差异 | 默认全部门店，最高管理员可改为一个或多个指定门店 | 配置门店账号的角色默认权限、单账号差异和门店范围；不得管理最高管理员或其他管理员 |
| 门店账号 | 角色默认权限加单账号差异 | 一个或多个明确门店 | 不得配置权限或扩大自身范围 |

- 不再设置 `viewer`、运营、区域管理员等独立角色；区域和多门店差异使用门店范围表达。
- 管理员的账号管理范围本期不随其业务数据范围收缩；管理员仍可管理全部门店账号。若未来引入区域化管理边界，必须另行定义并更新本规则。
- 新建或编辑管理员时，必须选择“全部门店”或至少一个指定门店，不允许保存空范围。
- 最高管理员与管理员之间的高风险操作差异由系统固定，不进入可配置操作矩阵。

## 2. 页面权限模型

账号的页面有效权限按下式计算：

```text
页面有效权限 = 角色默认权限 + 单账号额外允许 - 单账号额外禁止
```

- 最高管理员固定全选，不保存单账号页面差异。
- 管理员、门店账号默认继承各自角色权限。
- 单账号只保存相对默认值的“额外允许”和“额外禁止”，不得复制一套完整角色权限作为账号权限。
- 修改角色默认权限前，必须显示变更内容和受影响的继承账号数量并要求确认。
- 完全继承的账号自动使用新默认值。
- 已自定义账号的最终有效权限保持不变；系统应基于新默认值重新计算差异。
- 不提供批量将所有自定义账号重置为新默认值的操作；允许在单个账号上“恢复角色默认权限”。
- 页面权限保存后立即生效；旧会话必须重新校验，不能继续使用变更前权限。

账号管理页面保持两个页签：

1. `账号列表`：账号、角色、状态、门店范围、继承状态和单账号差异。
2. `角色权限`：纵向展示最高管理员、管理员、门店账号，横向展示当前已注册页面。最高管理员为只读全选；其他角色由有权管理者勾选。

本期不把新增、编辑、删除、导入、导出、审核等页面内动作拆成第三层可配置矩阵。页面内现有业务动作继续遵循固定角色规则和既有业务校验。

## 3. 页面登记和准入

只有可独立进入的实际页面或路由作为权限项。菜单分组不是独立权限项；账号拥有任一子页面时，对应菜单分组自动显示。

当前页面权限登记基线：

| 编号 | 页面 | 路由或别名 |
|---|---|---|
| A01 | 线索看板 | `/clues` |
| A02 | 线索明细 | `/clues/details` |
| B01 | 全国门店榜单 | `/ranking` |
| B02 | 单店结算 | `/settlement` |
| B03 | 订单费用明细 | `/details` |
| C01 | 核销表现 | `/sales` |
| D01 | 后台首页 | `/admin` |
| D02 | 账号管理 | `/admin/accounts` |
| D03 | 分佣规则 | `/admin/rules`、`/rule-admin` |
| D04 | 商品口径 | `/admin/product-types` |
| D05 | 线索分配规则 | `/admin/clue-allocation`、`/admin/clue-allocation/rules` |
| D06 | 分配试运行 | `/admin/clue-allocation/trial` |
| D07 | 分配记录 | `/admin/clue-allocation/records` |
| D08 | 总部线索池 | `/admin/clue-allocation/headquarters` |
| D09 | 用户建议 | `/admin/feedback` |
| D10 | 数据同步 | `/admin/sync`、`/sync-admin` |

- 登录、激活、忘记密码和个人设置属于通用账号能力，不进入页面矩阵。
- 尚未形成正式生产页面的月结、SAP、台账、批量开票以及订单分佣开票子页面 B04～B12 暂不登记。
- 新页面、独立子页面或新路由必须先登记稳定的 `page_key`，并关联正式路由、旧别名、专属 API、导出能力和是否包含门店数据。
- 最高管理员对新页面固定允许；管理员和门店账号默认拒绝，必须在角色权限页面明确配置后才能开放。
- 未登记页面不得形成可访问入口；下线、合并、改名时必须同步维护权限目录和历史显示名称。
- 前端菜单隐藏、前端路由拦截、后端 API 和导出必须使用同一权限结果。前端隐藏不能替代服务端拒绝。

## 4. 门店范围

门店范围显式表达为：

- 全部门店。
- 一个或多个指定门店。
- 无门店数据权限。

空门店清单绝不能解释为全部门店。

- 最高管理员固定全部门店。
- 管理员默认全部门店；最高管理员限制管理员范围时必须至少选择一个门店。
- 后台手工新建门店账号时，必须选择至少一个门店，否则不能保存。
- 门店账号可以在激活后由有权管理员调整为多门店。
- 查询、汇总、下钻、明细、导出和 API 必须使用同一门店过滤逻辑。
- 前端选择器只能展示账号有权查看的门店；后端不得信任请求直接传入的 `store_id`。
- 页面没有门店维度时只校验页面权限；页面包含门店数据时同时校验页面权限和门店范围。

## 5. 门店账号激活兼容

后台创建和门店自助激活是两条不同路径：

### 后台手工创建

- 管理员填写账号、密码并选择至少一个门店。
- 保存后账号立即处于已激活状态，可以直接登录。
- 已由后台手工创建并激活的门店再次进入自助激活时，应返回“已激活”，引导登录或重置密码。

### 门店自助激活

- 门店使用账户所属 ID 和所属账户关联 POI ID 完成同记录核验。
- 不要求管理员预先创建账号或选择门店。
- 系统根据核验结果自动创建或初始化门店账号，并绑定核验出的唯一门店。
- 自助激活阶段不得由门店自行选择其他门店；需要多门店时，激活后由有权管理员调整。
- “未激活门店”表示尚未产生已初始化门店账号的门店，不等于已存在一个空门店范围账号。

## 6. 账号迁移

- 系统初始应具备两个有效最高管理员，允许后续创建更多最高管理员。
- 环境配置管理员迁移为最高管理员。
- 现有数据库 `admin` 全部迁移为最高管理员；迁移后最高管理员数量可以多于两个。
- 现有 `viewer` 迁移为管理员，并按迁移前实际访问能力生成初始权限。
- 现有 `store` 迁移为门店账号，保留原 `user_store_scopes`；缺失范围显式标记为无门店权限并进入异常清单。
- 管理员初始角色默认权限来自原 `viewer` 的实际能力，门店账号初始角色默认权限来自原门店账号的实际能力。
- 迁移前的特殊差异转换为单账号额外允许或额外禁止。
- 迁移必须可重复、可回滚，并输出角色数量、门店范围数量、异常账号和前后对照。

## 7. 最高管理员生命周期

- 所有最高管理员拥有一致且不可单独调整的页面权限和门店范围。
- 任意一名最高管理员可以直接创建新的最高管理员、重置其他最高管理员密码或使其他最高管理员失效，不需要第二人确认。
- 最高管理员不得使自己失效。
- 最高管理员不支持直接降级为管理员；需要时先创建对应管理员，再使原最高管理员失效。
- 最高管理员失效后立即退出登录并禁止再次登录；账号不物理删除，保留历史关联。
- 上述敏感操作必须写入权限变更记录。

## 8. 权限变更记录

本规则中的操作审计仅指账号和权限变更历史，不扩展为订单、线索等业务操作日志。

必须记录：

- 创建、停用账号和角色变更。
- 角色默认页面权限变更。
- 单账号页面差异变更及恢复默认。
- 门店范围变更。
- 密码重置事件，但不得记录密码内容。
- 最高管理员创建、密码重置和失效。
- 操作者、操作时角色、时间、对象、变更前、变更后和执行结果。

查看范围：

- 最高管理员查看全部权限变更记录。
- 管理员只查看门店账号相关记录。
- 门店账号不可查看。

账号管理页提供“变更记录”入口，不新增第三个页签。记录支持按账号、操作者、时间和操作类型筛选；第一版不提供导出。记录长期保留，不允许业务账号修改或删除；不得记录密码、token、cookie 等秘密信息。

## 9. 授权判定

```text
有效访问 = 账号状态有效 ∩ 页面已授权 ∩ 请求门店属于授权范围
```

任一条件不满足时，前端不得展示入口，后端必须拒绝请求。角色、页面权限、门店范围、账号状态或密码变化后，权限结果立即生效，旧会话必须失效或重新校验。
