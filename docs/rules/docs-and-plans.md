# 文档与计划规则

## 权威分工

- Linear：需求池、范围、优先级、负责人、验收和 issue 状态。
- `project-profile.md`：项目身份、业务范围、阶段和当前缺口快照。
- `docs/plans/execution-plan.md`：当前 issue 的执行驾驶舱，不复制 Backlog。
- `docs/plans/delivery-plans/`：S3 后由套包生成的正式交付计划组。
- `docs/governance/authority-map.md`：现有文档的 authority、evidence、legacy、stale 和 missing 角色。
- `docs/devlog/`：按天追加的执行记录，不替代上述状态源。

## 修改规则

- 文档事实必须回溯当前代码、配置、真实环境或用户确认，不能从旧 README 或历史计划层层转述。
- 改产品范围时同步检查 README、项目产品介绍书、项目画像、架构和权威映射。
- 改路由、接口、模型、部署变量或运行入口时，全局搜索并更新相关当前文档与示例。
- 历史计划保留当时事实；发现过期时标注角色或链接当前 authority，不重写历史来伪造当时状态。
- 真实基础设施标识、生产 URL、IP、凭据和个人路径只允许出现在受控外部配置，不进入仓库文档。
- `【用户确认】`、`【系统推断】` 和 `【主入口回写】` 必须保持来源边界；扫描结果不得伪装成用户确认。

## 计划与状态

- 实现前确认有效 Linear issue；新增需求先入池，再经用户确认进入开发。
- issue 状态改变时同步当前执行驾驶舱；完成标准变化必须写回 Linear，不只留在聊天或日志。
- 只有真实完成并验证的事项可标记完成；构建通过不等于业务验收通过。
- 文档新增或拆分后刷新项目链接索引；S0.5 baseline 必须先 dry-run、人工复核，再正式写入。

## 最小验证

- 检查文档交叉引用、路径、命令和状态是否可定位。
- 运行全局文件校验、阶段路由、项目链接索引和治理测试。
- 搜索旧产品名称、过期架构说法和敏感基础设施标识，确认无当前文档继续引用。
