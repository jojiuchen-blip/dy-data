# 调试与故障排查规则

## 适用范围

适用于接口异常、页面错误、数据偏差、任务失败、部署故障、性能退化和状态不一致。

## 取证顺序

1. 明确现象、期望、影响范围、首次出现时间和可复现条件。
2. 核对当前 Linear issue、验收口径、最近提交与部署版本。
3. 明确本地、测试、开发或生产环境，以及实际配置来源。
4. 沿“源数据 → 数据库 → 服务/worker → API → 前端”逐层比较输入输出。
5. 将问题归类为需求或规则、实现、数据、配置、部署或状态回写问题，再决定修改点。

## 证据要求

- 优先读取真实日志、任务状态、接口响应、数据库记录和当前部署证据，不凭历史文档猜运行结果。
- 日志查询只取必要时间窗和字段，始终脱敏密钥、Cookie、数据库 URL、手机号和个人信息。
- 数据偏差必须给出时间、筛选、门店、状态、去重和金额单位等口径。
- 多层系统每层都要回答“实际输入、实际输出、从哪里开始偏离”。
- 若无法访问目标环境，明确标注未验证边界，不能把本地推断写成生产结论。

## 修复与回归

- 先建立能失败的自动化测试、查询或稳定复现步骤，再修复根因。
- 修复后反向验证上游输入和下游消费，防止只消除表面症状。
- 检查迁移、缓存、worker、页面和文档是否需要同步更新。
- 回写 Linear 状态、验收证据、执行计划和开发日志；未验证项保持未完成。

## 禁止行为

- 连续进行无证据的试探式补丁。
- 用前端兼容掩盖源数据或接口契约错误。
- 修完代码却不验证目标环境或不回写状态。
- 在终端或文档中扩散敏感运行信息。
