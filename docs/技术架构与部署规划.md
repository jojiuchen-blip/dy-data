# 技术架构与部署说明

> 本文是部署选择与演进说明。当前组件和边界以 `docs/architecture.md` 为准，实际生产步骤以 `docs/runbook.md`、`deploy/` 和 GitHub Actions workflow 为准。

## 1. 当前技术栈

- Python + FastAPI + Pydantic 提供 HTTP API。
- SQLAlchemy + PostgreSQL 承载业务持久化，Alembic 管理迁移。
- React 19 + TypeScript + Vite 提供需要登录的业务与管理页面。
- worker 执行采集、刷新、重建和浏览器导出任务。
- Docker Compose 编排数据库、迁移、API、worker、Web、浏览器和反向代理。

上述目录和服务均已存在；当前主形态不是 CSV 加静态 HTML，也不是只包含三个公开结算页面。

## 2. 部署模型

### 单机 Compose

`deploy/compose.yaml` 是当前可复现的单机生产骨架。只有 proxy 对外发布端口，其他服务位于内部 Docker 网络。TLS 可以在 proxy 或上游负载均衡终止。

### GitHub Actions

仓库同时保留托管平台与自管服务器的部署 workflow。实际启用哪条路径由仓库变量、secrets 和环境保护规则决定；仓库文档只记录变量名与操作方式，不保存真实项目 ID、服务器地址或线上 URL。

## 3. 配置与密钥

- 生产配置来自环境变量、平台 secrets 或服务器未跟踪 `.env`。
- `config.local.json`、`deploy/.env`、Cookie、浏览器 profile、下载目录、导出数据和数据库 URL 不提交。
- 部署模板使用 `CHANGE_ME_*` 或通用变量占位，敏感值只在受控环境注入。

## 4. 数据与迁移

- Compose 通过一次性 `migrate` 服务执行 `alembic upgrade head`，成功后再启动 API 和 worker。
- schema 变更必须追加 Alembic 迁移，并评估回填、索引、锁和恢复策略。
- 数据采集和刷新由 worker 负责；任务状态进入数据库供运行核对，不由前端临时脚本承担。

## 5. 安全边界

- 所有业务页面和明细导出需要登录。
- 管理接口和浏览器入口使用更严格的后端角色保护。
- 门店数据范围在后端执行；前端隐藏入口不能作为授权控制。
- 对外仅开放代理入口，不直接暴露 PostgreSQL、API、worker、noVNC 或 Chromium CDP。

## 6. 发布与回滚

发布至少包含治理门禁、pytest、前端构建、Compose 配置检查和数据库迁移判断。部署失败时先保留日志与版本证据，再根据变更类型回滚应用镜像或以前滚迁移恢复；不得直接修改历史迁移或生产表伪造状态。

具体命令、健康检查、备份、恢复、浏览器登录态和数据任务操作见 `docs/runbook.md`。

## 7. 后续演进原则

- 当单机资源、任务隔离或发布频率成为瓶颈时，再将数据库、worker 或浏览器采集拆分到独立基础设施。
- 拆分前先建立监控数据和明确容量问题，不因文档设想提前增加复杂度。
- 正式 BRD、页面说明、FOUNDATION 和 PRD 建立后，应将业务与接口决策迁移到套包权威链，本文继续只维护部署和演进说明。
