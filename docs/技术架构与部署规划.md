# 技术架构与部署规划

## 一、目标

本规划用于约定抖音订单分账数据看板的技术栈、生产数据采集方式和未来云服务器部署方向。核心原则是：生产数据链路必须由服务器上的稳定服务运行，不能依赖开发者个人电脑，也不能依赖 Codex 或其他 agent 定时执行。

Codex 可以用于开发、调试、生成代码、排查数据异常和辅助文档编写，但不能成为生产系统的一部分。

## 二、推荐技术栈

| 层级 | 技术选择 | 说明 |
| --- | --- | --- |
| 前端 | React + TypeScript + Vite | 用于构建页面 1、页面 2、页面 3 等数据看板。前端只通过 API 读取数据，不直接读取 CSV 或本地文件。 |
| 后端 API | Python + FastAPI | 现有采集和分账逻辑已经以 Python 脚本沉淀，后端继续使用 Python 便于复用业务口径。 |
| 数据库 | PostgreSQL | 作为订单、券、核销、退款、门店、POI、认证主体、抖音号/职人、SKU、分账比例、发票状态等数据的主存储。 |
| 定时任务 | Python worker / scheduler | 负责定时拉取抖音数据、导入人工维护数据、生成分账基础表和刷新汇总数据。 |
| 部署 | Docker Compose + Nginx 或 Caddy | 第一阶段采用单台云服务器部署，后续业务规模扩大后再考虑更复杂的编排方案。 |

参考官方文档：

- FastAPI Docker 部署：https://fastapi.tiangolo.com/deployment/docker/
- FastAPI 反向代理部署：https://fastapi.tiangolo.com/advanced/behind-a-proxy/
- Vite 生产构建：https://vite.dev/guide/build
- Vite 静态部署：https://vite.dev/guide/static-deploy
- Docker Compose：https://docs.docker.com/compose/
- PostgreSQL 文档：https://www.postgresql.org/docs/current/

## 三、生产数据链路

生产数据链路按以下方向设计：

```text
抖音开放平台 / 后台导出数据 / 财务维护数据
        |
        v
服务器定时任务 / worker
        |
        v
原始数据表 raw_*
        |
        v
清洗、匹配、分账计算任务
        |
        v
分账基础表 / 汇总表
        |
        v
后端 API
        |
        v
前端页面 1 / 页面 2 / 页面 3
```

生产环境中，前端页面只读取后端 API。CSV、JSON 和本地目录只作为过渡阶段的数据导入、人工排查或导出结果，不作为长期主数据源。

## 四、定时任务原则

定时任务必须满足以下要求：

- 在云服务器或受控运行环境中执行，不依赖开发者个人电脑在线。
- 不依赖 Codex、ChatGPT、脚本窗口或人工点击来完成每日更新。
- 使用环境变量或服务器配置读取密钥、账号、路径和数据库连接，不在代码中写死。
- 每次运行都记录任务状态，包括任务名称、开始时间、结束时间、运行结果、数据范围、成功条数、失败条数和错误信息。
- 支持幂等运行。同一订单、券、退款或门店记录重复拉取时，应通过唯一键更新，不重复插入。
- 任务失败不能静默。后台至少要能看到失败记录，后续可增加消息通知。

建议预留任务运行记录表：

| 字段 | 说明 |
| --- | --- |
| job_id | 任务运行 ID。 |
| job_name | 任务名称，如订单拉取、核销拉取、退款拉取、分账计算。 |
| started_at | 开始时间。 |
| finished_at | 结束时间。 |
| status | running、success、failed。 |
| date_range | 本次任务处理的数据时间范围。 |
| success_count | 成功处理条数。 |
| failed_count | 失败条数。 |
| error_message | 失败原因摘要。 |

## 五、服务拆分

第一阶段建议拆成以下服务：

```text
proxy
  - HTTPS 证书
  - 前端静态资源代理
  - API 反向代理

frontend
  - React/Vite 构建产物
  - 页面 1：全国门店销售情况榜单
  - 页面 2：单店月度分账看板
  - 页面 3：门店月度数据明细表

api
  - 登录保护
  - 看板查询 API
  - 明细查询 API
  - 导出 API
  - 任务状态查询 API

worker
  - 拉取订单数据
  - 拉取核销记录
  - 拉取退款记录
  - 导入抖音号/职人和门店映射
  - 导入财务维护的到票和审核状态
  - 生成分账基础表和汇总表

database
  - PostgreSQL 主库
```

第一阶段可以用 Docker Compose 在单台云服务器部署以上服务。后续如果数据量、访问量或任务量增长，再考虑拆分数据库、任务队列、对象存储或更复杂的编排方案。

## 六、仓库结构与协作边界

第一阶段建议采用轻量 monorepo，不拆分多个仓库。前端、后端 API、worker、共享业务口径和部署文件放在同一仓库内，通过目录边界降低协作成本。

目标目录结构如下：

```text
apps/
  web/        # React + TypeScript + Vite，前端页面
  api/        # FastAPI，查询、导出和任务状态 API
  worker/     # Python 定时任务，采集、清洗、汇总和入库
scripts/
  exports/       # 当前可运行的数据导出入口
  settlement/    # 当前可运行的分账基础表和看板生成入口
  diagnostics/   # 数据可用性、异常原因和字段检查入口
  exploration/   # 接口、字段、匹配关系的一次性探查脚本
  utilities/     # 辅助工具脚本
  tasks/         # PowerShell 本地任务入口
src/
  dy_data/       # 共享 Python 业务包，沉淀配置、客户端、IO 和分账口径
deploy/          # Docker Compose、Nginx/Caddy 和服务器部署配置
docs/            # 产品定义、技术架构、数据模型、接口契约和协作说明
tests/           # Python 和前端测试
```

当前阶段只整理已有脚本到 `scripts/`，不提前空建 `apps/web`、`apps/api`、`apps/worker` 和 `deploy`。这些目录在对应代码真正开始实现时创建，避免仓库出现只有目录没有责任人的空壳。

协作边界建议如下：

- 前端主要维护 `apps/web/`、页面状态、筛选交互、图表和 API 调用。
- 后端主要维护 `apps/api/`、`apps/worker/`、数据库模型、数据采集任务和部署运行配置。
- 双方共同维护 `docs/api-contract.md`、`docs/data-model.md` 和 `src/dy_data/settlement/` 中的业务口径。
- 历史探查脚本保留在 `scripts/exploration/`，除非已经确认进入生产链路，否则不直接迁入 `apps/worker/`。

第一阶段并行开发顺序建议如下：

```text
真实拉取数据字段盘点
        |
        v
docs/data-model.md 数据表草案
        |
        v
docs/api-contract.md API 返回结构草案
        |
        v
mock/ 脱敏示例数据
        |
        v
前端页面和后端 API 并行开发
        |
        v
worker 写入 PostgreSQL，API 读取真实库
        |
        v
Docker Compose 部署到云服务器
```

数据表草案应由后端和数据侧基于本地真实拉取结果主导，前端侧按页面展示和筛选需求校对字段是否够用。前端开发不依赖协作者本地数据，先读取 `mock/` 中的脱敏示例结构，等 API 契约稳定后再切换到后端接口。

## 七、配置和密钥

生产环境不得提交真实密钥。以下信息必须通过环境变量、服务器配置或云平台密钥管理注入：

- 抖音开放平台 `APP_ID`、`APP_SECRET`、`ACCOUNT_ID`。
- 数据库连接地址。
- 登录密钥或会话签名密钥。
- 文件存储配置。
- 飞书、多维表格或其他外部系统配置。

本地开发仍可使用 `config.local.json`，但生产服务必须能在没有本地个人路径的情况下运行。

## 八、数据库主数据方向

长期主数据建议进入 PostgreSQL。至少需要覆盖以下数据对象：

- 订单。
- 券。
- 核销记录。
- 退款记录。
- 门店。
- 认证主体。
- POI_ID。
- 抖音号/职人。
- SKU 和商品类型。
- 分账比例。
- 到票状态和发票审核状态。
- 任务运行记录。

飞书多维表格更适合作为人工维护界面，例如门店映射、到票状态、异常复核和财务确认，不建议作为系统主数据库。

## 九、上线前最小准备

第一阶段上云前至少需要完成：

- 后端脚本能在无本地固定路径的服务器环境运行。
- 数据采集任务能写入数据库。
- 分账计算能从数据库读取数据并写回汇总结果。
- 前端页面通过 API 查询数据。
- 服务器有登录保护，不对公网裸露数据。
- 任务运行有日志和状态记录。
- 失败任务可以重跑，不会产生重复数据。
- README 说明本地开发方式，部署说明放在 `docs/`。

## 十、暂不纳入第一阶段

以下事项不作为第一阶段必须完成项：

- Kubernetes。
- 复杂角色权限体系。
- 完整发票上传、文件存储和 OCR 识别流程。
- 自动化 CI/CD 全链路。
- 多云部署。
- 高可用数据库集群。

这些能力可以在数据链路稳定、页面 1-3 跑通、财务复核流程确认后再分阶段建设。
