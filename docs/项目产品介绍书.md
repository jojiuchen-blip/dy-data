# dy-data（抖音经营引擎）产品介绍书

## 一、产品定位

dy-data 是面向汽车经销商集团及门店的抖音经营数据平台。它将抖音订单、券核销、退款状态、门店与账号归属、销售表现、线索、跟进、分配规则和同步任务汇集到同一系统，为门店经营、集团管理、财务复核和后台运营提供统一工作台。

系统提供经营判断和分账参考，不执行真实资金划拨；涉及财务结论时必须能够回到订单、券、门店、账号和规则证据复核。

## 二、目标用户

- 门店经营人员：查看销售和线索，记录跟进，处理门店范围内的业务。
- 门店负责人：查看门店表现、订单明细、结算参考和线索执行情况。
- 集团财务：复核跨店核销、应收、应扣、净额与异常订单。
- 集团管理人员：比较门店表现，观察整体经营与线索分配效果。
- 授权管理员：管理账号、门店范围、规则、同步、反馈和数据运营任务。

## 三、核心业务域

### 1. 经营与结算

- 销售看板展示集团和门店的经营指标、趋势与商品类型表现。
- 门店排名支持按时间、门店和业务口径比较经营结果。
- 月度结算以订单和券明细为依据，识别销售归属门店、核销门店和跨店关系。
- 应收、应扣和净额是复核参考；退款、撤销核销、归属缺失、SKU 缺失等进入异常诊断。
- 订单明细和导出承接指标下钻，保证汇总结果可以追到具体业务记录。

### 2. 线索运营

- 线索总览和订单详情支持筛选、查看联系方式授权结果和跟进记录。
- 门店人员可以新增或删除允许范围内的跟进记录。
- 分配轮次、候选线索、总部池、试运行和审计记录支持管理员理解分配过程。
- 规则版本、门店组、门店评分和刷新任务共同形成可追溯的线索分配依据。
- 总部池用于承接缺少有效锚点、映射失败或不满足分配条件的线索，不伪造门店分配结果。

### 3. 后台管理

- 管理账号、角色、门店范围、初始化与密码重置。
- 管理 SKU 规则、非佣金归属账号和商品类型可见性。
- 查看并处理反馈，管理同步配置和触发受控同步任务。
- 管理线索分配规则、版本、发布/退役、门店组、试运行和重建。
- 管理能力必须由后端角色权限保护，不能只依赖前端菜单隐藏。

### 4. 数据平台

- worker 从抖音开放平台和受控浏览器采集链路获取必要数据。
- PostgreSQL 保存原始记录、维度映射、业务明细、汇总、线索与任务状态。
- FastAPI 提供认证、经营、结算、线索、反馈、任务和管理接口。
- React Web 提供需要登录的数据看板、业务工作台和管理后台。
- Alembic、Docker Compose、GitHub Actions、运行手册和监控日志支撑迁移与生产运行。

## 四、关键业务口径

- 销售归属与核销归属必须分别记录，不能用核销门店覆盖销售门店。
- 跨店核销由销售归属门店与核销门店的关系判断，结果必须可下钻复核。
- 金额沿用后端和数据库的精确口径；页面只负责格式化展示，不重新计算业务真相。
- 门店账号只能访问授权门店范围；管理员和最高管理员按后端权限访问管理能力。
- 线索分配结果、规则版本、评分快照和审计记录要能够说明“何时、依据什么、分配到哪里”。

## 五、当前页面与入口

当前 Web 路由覆盖登录、销售、门店排名、月度结算、订单明细、线索总览与详情，以及账号、SKU、商品可见性、同步、反馈和线索分配管理。页面清单以 `apps/web/src/App.tsx` 和其引用的页面组件为代码证据；页面存在不等于所有业务分支已经完成用户验收。

## 六、非目标与边界

- 不在系统内执行真实分账、付款或资金划拨。
- 不把浏览器 Cookie、开放平台密钥、数据库 URL、导出文件或真实个人信息提交到仓库。
- 不将模拟数据、历史静态 HTML 或一次性脚本视为生产产品主形态。
- 未形成正式 BRD、页面说明、FOUNDATION 和 PRD 前，现有代码与本文只构成 S0.5 既有项目证据链。

## 七、成功标准

- 经营指标与订单明细在同一筛选口径下可追溯。
- 跨店核销和分账参考能够解释并定位异常来源。
- 线索分配与跟进过程有明确状态、规则版本和审计证据。
- 账号、门店范围和管理操作受到后端权限控制。
- 数据采集、刷新、迁移和部署失败可观察、可诊断、可恢复。

## 八、当前产品形态

当前产品是由 React Web、FastAPI、PostgreSQL、worker、浏览器采集适配器和生产部署配置组成的多服务应用。真实 API 是前端默认数据源，模拟模式仅用于显式启用的受控开发；历史 CSV、脚本和静态产物仍可作为数据核对或兼容资产，但不再代表系统当前架构。
