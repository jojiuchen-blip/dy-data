规范如何生效

不是靠口头约定，而是靠 token、可视化预览和测试逐步绑定。

token -> preview -> decision -> check
1. 候选值 从现有 CSS、业务页面和问题清单提取，不凭空发明。
2. HTML 预览 把颜色、排版、组件、表格、移动卡片都画出来。
3. 人类确认 通过真实视觉效果讨论取舍，确认后再固化。
4. token 固化 确认值进入 tokens.json，并逐步成为实现来源。
5. 测试拦截 关键值和现有 CSS 绑定，避免协作时悄悄漂移。

图标体系

图标只作为动作、导航和状态的辅助信号，来源、尺寸和可访问名称必须统一。

@iconify/react · @iconify-icons/solar

库与入口

  • 项目唯一 UI 图标库为 @iconify/react + @iconify-icons/solar。
  • 业务代码统一通过 apps/web/src/components/SolarIcon.tsx 引用，不在页面里临时手写 SVG。
  • 默认变体为 Solar linear；查看、复制、退出、设置、详情等操作图标都走 linear。
  • bold-duotone 只用于移动底部导航激活态、一级导航强调、品牌或模块入口。
  • 同一语义有普通态和强调态时，在 SolarIcon.tsx 中拆成 cluesLine / clues 这类明确名称。
  • icon-only 控件必须提供可访问名称；装饰图标用 aria-hidden。
  • 规范页示例也必须引用与 SolarIcon.tsx 同名的 Solar symbol，不允许文本字形、CSS 箭头或重复手写 path。

移动端实现规则

  • 单列任务流：移动端独立实现，不把桌面右侧操作栏提前塞进同一个浮层布局。
  • 首屏判断：手机号、查看/复制、线索状态和当前轮次必须在首屏最前面。
  • 操作优先：跟进操作紧接联系方式，保存按钮全宽显示，触控目标不小于 44px。
  • 历史后置：订单商品和线索跟进历史保留完整信息，但放在跟进操作之后，避免店员先读审计信息。

明细工作台规则

  • 全局工作区：登录后的业务页面默认铺满 Shell 可用宽度，不再使用居中窄容器；具体阅读宽度由页面内卡片、表格或弹层自己控制。
  • 桌面端：明细工作台用于线索明细、订单明细这类高密度长表；在全宽基础上增加高度 calc(100vh - topbar) 和内部滚动。
  • 滚动归属：外层页面不纵向滚动，结果区使用 content-section--data-workspace，只有结果表格区域上下滚动；分页固定在结果区底部可见。
  • 筛选区：桌面优先单行密度，避免占用表格高度；移动端可折叠筛选并恢复自然页面滚动。
  • 适用边界：看板、后台表单、解释型页面使用全宽普通页面容器，但不套固定高度明细工作台。

协作约束

其他人参与开发时，也需要有明确边界，避免又回到散落硬编码。

guardrails
  • 新增颜色：除非同步更新 tokens.json 和预览 HTML，否则不要在组件里直接写新的 hex。
  • 新增组件：先判断它属于按钮、字段、状态、卡片、表格还是页面骨架，不要单独发明一套半径和阴影。
  • 新增长列表：桌面优先表格，移动端必须设计卡片或摘要，不把宽表直接塞进手机。
  • 新增明细长表：如果用户主要任务是筛选后逐行处理记录，桌面端使用明细工作台模板，保证分页可见且只让结果表格滚动。
  • 新增 sticky/fixed：必须声明对应 z-index 层级和导航避让间距，避免层叠互相遮挡。
  • 规范变更：先改 HTML 预览让人确认，再把值迁移到产品 CSS 和测试。
