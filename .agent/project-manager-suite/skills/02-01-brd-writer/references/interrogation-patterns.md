# 追问与鉴别模式

本文件在 Phase B（诊断）、Phase C（追问）、Phase D.5（前提挑战）执行时读取。

## 反奉承规则（强制）

苛刻不是表演，是信息密度。每句判断必须有立场和依据，禁止空话过关。

禁止说的话以及替代方式：
1. ❌"这个想法很有意思" → ✅ 必须表态：这行得通，因为……；或这行不通，因为……
2. ❌"有很多方式可以想" → ✅ 必须选一个方向，并说明什么证据能改变你的判断。
3. ❌"你可以考虑……" → ✅ 必须说"应该这样做，因为……"或"这条路走不通，因为……"
4. ❌"这个需求挺合理的" → ✅ 必须追问合理性依据：谁在什么场景下真的遇到了这个问题？
5. ❌"以后再优化" → ✅ 必须明确：是 Out Scope 有理由推迟，还是在回避范围定义。

底线：如果一段输出删掉后不影响任何决策，这段输出就不该存在。

## 需求真伪鉴别（Phase B 使用）

基于 project-profile.md 中已有信息，对需求进行真实性检验：

- **场景验证**：这个需求对应的用户场景是真实存在的，还是推测出来的？有没有具体的用户行为或反馈作为佐证？
- **痛点验证**：用户现在怎么解决这个问题的？如果什么都不做，用户会怎样？如果答案是"没什么影响"，需求真实性存疑。
- **来源追溯**：这个需求是从用户行为中观察到的，还是"老板说的""销售反馈的""竞品有的"？来源不同，可信度不同。
- **伪需求红旗**：以下特征中出现任意一个，需要追问——
  - "所有用户都需要" → 说不出具体是谁
  - "大家都觉得不好用" → 没有具体场景和数据
  - "别的工具有过类似功能" → 别的工具有不等于使用者需要
  - "领导/客户提了" → 一个人的意见不等于需求
  - "应该做" → 没有为什么的"应该"是伪需求

## 追问范式库（Phase C 使用）

当需求方回答触发以下模式时，必须使用对应的追问手法，而不是简单地"请再选一次"：

**范式 1：模糊角色 → 强制具体化**
- 触发："给运营人员用""给同事用""用户需要……"
- 追问："'运营人员'是哪岗？内容运营、活动运营、商品运营，还是客服？或者就是你自己 / 同岗位的人？他们日常用什么工具完成这件事？说一个具体的人和他的具体工作场景。"

**范式 2：无依据断言 → 要求证据**
- 触发："使用者肯定需要""大家都觉得""我们流程上很缺这个"
- 追问："'肯定需要'的依据是什么？是你观察到的使用者行为，还是有人跟你说过？说出具体的场景、人、时间。如果举不出来，这个需求需要重新审视。"

**范式 3：类比驱动 → 回归我们场景**
- 触发："XX 工具有这个功能""行业标配""别人都做了"
- 追问："别的工具有不等于我们的使用者需要。我们的使用者在什么场景下需要这个？如果那个工具没做这个功能，你还会做吗？如果答案是'不会'，这不是需求，是焦虑。"

**范式 4：未定义术语 → 精度要求**
- 触发："更好的体验""更高效""智能化""优化一下"
- 追问："'更好的体验'不是产品特性，是感受。具体是哪个操作步骤耗时太长？哪个信息找不到？哪个流程断了？用可度量的语言重新描述。"

**范式 5：范围膨胀 → 边界收敛**
- 触发："顺便也加上……""还需要支持……""最好也能……"
- 追问："这个是本次必须做的，还是'做了更好'？如果不做，核心场景能不能跑通？如果能，这个进 Out Scope，下个版本再评估。"

**范式 6：角色笼统 → 拆分角色与痛点**
- 触发："用户需要这个功能""提升用户体验""给用户用的"
- 追问："'用户'具体是谁？这个功能涉及几种角色？每种角色的痛点一样吗？比如操作的人关心效率，管理的人关心数据可控，被服务的人关心响应速度——你说的是哪个角色的什么痛点？"

## 前提挑战（Phase D.5 使用）

当 P0 字段确认率接近 100%、即将进入终稿阶段时，触发一次前提挑战。目的不是推翻已有决策，而是在锁定终稿前做最后一次"这件事本身对不对"的审视。

挑战四个前提（逐条向需求方确认，不可跳过）：
1. **问题对不对**：我们要解决的问题，是用户真正的痛点，还是我们替用户想出来的问题？用一句话重述问题，让需求方确认"是的，这就是要解决的问题"。
2. **不做会怎样**：如果这个功能/项目不做，用户会怎样？如果答案是"没什么大影响，现在也凑合能用"，需要重新评估优先级或收窄范围。
3. **已有方案覆盖**：当前系统、现有流程、已有工具，是否已经部分解决了这个问题？如果是，新方案的增量价值在哪里？要做的是完全重建，还是在已有基础上补缺口？
4. **方案有没有效**：需求方提出的解决方案，到底能不能解决用户的问题？这是识别"伪需求中的伪解决"的关键——**痛点是真的，但方案可能是需求方拍脑袋想出来的，与痛点之间没有必然的因果链**。挑战时要：
   - 重述方案的解决路径："你说要做 X，是认为 X 能让用户的痛点 Y 消失或缓解，对吗？"让需求方亲口承认这个因果假设。
   - 追问因果证据：方案为什么能解决问题？已经在小范围验证过吗？有类似案例吗？目标用户实际测试过吗？
   - 警惕信号：需求方答不上"为什么这个方案能解决"，或答案是"我觉得应该可以"、"先试试看"、"业内都这么做"，没有任何对自家用户/场景的实证。
   - 若方案与痛点之间是凭空跳跃或想当然 → 不是问题不真实，而是**方案选错了**，须回退 Phase C 重新挑选/设计方案，而不是带着无效方案进终稿。

处理规则：
- 四个前提全部确认通过 → 进入 Phase E。
- 任一前提被需求方否定或动摇 → 回退到 Phase C 继续收敛，不得强行推进终稿。
- 前提挑战在每次收敛周期中触发一次。若四个前提全部通过，不再重复触发；若任一前提导致回退 Phase C 并产生字段变更，重新逼近 100% 后须再次触发前提挑战——因为新方案的前提有效性需要重新审视。
