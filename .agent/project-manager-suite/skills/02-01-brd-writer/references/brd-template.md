# BRD 终稿模板

本文件在 Phase F 输出终稿时读取。终稿根据项目类型裁剪章节，"不适用"的章节不出现在终稿中，而不是留空。

**章节编号说明**：下方代码块里的 `## 1.`～`## 13.` 是**模板骨架编号**（跳过 §5、§7），只用于对照裁剪矩阵和 `chapters finalize --include` 的入参；终稿的实际章节编号以 `chapters finalize` 输出的 `heading_outline` 为准——保留下来的章节会从 1 开始连续重编号，不保留跳号。`save-brd` 按重编号后的序列校验，照抄模板骨架编号会校验失败。

## BRD 头部（所有类型必有）

```markdown
# BRD: <项目名称>

> 项目类型: 创新型 | 改造型 | 扩展型 | 集成型 | 运营型 | 合规型
> 生成时间: YYYY-MM-DD HH:MM
> Skill: brd-writer
```

## 附录：下游交接清单

> 下表用字段语义名引用，不硬编码章节号——章节号随项目类型裁剪会变化。终稿生成时，AI 根据实际终稿的章节编号填入具体引用。
>
> **裁剪规则**：若某行下游 skill 的必要读取字段在终稿中全部不存在（如集成型无页面定位，page-designer 和 page-explainer 无输入），则删除该行，不留空行。每行中标注"若有"的字段不影响整行去留——只要该行还有至少一个字段在终稿中存在，该行保留，缺失字段标注"本次不适用"。

| 下游 Skill | 需读取的字段（语义名） | 备注 |
|-----------|----------------------|------|
| page-designer | 利益相关角色、各角色痛点与场景、核心价值模型、页面定位（全部） | 据此拆页面和信息架构 |
| page-explainer | 核心价值主张（若有）、利益相关角色诉求、页面 / 入口定位 | 评判基准 |

## 各项目类型的章节裁剪规则

| 章节 | 创新型 | 改造型 | 扩展型 | 集成型 | 运营型 | 合规型 |
|------|--------|--------|--------|--------|--------|--------|
| §1 背景 | 必有 | 必有（改造动因） | 必有 | 必有（集成目的） | 必有 | 必有（法规要求） |
| §2 目标与指标 | 必有 | 改为：改造目标指标 | 必有 | 改为：集成目标 | 改为：效率目标 | 改为：合规达标标准 |
| §3 角色与场景 | 必有（全套） | 改为：利益相关角色+当前系统痛点 | 必有（全套） | 改为：利益相关角色+上下游系统画像 | 改为：内部用户角色+当前工作流 | 改为：利益相关角色+合规影响范围 |
| §4 价值主张 | 必有 | 不出现 | 必有 | 不出现 | 不出现 | 不出现 |
| §6 核心价值模型 | 必有 | 改为：改造价值模型 | 必有 | 改为：集成价值模型 | 必有 | 改为：合规达标模型 |
| §8 MVP 范围 | 必有 | 改为：改造范围（分期） | 必有 | 改为：集成范围 | 必有 | 改为：整改范围 |
| §9 备选方案 | 必有 | 必有（技术方案选型） | 必有 | 必有 | 视情况 | 视情况 |
| §10 前提假设 | 必有 | 必有（兼容性假设） | 必有 | 必有（第三方稳定性） | 视情况 | 视情况 |
| §11 风险与兜底 | 必有 | 必有（迁移风险） | 必有 | 必有（第三方风险） | 视情况 | 必有 |
| §12 项目周期 | 必有 | 必有 | 必有 | 必有 | 必有 | 必有（含合规 deadline） |
| §13 页面定位 | 视情况 | 视情况 | 视情况 | 不出现（无独立页面） | 必有（操作/配置/查看） | 视情况 |
| 附录 | 必有 | 必有 | 必有 | 必有 | 必有 | 必有 |

"视情况"指：根据对话中收敛的信息判断是否需要，不强制。
§5 市场与竞品差异化、§7 商业化路径不作为固定章节，上表的 §编号因此有跳号——它们只是模板骨架编号。终稿的实际章节编号由 `chapters finalize` 连续重编（1、2、3……不留跳号），附录下游交接清单也由脚本按重编号后的章节输出引用，下游 skill 读终稿时以终稿实际编号为准。
