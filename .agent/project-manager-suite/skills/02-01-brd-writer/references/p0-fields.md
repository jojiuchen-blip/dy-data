# P0 必须确认字段

本文件在 Phase A 项目定性后读取，用于确定当前项目类型的 P0 必填字段集。

## 字段分类说明

每个字段标注 `[决策]` 或 `[事实]`：
- **决策型**：涉及取舍、选择、判断的字段 → 台账必须记录方法论依据（Phase E 方法论门检查）。
- **事实型**：信息枚举、现状描述、客观记录 → 台账记录信息来源即可（如 `来源: project-profile` / `需求方确认` / `数据佐证`），不强套方法论。

## 通用 P0（所有项目类型必填，未确认不得终稿）

> **锁定时机**：字段 #1 在 **Phase A** 元字段阶段锁定；#2-#4（项目背景 / 利益相关角色 / 核心痛点）在 **Phase B 批量锁定**——信息已在 project-profile.md 中就绪，完成诊断 + 角色识别 + 真伪鉴别后统一入账；其余字段在 **Phase C** 按阻塞优先级单题追问后锁定。

1. `[决策]` 项目类型（创新/改造/扩展/集成/运营/合规）。
2. `[事实]` 项目背景（或等效动因：改造动因/集成目的/法规要求）。
3. `[事实]` 利益相关角色（枚举所有涉及的角色，标注需求提出者与使用者是否为同一人，标注角色间是否存在利益冲突；运营型时须覆盖内部用户角色）。
4. `[事实]` 核心痛点（每个关键角色的具体痛点，必须是可描述的场景而非笼统感受；改造型时须覆盖当前系统痛点（性能/成本/可维护性））。
5. `[决策]` 核心价值模型（业务价值：提效/提体验/提服务水平均可；或等效：改造价值模型/集成价值模型/合规达标模型）。
6. `[决策]` 范围定义（In Scope / Out Scope 或等效范围）。
7. `[事实]` 关键风险与兜底策略。
8. `[事实]` 项目周期（预计完成或落地时间）。

## 创新型追加 P0

11. `[事实]` 目标使用者与核心场景（JTBD 格式）。
12. `[事实]` 用户当前替代方案。
13. `[事实]` 需求验证证据。
14. `[决策]` 核心价值主张。
15. `[决策]` 核心衡量指标（指标名+计算公式+目标值+统计周期；承接通用"目标与成功标准"）。
16. `[决策]` 辅助业务指标体系（按业务对象选填：工单解决时长、操作员效率、响应时长、错误率、处理量等）。

## 改造型追加 P0

11. `[决策]` 改造目标指标（性能/成本/可维护性；承接通用"目标与成功标准"）。
12. `[决策]` 备选技术方案对比。

## 扩展型追加 P0

11. `[事实]` 目标使用者与核心场景（JTBD 格式）。
12. `[事实]` 需求验证证据。
13. `[决策]` 核心价值主张。
14. `[决策]` 核心指标（指标名+计算公式+目标值+统计周期；承接通用"目标与成功标准"）。

## 集成型追加 P0

> 集成型用于打通上下游系统、对接外部接口的项目，默认无独立页面。

11. `[事实]` 上下游系统画像。
12. `[事实]` 当前对接方式。
13. `[决策]` 集成目标（打通什么、省多少人力；承接通用"目标与成功标准"）。
14. `[决策]` 备选集成方案对比。

## 运营型追加 P0

11. `[事实]` 当前工作流。
12. `[决策]` 效率目标（承接通用"目标与成功标准"）。

## 合规型追加 P0

11. `[事实]` 当前合规差距。
12. `[决策]` 合规达标标准（法规条款映射；承接通用"目标与成功标准"）。
13. `[决策]` 整改范围与优先级。

## 页面定位全套字段（当 P0 要求时必填）

> 由 `has_pages` 派生字段控制展开：运营型恒展开，集成型恒不展开，其他类型默认展开（内部工具默认含页面）。
> 不再在各类型追加 P0 中单独列出指针行——由脚本层的 buildFieldSet 统一处理。

1. 项目覆盖对象（枚举所有端）。
2. 各端目标用户。
3. 各端主要用途。
4. 页面定位判断（操作/配置/查看）。
5. 下游待确认边界（一段简短的边界声明：按"类别/性质"粒度，说明哪些事项 BRD 不锁、交由下游 skill 去定。不列具体条目，不代替下游给出字段结构 / 数据模型 / 阈值数值 / 文案 / 视觉与交互动线等明细——那属于 PRD 及下游 skill 的职责）。
