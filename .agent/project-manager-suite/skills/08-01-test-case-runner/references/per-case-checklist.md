# 单条用例完成后检查清单

每条用例的四段式闭环（数据准备→测试→验证→清理）执行完毕后，**必须逐项完成以下动作再进入下一条**：

## 必做项

1. **写入报告** — 在 `reports/测试验收-{业务域}.md` 追加该用例的结果行。若文件不存在则按 `references/report-template.md` 创建业务域报告 + 更新 index.md
2. **更新统计** — 业务域报告头部的 `PASS / FAIL / BLOCKED` 计数同步更新
3. **核对按钮证据** — 如果本条用例涉及任何按钮操作（保存/刷新/搜索/提交/同步/打开弹窗等），报告中必须写明对应的弹窗 / toast 截图；若页面设计上不出现弹窗，也必须在报告中显式注明“未出现弹窗”
   - 若该按钮操作存在二次确认弹窗（如删除、确认提交、覆盖、同步执行），必须同时核对两张证据：`确认弹窗截图` + `结果 toast/结果弹窗截图`
   - 若交互过程实际验证过但留档缺少其中任一张，报告必须显式写“证据不完整，需补采”，不能把单张截图描述成完整过程证据
4. **FAIL 时双写** — 如果结果为 FAIL：
   - 业务域报告的"失败详情"节写 `> 详见 [defects.md](defects.md) #N`（业务域报告与 defects.md 同在 `reports/` 目录下）
   - 在 `reports/defects.md` 追加缺陷行 + 详情节（预期/实际/证据/可能原因）
   - 如果 defects.md 不存在，按 `references/defect-template.md` 创建

## 格式参考

报告结果行：
```
| TC-B1-01 | 单车用户 | API | ✅ PASS | HTTP 200, code=200, data.length=1 |
```

缺陷详情节：
```
### #N TC-XX-NN：缺陷标题
- **预期**: ...
- **实际**: ...
- **证据**: ...
- **可能原因**: ...
- **状态**: OPEN
```

## 进度简报

每 3 条用例完成后，向用户简报：`X/N 完成，M PASS，K FAIL`
