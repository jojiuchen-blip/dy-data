# 异常处理手册

执行过程中遇到异常时，查此表决定处理方式。

## 核心原则

**Playwright 模拟的是真人操作。如果 Playwright 点不到、滚不动、看不见，说明真实用户也做不到 — 这本身就是 bug，直接判 FAIL 并记录缺陷。**

绝不要用 `page.evaluate()` / JS 直接操作 DOM 来绕过 Playwright 的交互限制。这样做会掩盖真实的用户体验问题，让本该暴露的缺陷被跳过。

## 异常处理表

| 异常 | 处理方式 |
|------|---------|
| SQL 执行失败 | 记录错误，该用例标记 BLOCKED，继续下一条 |
| curl 超时/无响应 | 重试 1 次，仍失败则标记 BLOCKED |
| 浏览器页面加载失败 | 重试 1 次，仍失败则标记 BLOCKED |
| 预期结果与文档描述模糊 | 按文档字面意思判定，标记 ⚠️ 待确认 |
| 数据清理失败 | 立即告警，必须人工介入后才能继续 |
| browser_click 超时（元素不可达） | **直接判 FAIL** — Playwright 点不到 = 用户也点不到，这是 UI 缺陷。将超时错误作为证据记入缺陷报告 |
| 同一条用例出现互相矛盾的证据 | **先暂停登记缺陷**，回滚到严格串行模式重跑：`准备 -> 提交 -> 回查 -> 验证 -> 清理`。若串行重跑后恢复正常，则标记为“执行污染/误报”，不要记成产品缺陷 |

## 用例结果状态

| 状态 | 含义 |
|------|------|
| ✅ PASS | 所有检查项均符合预期 |
| ❌ FAIL | 至少一个检查项不符合预期 |
| ⏸️ BLOCKED | 无法执行（环境/数据问题） |
| ⚠️ WARN | 通过但有疑问待确认 |

## page.evaluate 的正确用途

`page.evaluate()` 只能用于**读取信息**（如获取元素属性、检查 DOM 状态），**绝不能用于代替用户交互**（点击、滚动、输入）。

正确：`await page.evaluate(() => document.querySelector('.popup').scrollHeight)` — 读取容器高度用于诊断
错误：`await page.evaluate(() => el.click())` — 绕过 Playwright 直接点击，掩盖了交互问题
