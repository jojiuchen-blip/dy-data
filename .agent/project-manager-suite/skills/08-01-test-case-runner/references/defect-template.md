# 缺陷跟踪模板

## 文件说明

全项目唯一一份缺陷文件，集中管理所有 FAIL 和 BLOCKED 的详情，便于跨业务域查看全貌、追踪修复状态、分配给开发。

路径：`docs/test-case/reports/defects.md`

## 文件结构模板

```markdown
# 缺陷跟踪

> **最后更新**: YYYY-MM-DD HH:MM

## 缺陷列表

| # | 用例编号 | 业务域 | 缺陷描述 | 严重度 | 状态 | 责任人 | 发现日期 | 修复日期 |
|---|---------|------|---------|--------|------|--------|---------|---------|
| 1 | TC-B1-02 | 模块1-列表选择 | 多条记录场景仅返回 2 条，预期 3 条 | P1 | OPEN | — | 2026-03-13 | — |
| 2 | TC-STA-05 | 模块2-状态展示 | `OVER_LIMIT` 状态未触发 | P2 | FIXED | 人A | 2026-03-14 | 2026-03-15 |

## 缺陷详情

### #1 TC-B1-02：多记录返回数量不符
- **预期**: data 数组长度 = 3
- **实际**: data 数组长度 = 2
- **证据**: `{"code":0,"data":[{...},{...}]}`
- **可能原因**: 数据准备 SQL 插入了 3 条，但第 3 条主键重复导致去重
- **状态**: OPEN
- **回归**: —（修复后回填回归结果）
```

## 缺陷状态流转

```
OPEN → FIXED → RETEST-PASS（关闭）
         ↘ RETEST-FAIL → 重新 OPEN
```

## 严重度定义

| 等级 | 含义 |
|------|------|
| P0 | 接口不可用 / 白屏 / 数据丢失 |
| P1 | 核心功能不符合预期 |
| P2 | 非核心功能异常或展示偏差 |
| P3 | 体验问题、文案错误 |

## 写入规则

- 每发现一个 FAIL，立即在 defects.md 追加一行 + 一个详情节
- 报告文件中的"失败详情"节简写为引用链接，详细内容统一在 defects.md
- 缺陷修复后由开发更新状态为 FIXED，测试回归后更新为 RETEST-PASS 或 RETEST-FAIL
