# 截图视觉审查标准

> 每张截图保存后，必须用 Read 工具读取截图文件，以人眼视角判断页面是否正常可用。
> 本文件定义"正常可用"的判断标准。

---

## 核心原则

**你不是在检查 DOM，你是在替真实用户看这个页面。**

一个普通用户打开这个页面，3 秒内的第一反应是什么？
- "正常的 App 页面" → 通过，进入逐项验证
- "这页面是不是坏了？" → 不通过，直接 FAIL

---

## 检查清单

按以下顺序扫描截图，任何一项不通过即判 FAIL：

### 1. 样式是否加载

| 正常 | 异常 |
|------|------|
| 有背景色区分（卡片白底、页面灰底等） | 全黑、全白、纯色无层次 |
| 卡片有圆角、阴影 | 元素无圆角、无阴影、像裸 HTML |
| 按钮有品牌色填充 | 按钮无样式、只有文字或默认边框 |
| 文字有大小层级（标题大、正文小） | 所有文字同一大小 |

**常见根因**：CSS 文件未加载、Tailwind 构建产物不完整、CDN 资源 404

### 2. 布局是否正常

| 正常 | 异常 |
|------|------|
| 元素从上到下有序排列 | 元素重叠、错位 |
| 内容在视口内，无水平滚动 | 内容溢出屏幕右侧 |
| 页面充分利用桌面视口宽度 | 页面缩在左上角或大量空白浪费 |
| 图片/图标正常显示 | 图片破裂图标（alt 文本外露） |

**常见根因**：视口尺寸未匹配项目实际、响应式断点错误、Flex/Grid 布局异常

### 3. 内容是否可读

| 正常 | 异常 |
|------|------|
| 文字清晰可辨 | 文字模糊、被截断、颜色与背景接近看不清 |
| 中文正常显示 | 乱码、方块字、英文占位符 |
| 数据已加载（列表行、名称、编号等业务字段有真实值） | 显示 Loading...、骨架屏卡住、空白 |
| 空态有明确提示文案 | 该有数据的地方一片空白无任何说明 |

**常见根因**：字体未加载、API 请求失败、数据未返回

### 4. 交互元素是否可识别

| 正常 | 异常 |
|------|------|
| 按钮看起来可以点击 | 按钮和普通文字无法区分 |
| 输入框有边框和提示文字 | 输入框不可见 |
| 弹窗有半透明遮罩和关闭方式 | 弹窗无遮罩、无法关闭 |

---

## 输出格式

快检不需要写报告，只需在测试记录中给出结论：

- **通过**：一句话带过，如"页面渲染正常，样式/布局/内容均无异常"
- **不通过**：写明具体问题 + 判 FAIL，如"CSS 未加载，页面全黑无圆角无阴影，直接判 FAIL"

---

## 典型案例

### 案例 1：CSS 构建产物不完整（P0）

**现象**：页面背景全黑，卡片无白底无圆角，文字直接贴在黑色背景上，按钮无样式
**根因**：Tailwind CSS v3 语法 + v4 PostCSS 插件，构建成功但 CSS 只有 96 个类（正常应 283 个）
**教训**：`npm run build` 退出码为 0 不代表产物正确，必须检查 CSS 文件大小和关键类名

### 案例 2：视口尺寸与项目实际不匹配（P1）

**现象**：页面被异常拉伸或挤压；两侧大片空白或内容溢出
**根因**：截图时浏览器视口与项目设计目标视口不一致
**教训**：截图前先确认项目目标视口（默认桌面 `browser_resize(1440, 900)`；如项目明确为移动端，按用例指定）
