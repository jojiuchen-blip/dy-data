# delivery-planner / plan-anatomy

这份文档解释多文件开发计划中的章节职责，以及每个 Task 字段应该怎么写。

正式开发计划由三类文件组成：
- 主开发计划：`docs/plans/delivery-plans/main-delivery-plan-<slug>.md`
- 任务看板：`docs/plans/delivery-plans/task-kanban-<slug>.md`
- 子开发计划：`docs/plans/delivery-plans/sub-delivery-plan-<slug>-<TaskID>-<short-name>.md`

## 1. 计划头部元信息

作用：让任何新线程在 30 秒内知道这份计划的版本、适用范围、参与角色、需求基线和目标。

至少要写：
- 版本
- 发布日期
- 前序版本
- 适用范围
- 参与角色
- 执行约束
- 目标
- 当前需求基线

不要只写“开发计划”四个字然后直接进入任务。

## 2. 本计划使用指南

作用：把“先读什么、再做什么、完成前怎么验”写清楚，让 AI 不需要猜流程。

至少要写：
- AI 执行工作流
- PRD 加载约束
- 读前门禁
- 完成前验证门禁

这部分本质上是在定义 runbook，而不只是文档目录。

## X. 环境依赖声明

作用：把“环境是否具备、该怎么检查”硬编码在主开发计划里，前置报错。

`coding-standards` 在 S4 开工前会运行环境自检脚本读取本章节：

```bash
node <suite-path>/skills/06-01-coding-standards/scripts/verify-task-context.mjs <主开发计划路径> <TaskID> --env-check
```

> `<suite-path>` 指套件根目录：源码仓库联调时为 `project-manager-suite/`，安装到宿主后为 `.agent/project-manager-suite/`；命令默认在宿主项目根目录执行。

声明的检测命令逐条执行、声明的目录逐个检查，任一失败则 `envReady: false`（exit code 2），禁止开始写代码。

至少要包含：
- 必需运行时（Node、Java、Python 等版本与检测命令）
- 前后端及 Python 工程依赖目录及判断标识（如 `node_modules/` 存在、虚拟环境已激活）

**可解析格式**：脚本只认下面两种表行，写法不符会被当作 0 条声明。

运行时检测行——三列表，第 3 列必须是反引号包裹的完整可执行命令；脚本逐条执行，命令退出码非 0 记为环境缺失：

```markdown
| 依赖项 | 版本要求 | 检测命令 |
|---|---|---|
| Node.js | >= 18 | `node -v` |
| MySQL 客户端 | >= 8.0 | `mysql --version` |
```

工程依赖目录行——两列表，第 1 列是反引号包裹的目录（相对宿主项目根），第 2 列是反引号包裹的文件 / 目录名后跟「存在」或「已就绪」；脚本按 `<宿主根>/<目录>/<标识>` 检查该路径是否存在：

```markdown
| 工程目录 | 就绪标识 |
|---|---|
| `frontend/` | `node_modules/` 存在 |
```

若整个章节解析不出任何声明（0 条），脚本不拦截（`envReady: true`，exit 0），但会输出一条警告，提醒核对表格格式或确认计划确实没有环境依赖。

## 3. 差距基线

作用：先把当前问题和目标态差距列清楚，再进入分阶段拆解。

写法要求：
- 每条差距都能对应后续至少一个任务
- 要写优先级、影响范围、状态
- 不要把差距写成抽象口号，例如“系统待优化”

## 4. 分工与边界

作用：明确碳硅边界、角色边界、高冲突文件归属和协作规则。

必须体现：
- AI 负责技术判断、代码理解、计划拆解
- 人类主要负责审核、业务决策、验收确认
- 技术判断类动作默认写成 `AI 执行 -> 人审核`

## 5. 执行阶段（Phase / Task 索引）

这是主开发计划中的阶段索引，用来组织 Phase 和子开发计划入口。

### 5.1 Phase 的职责

Phase 用来表达阶段目标、前后关系、并行性和状态。主开发计划里的 Phase 不写完整 Task 正文，只写任务索引。

一个合格的 Phase 至少要让读者知道：
- 为什么要先做这一阶段
- 谁负责
- 能否并行
- 当前完成到哪里

如果本计划属于“新建计划 / 多模块联调 / 高复杂度变更”场景，建议把关键 Phase 升级为 harness-ready 写法，补：
- `Entry Criteria`：什么条件满足后才允许进入该阶段
- `Exit Criteria`：达到什么结果才允许离开该阶段

写法要求：
- `Entry Criteria` 不写空话，要写前置任务、冻结物或输入资料
- `Exit Criteria` 不写“开发完成”，要写可验证结果或可交付物
- 若阶段存在外部阻塞，可在 `Entry Criteria` 中显式标注阻塞条件

### 5.2 子开发计划的职责

每个子开发计划只承载一个 Task 正文。执行者从任务看板定位 Task，再打开对应子开发计划读取 PRD、核心逻辑、核心文件和验证方法。

子开发计划必须回链：
- 主开发计划
- 任务看板

### 5.3 Task 的必填字段

每个任务都必须包含以下字段：

#### `PRD 双链·读`

写法：
- 列出真正需要读的需求 / PRD 文件和章节
- 不要写“相关 PRD”或“见 mainprd”这种模糊说法

用途：
- 建立计划 -> PRD 的正向链路
- 让执行者知道动手前应该读哪里

#### `核心逻辑`

写法：
- 解释该任务要落哪条业务规则
- 说明为什么这个任务存在
- 如涉及技术判断、跨文档比对、数据口径确认，要写明 `AI 执行 -> 人审核`

不要只写：
- “修改接口”
- “补数据库字段”
- “更新前端页面”

#### `核心文件`

写法：
- 列出真实文件路径
- 文件应来自真实代码、SQL、文档或测试资产
- 文件数量以能定位工作面为准，不宜空泛，也不要无限罗列

不要写：
- “相关代码”
- “前后端文件”

#### `完成标准`

写法：
- 必须是可以核查的显式条目
- 最少要包含一条真实验证口径
- 涉及固定行数、配置项、枚举值、字段口径时，要把关键值写出来

不要写：
- “数据完整”
- “配置补齐”
- “链路打通”
- “符合 PRD”

建议写法：
- 某配置表关键行数与枚举值已在目标库校验
- 某接口返回字段与 PRD 指定章节一致
- 保存后刷新回读，展示值与配置源一致

#### `完成收尾：状态同步`

写法：
- 作为 Task 正文的最后一个必填字段，说明完成实现、验证和 foundation 漂移判断后要做什么
- 至少写明：把 Task 完成事实、验证证据、完成日期、foundation 漂移结论和建议下一 Task 提交给 `ai-project-manager`；由它调度 `delivery-planner` 同步主开发计划、任务看板和当前子开发计划的状态
- 写明未完成状态同步收尾前，不得标记 Task 已完成

用途：
- 把"完成后要回写状态"从口头约定变成 Task 自带的收尾步骤
- 结构校验脚本 `validate-plan-structure.mjs` 会检查每个 Task 是否具备该字段

#### `Owner`

写法：
- 标明负责执行和审核关系
- 技术判断类任务优先写 `AI 执行 -> 人审核`

#### `前置`

写法：
- 明确依赖的是哪个 Task 或 Phase
- 没有依赖时写 `无`

#### `状态`

写法：
- 使用统一状态：`待审阅`、`待开发`、`进行中`、`已完成（YYYY-MM-DD）`
- **待审阅**：子开发计划初次生成后的默认初始状态。处于此状态时，AI 不得开始执行该任务
- **待开发**：人类 Owner 明确确认"审阅通过"后，由人类或 AI 将状态从 `待审阅` 变更为 `待开发`，此后 AI 方可进入执行。**不得在未获得人类明确审阅通过的情况下自行从 `待审阅` 切换为 `待开发`**
- 状态变更后要同步回写看板

**状态机：谁在什么时机翻状态**

| 状态变更 | 谁来做 | 什么时机 |
|---|---|---|
| `待审阅` → `待开发` | 人类 Owner 确认"审阅通过"后，由人类或 AI 执行变更 | 子开发计划审阅通过时 |
| `待开发` → `进行中` | `ai-project-manager` 调度 `delivery-planner` 执行 | S4 开工前，经用户确认开工该 Task 后 |
| `进行中` → `已完成（YYYY-MM-DD）` | `coding-standards` 完成实装后提交完成事实，`ai-project-manager` 调度 `delivery-planner` 回写 | Task 完成标准全部核查通过、执行完"完成收尾：状态同步"时 |

每次翻状态都必须在**三处同步**：任务看板、主开发计划执行阶段表、当前子开发计划。同一时刻有且仅有一个 Task 处于 `进行中`。

S4 开工前，`ai-project-manager` 会触发一致性校验脚本核对这三处：

```bash
node <suite-path>/skills/05-01-delivery-planner/scripts/check-plan-consistency.mjs <主开发计划路径> --json
```

它检查主计划与看板中每个 Task 状态一致、有且仅有一个 Task「进行中」、该 Task 链接的子开发计划存在且状态同为「进行中」、已完成 Task 两边完成日期一致；全部满足则 exit code 0（通过），发现不一致则 exit code 2 并列出错误清单。判定来源、报错修复方法详见 `SKILL.md` 的「S4 开工前一致性校验」一节。

### 5.4 Task 的增强字段（harness-ready）

以下场景建议默认补增强字段：
- 新建完整计划
- 跨端 / 跨系统任务
- 高风险改造或完工前任务
- 需要真实联调、真实保存、真实回查的任务

#### `Requirement ID`

写法：
- 优先引用真实需求编号、PRD 章节号或需求键
- 若当前 PRD 没有编号，可使用临时键，例如 `REQ-B3-DETAIL-BUTTON-01`

用途：
- 建立需求到任务的可追溯链
- 让后续验证与回归能回到同一个需求单元

#### `Verification Method`

写法：
- 写明“如何验证”，而不是只写“需要验证”
- 优先写真实接口调用、真实查询、真实保存回查、真实页面回显、测试报告核对等方法

不要写：
- “验证通过”
- “测试一下”

#### `Evidence`

写法：
- 写明验证证据会落到哪里，例如接口样例、SQL 回查、截图、测试报告、日志文件
- **Playwright / 浏览器截图**：统一存放到项目根目录的 `pwScreenShot/` 目录下，禁止散落到根目录或其他位置；子开发计划中若需明确路径，写为 `<projectRoot>/pwScreenShot/<截图文件名>`
- 若当前证据尚未生成，写计划中的预期落点

用途：
- 建立 Task -> 证据 的反向链
- 方便 harness 或后续 AI 线程快速复核

#### `Failure Handling`

写法：
- 写明“什么情况下不能继续、应该阻塞还是降级”
- 至少覆盖 PRD 缺失、代码 / PRD 冲突、验证资产缺失三类常见失败信号

不要写：
- “有问题再说”
- “按实际情况处理”

增强字段的核心目的：
- 让计划更适合后续 harness 评估，而不是只供人工目测

## 6. 任务看板

作用：给新线程一个低成本总览，同时承接状态回写。

任务看板是独立文件 `task-kanban-<slug>.md`，主开发计划只保留入口链接。

至少要有：
- 任务 ID
- 子开发计划链接
- Owner
- 前置
- 状态
- 完成日期
- 备注

只改子开发计划不改看板，会导致计划体系失真。

## 7. 发布闸门

作用：把“什么情况下能进入联调 / 发布 / 阶段完成”写成可勾选条件。

至少覆盖：
- PRD 与规则是否对齐
- 真实验证是否具备
- 状态回写是否完成

增强写法建议：
- 对关键任务检查是否存在 `Requirement ID -> Verification Method -> Evidence`
- 若证据不足或冲突未决，闸门状态应明确为阻塞，而不是模糊通过

## 8. 风险与应对

作用：把可能拖慢执行或造成返工的问题前置。

建议写：
- 风险描述
- 影响
- 应对策略
- Owner
- 状态

复杂计划可升级为“风险 / 假设 / 问题 / 依赖”视角：
- `Risk`
- `Assumption`
- `Issue`
- `Dependency`

如果不扩成独立表，至少也要把假设、阻塞依赖和已发生问题写进本章，而不是只写“风险”。

## 9. AI 执行示例

作用：用 1-2 个例子说明 AI 应该如何照着计划执行，避免把计划读成普通文档。

优先覆盖：
- 开始一个任务前怎么读
- 更新现有计划时如何同步关联章节

## 10. PRD → 任务反向索引

作用：把 PRD -> 任务 建成反向链路，防止计划脱离业务语义。

至少要做到：
- 关键 PRD 章节都能找到对应任务
- 新增或删除任务后同步更新

## 11. 五种常见失败信号

### 失败信号 1：只有任务标题，没有任务依据

表现：Task 只剩“改接口”“补表”“改页面”。

修正：补齐 `PRD 双链·读`、`核心逻辑`、`核心文件`。

### 失败信号 2：只有结构像完整计划，内容却没有真实落点

表现：章节很完整，但没有真实文件、真实字段、真实接口。

修正：回到 PRD / 代码 / SQL / 验证资产重新取证后再写。

### 失败信号 3：只补了局部任务，没有同步系统性章节

表现：补了 Phase，但没补看板、风险、闸门、反向索引。

修正：把计划当成一个整体文档，而不是局部 patch 列表。

### 失败信号 4：任务写成过程流水账，没有交付物边界

表现：Task 只是在列动作，例如“先改后端、再改前端、再联调”。

修正：回到交付物或结果拆解，按 work package 重新划分任务。

### 失败信号 5：完成标准存在，但没有验证方法和证据落点

表现：写了“接口对齐 PRD”“保存可用”“链路通过”，但没有说明怎么验证、证据存哪里。

修正：补 `Verification Method` 与 `Evidence`，未补齐前不要标记为完成。
