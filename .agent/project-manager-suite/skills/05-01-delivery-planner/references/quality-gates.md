# delivery-planner / quality-gates

在宣称“计划已写好”或“计划已更新完成”前，至少过完这份清单。

## 一、结构门禁

逐项确认：
- [ ] 主开发计划位于 `docs/plans/delivery-plans/main-delivery-plan-<slug>.md`
- [ ] 任务看板位于 `docs/plans/delivery-plans/task-kanban-<slug>.md`
- [ ] 子开发计划位于 `docs/plans/delivery-plans/sub-delivery-plan-<slug>-<TaskID>-<short-name>.md`
- [ ] 主开发计划的执行阶段以子开发计划索引组织任务
- [ ] 任务看板中的每个 Task 都链接一个子开发计划
- [ ] 每个子开发计划只包含一个 Task 正文
- [ ] 有计划头部元信息
- [ ] 有本计划使用指南
- [ ] 有环境依赖声明
- [ ] 有 PRD 加载约束
- [ ] 有读前门禁 / AI 自检清单
- [ ] 有完成前验证门禁
- [ ] 有差距基线
- [ ] 有分工与边界
- [ ] 有执行阶段（Phase / Task）
- [ ] 有任务看板
- [ ] 有发布闸门
- [ ] 有风险与应对
- [ ] 有 AI 执行示例
- [ ] 有 PRD → 任务反向索引

如果缺任何一项，默认不能称为正式开发计划。

## 二、任务字段门禁

检查每个子开发计划里的 Task，确认它具备：
- [ ] `PRD 双链·读`
- [ ] `核心逻辑`
- [ ] `核心文件`
- [ ] `完成标准`
- [ ] `完成收尾：状态同步`
- [ ] `Owner`
- [ ] `前置`
- [ ] `状态`

若有 Task 缺字段，说明拆解还没完成。

对新建计划 / 高风险计划，再额外抽查关键 Task：
- [ ] 有 `Requirement ID`
- [ ] 有 `Verification Method`
- [ ] 有 `Evidence`
- [ ] 有 `Failure Handling`

## 三、真实依据门禁

逐项确认：
- [ ] 读过 `<suite-path>/skills/00-01-ai-project-manager/references/defaults/tech-stack.md`（套包内置技术栈参考，所有 Task 技术实现细节应与实际技术栈一致）
- [ ] 读过仓库规则源
- [ ] 读过当前计划或目标样板计划
- [ ] 读过 mainprd / 需求导航文档
- [ ] 读过与任务相关的 subprd / API / 数据库章节
- [ ] 读过至少一组真实代码或 SQL 文件
- [ ] 读过至少一组测试或验证资产（如任务需要验收口径）

如果以上任一项缺失，计划很可能只是模板填空。

## 四、过程与拆解门禁

逐项确认：
- [ ] 关键 Phase 写了可核查的 `Entry Criteria` / `Exit Criteria`，或能说明为何暂不需要
- [ ] Task 拆解围绕交付物 / 结果，而不是纯动作流水
- [ ] 跨端、跨系统、核心文件过多的任务已继续拆分
- [ ] 计划中说明了证据不足、需求冲突、验证资产缺失时的失败分支

## 五、完成标准门禁

逐项确认：
- [ ] 完成标准是可核查条目，不是抽象描述
- [ ] 涉及固定配置或枚举时，写了具体值 / 行数 / 字段口径
- [ ] 至少包含一条真实验证方式或真实结果校对方式
- [ ] 没出现这些高风险词：`数据完整`、`配置补齐`、`链路打通`、`符合预期`
- [ ] 对关键任务，`Verification Method` 与 `Evidence` 能闭合成可复核链路

## 六、协作边界门禁

逐项确认：
- [ ] 计划体现“AI 直接执行，人类 review”
- [ ] 技术判断类步骤写成 `AI 执行 -> 人审核`
- [ ] 没把“分析 / 对比 / 判断”这种纯技术动作分配给人类执行
- [ ] 人类步骤是可直接操作或可直接审核的动作

## 七、更新模式门禁

如果本次不是新建，而是更新已有计划，还要额外确认：
- [ ] 已同步回写相关 Task 状态
- [ ] 已同步回写日期、依赖或备注
- [ ] 已更新任务看板
- [ ] 已检查发布闸门是否受影响
- [ ] 已检查风险与应对是否需要新增 / 关闭
- [ ] 已同步更新 PRD → 任务反向索引
- [ ] 新增验收口径已经进入计划正文，而不是只留在聊天记录中

## 八、S4 开工前一致性门禁

进入 S4（代码实装）前，`ai-project-manager` 会触发本 skill 的 `s4_pre_coding_plan_consistency_check`。运行命令：

```bash
node <suite-path>/skills/05-01-delivery-planner/scripts/check-plan-consistency.mjs <主开发计划路径> --json
```

> `<suite-path>` 指套件根目录：源码仓库联调时为 `project-manager-suite/`，安装到宿主后为 `.agent/project-manager-suite/`；命令默认在宿主项目根目录执行。

脚本读主开发计划执行阶段表、任务看板和当前子开发计划三处的 Task 状态。逐项确认：
- [ ] 当前开工 Task 已在任务看板置为「进行中」（或主计划里有驾驶舱表声明当前 Task）
- [ ] 有且仅有一个 Task 处于「进行中」
- [ ] 主计划执行阶段表与任务看板中每个 Task 的状态一致
- [ ] 「进行中」Task 链接的子开发计划文件存在，且其 `**状态**` 字段同为「进行中」
- [ ] 已完成 Task 在主计划与看板两边的完成日期一致

脚本 exit code 0 表示以上全部满足，可放行进入 `coding-standards`；exit code 2 表示存在不一致，输出错误清单（含修复方法），修复后重跑，未通过前不得开工写代码。

## 九、Harness 增强门禁

如果本次计划希望更适合 harness 评估，再额外确认：
- [ ] 关键需求具备 `Requirement -> Task -> Verification -> Evidence` 追溯链
- [ ] 关键 Phase 具备进入 / 退出条件
- [ ] 阻塞条件会显式影响发布闸门，而不是口头说明
- [ ] 标准映射体现在执行规则中，而不是只写“参考某某标准”

## 十、四类常见不合格结果

### 1. 外形像完整计划，内容像空模板

表现：章节都在，但没有真实文件、真实字段、真实接口、真实验证资产。

结论：不合格，回到取证阶段。

### 2. 只有任务标题，没有业务规则

表现：Task 写成“改接口 / 改页面 / 改表 / 联调”。

结论：不合格，补 `PRD 双链·读` 与 `核心逻辑`。

### 3. 只更新局部，不更新全局联动章节

表现：补了一个 Phase，但看板、风险、闸门、索引都没动。

结论：不合格，按“计划整体文档”重新回写。

### 4. 提到了标准，但没有转成字段或门禁

表现：正文里写“按 29148 / 12207 / DoD 执行”，但任务结构和验证口径没有任何变化。

结论：不合格，说明标准仍停留在名词层，没有落到可执行规则。
