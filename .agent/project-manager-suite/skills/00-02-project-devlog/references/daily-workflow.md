# 每日开发日志工作流

本文件定义 `project-devlog` 的每日开发日志功能（新建、追加、收口）。

---

## 功能 1：新建日志

**触发条件**：当天该用户的日志文件不存在。

**步骤**：
1. 获取当前日期和 git 用户名
2. 检查 `logs/YYYYMMDD_refactor_log_<用户名>.md` 是否存在
3. 若不存在，复制 `assets/daily.md` 模板
4. 自动填入：日期、操作人、关联计划文件路径
5. 开始记录当日第一项任务

---

## 功能 2：追加更新（AI 主动触发）

**触发条件**：命中日志强触发条件，或同一主题已积累到值得合并写入的阶段性结论后。

**强触发条件**：
- 阶段发生变化
- 形成新的正式交付物
- 当前会话准备收口，且需要为下一轮留下承接状态
- 已形成一组阶段性结论，不写日志会明显增加下轮恢复成本

**适合合并写入的实质产出**：
- 提交了代码（git commit）
- 修改了一组相关文件（创建/编辑/删除）
- 完成了一组连续决策并已沉淀到项目画像 / 执行计划
- 修复了 bug 或完成了联调验证
- 完成了一次阶段边界或交付边界的收口

**不算实质产出**（不触发）：
- 只是回答问题
- 只是读文件/查代码
- 讨论方案但未执行
- 只确认了一个待确认项，且同主题仍在连续收敛
- 刚更新完画像 / 计划，又立刻对同一小步动作单独补一条日志

**步骤**：
1. 优先读取当日日志文件与当前计划权威文件
2. 先判断是否属于“同一主题连续追加”：
   - 若与最近一条日志属于同一主题、同一阶段、同一会话，优先合并为一次阶段性更新
   - 只有在主题已切换，或上一条记录已经形成稳定结论时，才新开一段补充更新
3. 在日志文件末尾追加：

```markdown
---

## 补充更新（HH:MM）

### 任务 N：标题
- **目标**：
- **操作**：
- **结果**：
- **涉及文件**：
```

4. 同时更新"一、执行概要"的表格（仅在确有新任务时加一行；同主题合并时优先更新原任务描述）
5. 若该任务已完成计划项，同次提醒或同步回写计划状态

**合并写入约束**：
- 同一主题下连续 2-5 个小决策，优先整理成一条“阶段性结论”再写
- 不按“问一次、答一次、写一次”执行
- 日志应优先记录“结论包”，而不是复刻聊天流水

---

## 功能 3：收口汇总

**触发词**：用户说"写日志"、"总结今天"、"日志收口"。

**步骤**：
1. 读取当日已有日志文件（如果有追加记录）
2. 执行 `git log --since="today" --format="%h %ai | %s"` 获取今天全部提交
3. 填充/更新以下章节：
   - **三、变更总览**：
     - **文件变更清单**：从各任务「涉及文件」聚合去重，标注操作类型（新建/修改/删除）
     - **Git 提交记录**：从 git log 填入
   - **四、发现的问题**：如果今天没有 bug，写"无"
   - **五、复盘**：
     - 至少列出 1 条"做得好的"
     - 如果有问题，按"现象→根因→经验→🔧是否提炼为规则"的格式
     - 输出"今日经验总结"，每条标注是否建议加入规则
   - **六、待跟进事项**：列出明天要做的任务
4. 向用户展示完整日志或关键变更摘要

**经验提炼规则**：
- 如果某条经验标注 🔧 "建议写入项目规则"，优先判断应进入：
  - `project-rules.md`
  - `docs/rules/devlog.md`
  - 其他对应任务规则包
- 日志写完后提醒用户：
  - "以下经验建议沉淀为项目规则：[内容]。是否现在回写到对应规则文档？"

---

## 章节结构说明

每份日志必须包含以下 6 章 + 1 个可选章，顺序不可变，无内容写"无"而非删除：

| 章 | 标题 | 必填 | 说明 |
|---|------|------|------|
| 一 | 执行概要 | ✅ | 30 秒看完今天做了几件事，可附一句话关键结论 |
| 二 | 操作详情 | ✅ | 每个任务的目标/操作/结果/文件；复杂决策可用分析体展开 |
| 三 | 变更总览 | ✅ | 文件变更清单（操作/文件/说明）+ Git 提交记录 |
| 四 | 发现的问题/缺陷 | ✅ | 无 bug 写"无" |
| 五 | 复盘 | ✅ | 做得好 + 遇到的问题 + 经验(含🔧标签) |
| 五·附 | 方法论沉淀 | ⬜ | 可复用的方法论、设计原则。普通开发日不填 |
| 六 | 待跟进事项 | ✅ | checkbox 待办列表 |

### 第二章的两种写法

- **执行型任务**（默认）：目标 → 操作 → 结果 → 涉及文件
- **决策型任务**（复杂方案选型时可选）：背景问题 → 方案对比 → 最终决策 → 涉及文件

---

## 追加更新规则

同日后续窗口追加内容时，在文件末尾使用：

```markdown
---

## 补充更新 N（HH:MM · 窗口 N）

### 任务 N：标题
- **目标**：
- **操作**：
- **结果**：
- **涉及文件**：
```

**关键约束**：
- 补充更新**只追加任务详情**（第二章内容），不重复写复盘或其他章节
- 第五章（复盘）在收口时统一写一次，避免碎片化
- 同时更新第一章（执行概要）的表格，加一行新任务
- 若同一主题仍在连续收敛，优先合并到最近相关任务下，不机械新增新的“补充更新 N”
