# Project Devlog

本 skill 是 `project-manager-suite` 的默认状态回写与开发日志能力。

它负责：

- 每轮推进后沉淀项目状态
- 每日开发日志的新建、追加和收口
- 为下一轮推进提供承接信息

它不负责：

- 重新定义项目范围
- 详细任务规划
- 替代需求或方案能力

## 日志的三重职责

1. **流程文件** — 记录做了什么、怎么做的
2. **复盘基底** — 提供回顾与改进的素材
3. **经验指导** — 避免下次任务犯同样的错

> 复盘中的每条经验必须标注 🔧 "是否建议提炼为项目全局规则"。

## 使用原则

1. 记录事实，不夸大结果。
2. 区分"已完成""待确认""有风险"。
3. 记录足够承接下一轮的信息，不写流水账。
4. 优先复用本轮已有产出，不重复生成新内容。
5. 先吸收现有上下文（画像、计划、本轮产出），再补最少缺口。

## 默认输入

优先读取：

- 本轮新产出的需求摘要、方案、计划或验收结果
- 项目画像中的当前阶段和优先级
- 当前执行计划中的进行中任务和下一步任务

若宿主项目已存在自己的规则文件、计划文件、状态回写文件或日志目录，应优先按宿主项目映射后的权威入口执行。

## 功能概览

1. **每轮状态回写** — 沉淀轮次级项目推进结果
2. **新建日志** — 见 `references/daily-workflow.md`
3. **追加更新** — 见 `references/daily-workflow.md`
4. **收口汇总** — 见 `references/daily-workflow.md`

## 文件命名与位置

宿主项目已有日志目录时，优先沿用宿主项目既有位置；若没有，再使用以下默认约定：

```
logs/YYYYMMDD_refactor_log_<用户名>.md   # 每天一个；<用户名> 默认来自 git user.name，缺失时用 actor
```

> v2.0.0：移除原"按人按天分目录"假设和周报机制——本套包默认场景为单 / 小团队内部工具，不预设多人协作分目录。如宿主项目确实是多人协作且有强需求，可在文件名中添加自定义后缀。

## 输出要求

- 用简洁、可追踪的表述
- 不写长篇叙事
- 明确下一步动作和责任归属
- 至少应沉淀：本轮动作、本轮产出、当前结论、下一步建议
- 若本轮没有形成完整结果，也要明确当前卡点、待确认项和下一步承接建议

## 与项目规则的关系

1. 写日志前先读项目规则文件中的启动规则、任务边界和回写纪律
2. 写日志时遵守状态回写协议，不得把单轮状态写进规则层
3. 日志中的已完成事项、待跟进事项、计划状态必须与当前计划权威文件一致
4. 若日志中出现高频、稳定、可执行检查的经验，再考虑升级回写到 `project-rules.md` 或专项规则

## 资源入口

- 结构化写入工具：`tools/devlog-sync.mjs`（套件自带的命令行工具，把一轮推进的目标 / 动作 / 结果按模板结构写入当日日志，必要时自动新建日志文件）。运行环境能执行 Node.js 时优先用它，不要手工复制模板：

```bash
node <suite-path>/tools/devlog-sync.mjs <hostRoot> --title <标题> --goal <目标> --action <动作> --result <结果> --json
```

> `<suite-path>` 指套件根目录：源码仓库联调时为 `project-manager-suite/`，安装到宿主后为 `.agent/project-manager-suite/`；命令默认在宿主项目根目录执行。工具不可用（如无 Node.js）时，再按下面的模板与工作流手工维护日志。

- 每日日志模板：`assets/daily.md`
- 每日操作工作流：`references/daily-workflow.md`

## V1 边界

本 V1 先解决：

- 每轮推进后有最小可追踪状态
- 每日开发日志新建、追加与收口
- 为下一轮继续推进提供承接信息

本 V1 暂不解决：

- 长周期多阶段复盘体系
- 高级变更治理规则
- 复杂产品策略设计
