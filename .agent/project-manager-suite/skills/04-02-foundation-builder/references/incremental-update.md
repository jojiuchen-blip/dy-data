# 增量更新

> 本文件在 Phase 1 检测到自身前次产物存在时由 SKILL.md 指令加载。
> 影响 Phase 2/3/4/5 的执行逻辑。

## 触发条件

Phase 1 检测到 `docs/prd/foundation/foundation-{glossary,schema,api}-<slug>.md` 中任意一个已存在时加载。

## 变更检测逻辑

```
读取前次产物（docs/prd/foundation/foundation-{glossary,schema,api}-<slug>.md）
  ↓
读取当前上游产物（page-delivery + 页面代码 + BRD）
  ↓
对比差异来源：
  ┌─ 页面新增 → delivery 中有新路由/新文件，前次产物未覆盖
  ├─ 页面删除 → delivery 中移除了路由/文件
  ├─ 页面修改 → 同一文件内容变化（字段增减、组件变更）
  └─ BRD 变更 → 项目类型或业务模型调整
```

## 各 Phase 的增量处理

| Phase | 新增 | 删除 | 修改 |
|-------|------|------|------|
| Phase 2 术语表 | 提取新术语追加到对应分组 | 标记关联术语为 `⚠️ 待确认`（不自动删除） | 检查术语定义是否需更新 |
| Phase 3 Schema | 新推导表/字段追加 | 标记关联表为 `⚠️ 待确认` | 对比字段变更，更新对应表 |
| Phase 4 API | 推导新接口追加 | 标记关联接口为 `⚠️ 待确认` | 更新接口的请求/响应字段 |
| Phase 5 自查 | 全量重跑（不做增量） | 全量重跑 | 全量重跑 |

**核心原则**：
- 新增 → 自动追加
- 删除 → 只标记不自动删，需用户确认
- Phase 5 一致性自查 → 始终全量重跑（成本低、可靠性要求高）

## 增量产物的变更标记

总览表格中增加 `变更` 列：

```markdown
| # | 表名 | 所属业务域 | 作用 | 变更 |
|---|------|----------|------|------|
| 1 | user_config | 配置管理 | 用户偏好配置 | |
| 2 | order_label | 订单管理 | 订单标签 | **新增** |
| 3 | ~~old_cache~~ | ~~缓存~~ | ~~已废弃~~ | **删除** |
| 4 | daily_status | 状态管理 | 状态配置 | **变更** |
```

标记规则：
- 未变化的行：`变更` 列留空
- 新增行：`变更` 列填 **新增**
- 删除行：全行加删除线 `~~xxx~~`，`变更` 列填 **删除**
- 变更行：`变更` 列填 **变更**

## 增量模式的用户交互

进入增量模式时，先向用户呈现变更摘要：

```
检测到前次 foundation 产物，进入增量更新模式。

上游变更摘要：
- 新增页面: N 个（页面名1、页面名2）
- 修改页面: N 个（页面名3）
- 删除页面: N 个（页面名4）
- BRD 变更: 有/无

预计影响：
- 术语表: +N 条新术语，M 条待确认
- Schema: +N 张新表，M 张表字段变更，K 张表待确认
- API: +N 个新接口，M 个接口字段变更，K 个接口待确认

是否继续？
```

用户确认后，按 Phase 2 → 3 → 4 → 5 → 6 顺序执行增量更新，每个 Phase 仍需用户确认后才进入下一个。
