- 不要让脚本反向定义阶段规则

## 0. 一页版总规则

如果你只记得住一页，就记这 8 条：

1. 运行环境允许时，优先调用工具脚本完成校验、阶段判断、骨架补齐和日志回写，不先手工硬推。
2. 先识别宿主项目已有的全局文件和状态入口，不要先新建。
3. 缺项目画像时，先发起最小访谈，再创建最小画像载体。
4. 套件已加载时，用户说“启动一个新项目”就直接进入主入口，不额外追问“是否按 `ai-project-manager` 流程推进”。
5. 项目启动、上下文识别和阶段判断完成前，不得先进入 `superpower` 等通用增强类 skill。
6. 命中阶段后，正式交付物交给目标 skill 独占产出，主入口不越权代做。
7. 每轮结束都要回写到宿主权威文件，不把结果留在对话里；其中项目画像 / 执行计划可即时回写，`project-devlog` 默认只在里程碑或交接点合并写入。
8. 涉及宿主目录、阶段产物落点或 skill → 文件夹映射时，直接回看 `PIPELINE.md`；本文件不复写那套规则。

## 1. 运行流程（5 步）

### Step 0：脚本优先检查

当运行环境支持本地执行脚本，且当前套件工具可用时，主入口默认先走这条链路：

1. 调用 `tools/validate-global-files.mjs`
2. 调用 `tools/route-check.mjs`
3. 仅在确有骨架缺口时调用 `tools/bootstrap-host.mjs`
4. 一轮有效推进结束后，按需调用 `tools/devlog-sync.mjs`

`route-check.mjs` 的命令口径固定为位置参数，不使用 `--host-dir`：

```bash
node <suite-path>/tools/route-check.mjs <host-project-root> [--target-stage S0.5|S1|S2|S3|S4|S5|S6|S7] [--json]
```

`<suite-path>` 指套件根目录：源码仓库联调时为 `project-manager-suite/`，安装到宿主后为 `.agent/project-manager-suite/`；命令默认在宿主项目根目录执行。

注意：`--host-dir` 是 `page-designer/scripts/page-ledger-query.mjs` 的参数口径，不能套用到 `route-check.mjs`。

脚本优先的目的：

- 把结构识别、门禁判断、骨架补齐、日志回写这类稳定动作收进执行层
- 减少主入口每轮都靠纯文本协议手工重复判断
- 保证 hooks / 插件 / 本地终端执行时尽量遵循同一套动作顺序

只有以下情况，才退回纯协议执行：

- 当前平台不支持本地脚本
- 当前环境找不到工具脚本
- 当前动作属于首轮访谈、业务澄清或复杂例外判断
- 当前问题超出 V1 工具覆盖边界

### Step 1：启动 — 识别全局文件与状态回写入口

按以下顺序识别宿主项目是否已有 3 类全局文件，以及状态回写入口：

1. 全局规则文件（项目怎么运行）
2. 项目画像文件（项目当前是什么）
3. 当前执行计划文件（当前执行驾驶舱，现在做什么）
4. `project-devlog` 入口与日志载体（最近发生了什么）

识别原则：先看职责不看文件名；先用宿主现有权威源，不先新建默认文件。
宿主已有文件的映射关系、模板创建条件与默认落地策略，统一以 `global-files-protocol.md` 为准。

若工具可用，本步骤应优先以 `validate-global-files.mjs` 的输出为准；主入口只补脚本尚未覆盖的识别信息或解释性判断。

**每日首次进入时**，额外确认：
- 当前操作者身份（优先用画像中的 `身份识别方式`）
- 计划入口是否明确（优先用画像中的 `计划入口文件`）
- 最近状态入口是否明确（优先用画像中的 `最近状态入口`）
- 今天从哪里继续（结合 `当前阶段` + `待确认项`）
- 当前任务类型所需专项规则和资料入口

### Step 2：读取 — 建立上下文

按固定顺序读取已识别的全局文件与状态回写入口。

**读取全局规则文件**时识别：项目结构、协作边界、技术边界、文档入口、权威源约定。

**读取项目画像文件**时，按优先级读取：

| 优先级 | 读取内容 |
|--------|----------|
| P1 | `项目名称`、`项目一句话目标`、`当前阶段`、`当前最适合进入的阶段`、`当前轮应输出的交付物` |
| P2 | `协作模式`、`身份识别口径`、`当前任务执行主体`、`项目覆盖对象` |
| P3 | 规则入口、计划入口、状态入口、PRD 入口、身份识别方式、`当前页面主要给谁用`、`当前页面主要用途`、`页面定位标签` |
| P4 | `第一版核心目标`、`第一版范围`、已有材料 |
| P5 | 待确认项 |

**读取执行计划文件**时识别：当前正式计划文件组、当前活跃 Phase / Task、下一步任务、完成标准摘要、当前阻塞与前置依赖。

**读取状态日志载体**时识别：最近日志、最近验收、最近决策、待跟进事项；若命中日志或状态沉淀任务，默认由 `project-devlog` 承接，并写入项目画像映射的状态入口；未配置时才使用默认 `logs/`。

### Step 3：补齐 — 填缺口

本节只定义运行时如何判断缺口、哪些缺口会阻断推进，以及补齐后是否可继续；全局文件协议细节与宿主骨架实现细节不在本节重复维护。

按以下优先级补齐缺失信息：

| 缺口层级 | 内容 | 补齐方式 |
|----------|------|----------|
| 启动级（不补则无法恢复上下文） | 协议中的“启动最小必需字段包” | 用户确认优先；无法确认时停机等待，不自行编造 |
| 识别级（不补则无法稳定承接） | 协议中的“身份识别字段包” + 计划入口、状态入口、PRD 入口 | 系统推断优先；无法推断写 `待确认` |
| 页面级（当前轮涉及页面则不补会直接导致页面方向失真） | 协议中的“页面任务必补字段包” | 业务字段由用户确认优先；`页面定位标签` 由主入口归一回写；未补齐前不得进入 S2 |
| 推进级（不补则路由失真） | 协议中的“运行时回写字段包” | 主入口判断回写 |
| 画像级（影响当前判断才补） | 第一版核心目标、第一版范围、已有材料 | 用户确认优先；不追问非当前阶段所需信息 |

缺失文件时的降级处理（**必须严格遵守**）：

| 缺失文件 | 处理规则 |
|----------|----------|
| **项目画像** | **绝对禁止直接静默生成！** 必须立即停机，触发最小访谈（见 §3），由用户回答后才能生成；具体创建前置条件与模板落地规则，统一以 `global-files-protocol.md` 为准。 |
| **全局规则** | 若无明确规则，不强行生成大维度的 `project-rules.md`，仅需在内存中标记“规则缺失”，使用默认配置即可。 |
| **执行计划** | **属于启动记忆骨架的重要缺口。** 它是当前执行驾驶舱，不是完整开发计划正文。若宿主缺少计划载体，本轮应标记为待补齐，并按 `global-files-protocol.md` 中的创建规则处理。 |
| **状态回写** | 不阻断当前轮；默认由 `project-devlog` 承接，具体落地载体与写入口径统一以 `global-files-protocol.md` 为准。 |
| **整体骨架缺口** | 若宿主项目连基本目录层级都未建立，主入口必须主动进入“骨架补齐”状态；根目录判定、目录骨架、`docs/rules/` 默认规则生成与安装迁移策略，统一以 `routing.md` 为准。 |

补齐动作口径：
- 是否映射宿主已有文件、是否允许基于模板创建、以及各类文件的创建前置条件，统一以 `global-files-protocol.md` 为准。
- 本节只定义“当前缺口是否阻断、先补哪类信息、补齐后是否可继续推进”。

细则导航：
- 全局文件映射、模板创建条件与状态回写口径：统一以 `global-files-protocol.md` 为准。
- 宿主骨架补齐、`docs/rules/` 默认规则生成与安装迁移策略：统一以 `routing.md` 为准。

**字段来源口径**：
- **用户确认**：信息需要用户拍板才成立（含自然对话中明确表达的），不限于模板式访谈
- **系统推断**：可从仓库/文件/路径/角色映射自行识别的，不应转嫁给用户回答
- **主入口回写**：主入口综合判断后写回的阶段结论、交付物和执行主体

### Step 4：路由 — 判断阶段与下一步能力

根据项目画像判断当前最适合进入的阶段（见 §2 阶段判断表），然后决定：
- 停留在主入口继续澄清/补文件
- 进入某个子能力

路径约束补充：阶段判断若依赖上游产物路径或宿主目录，直接引用 `PIPELINE.md`；`runtime.md` 不展开物理路径细节。

启动阶段防混淆约束：
- 当主入口只是因为“当前轮目标开始收敛到页面”而提前补问“页面任务必补字段包”时，这个动作仅用于补 S2 门禁字段，**不等于已经进入 S2**
- 只要 `project-profile.md`、`execution-plan.md` 或基础目录骨架仍缺失，当前状态仍属于“启动 / 骨架补齐”，不得对外表述为“已经进入 S2”或“正在进入 page-chief”
- 只有在启动最小必需字段包、画像/计划载体和基础骨架都已补齐后，主入口才允许正式做阶段判断；阶段判断通过后，才能对外宣称“可进入 S2”

若工具可用，本步骤应优先读取 `route-check.mjs` 的阶段判断、门禁结果、阻断原因与下一步建议；主入口负责解释结果、补齐脚本尚未覆盖的业务判断，并决定如何与用户沟通。

进入子能力的必要条件（**全部满足**才进入）：
1. 当前阶段足够明确
2. 主入口已建立最小统一上下文
3. 当前轮目标超出主入口的"判断与路由"职责
4. 进入子能力能带来明确的新交付物
5. **若当前阶段发生变化，必须先完成阶段切换日志回写（调用 `project-devlog`），回写未完成前不得进入子能力**

进入子能力后的执行纪律（**必须遵守**）：
1. 一旦进入子能力，当前阶段的正式交付物必须由目标 skill 按其自身工作流独占产出。
2. 主入口只负责交接最小上下文：当前阶段、目标 skill、当前轮交付物、回写目标、待确认项。
3. 主入口不得在进入子能力后，继续直接代写该阶段正文产物，例如 BRD、页面代码、技术地基、完整版 PRD、开发计划、测试用例或测试结果。
4. 子能力完成后，主入口负责将阶段结论、入口信息和下一步建议回写到全局文件。

skill 优先级（**先后顺序不可混淆**）：
1. **流程调度型优先**：先由 `ai-project-manager` 做文件识别、上下文建立、阶段判断。
2. **阶段交付型其次**：阶段明确后，交由阶段 skill 独占产出正式交付物。
3. **专项执行型最后**：页面、研发、日志等专项 skill 只在命中具体执行任务时进入。
4. **通用增强类补充能力最末位**：`superpower` 等不掌握项目阶段协议的泛化 skill，不得在项目启动、访谈、阶段判断前先行接管。

刚性规则 / 弹性规则：
- **刚性规则**：缺画像先访谈；命中阶段后不得越权代做；S2 必须先由 `page-chief` 调度完成页面环节（`page-designer` → `page-explainer`，必要时回环），再由 `prd-chief` 调度完成 PRD 环节（`foundation-builder` → `prd-writer`）；已有权威文件不重复新建第二权威源；通用增强类 skill 不得早于主入口阶段判断。
- **弹性规则**：访谈问题顺序、材料读取深度、待确认项数量、页面类任务的追问措辞可按上下文调整。

页面任务识别与牵引规则（**必须主动执行，不依赖用户主动提及**）：
1. 只要当前轮交付物已经开始收敛到页面、原型或前端界面，主入口必须主动补齐"页面任务必补字段包"。
2. 不得把"这是操作页还是配置页"这类定位标签，寄希望于用户自行主动说出；系统必须先问清"给谁用、主要用途"，再由主入口归一。
3. 若 `项目覆盖对象`、`当前页面主要给谁用`、`当前页面主要用途` 任一缺失，或 `页面定位标签` 仍为 `待确认`，主入口不得进入 S2，不得进入 `page-chief`。
4. 主入口必须把业务语言回答归一成 `页面定位标签` 回写到画像中，供后续页面设计类 skill 直接读取。

页面定位标签归一规则（默认口径）：
- `当前页面主要用途 = 业务处理` 时，默认回写 `页面定位标签 = 操作`
- `当前页面主要用途 = 系统管理` 时，默认回写 `页面定位标签 = 配置`
- `当前页面主要用途 = 内容展示` 时，默认回写 `页面定位标签 = 查看`
- 若用户已明确指定页面定位，与上述默认归一规则冲突时，以用户明确指定为准
- 若信息仍不足以稳定归一，回写 `待确认`，并继续停留主入口补问

**S2 特殊强约束：**
- 当当前阶段命中 **S2 页面设计、技术地基与完整版 PRD** 时，必须先确认“页面任务必补字段包”已补齐。
- 当当前阶段命中 **S2 页面设计、技术地基与完整版 PRD** 时，**第一动作必须是先进入 `page-chief`**，由其调度 `page-designer` 产出页面代码 / 页面交付清单；主入口不得再假设存在独立的页面专项子能力作为直连目标。
- S2 首轮默认交付物必须是：**页面代码 / 页面交付清单 + 待确认项**。
- 仅有页面代码 / 页面交付清单，不代表页面环节已完成；在 `page-explainer` 产物未齐、交互语义未全部 `locked`、或仍有未解决 `design_gap` / `logic_conflict` 时，主入口必须继续停留在 `page-chief` 调度链路内。
- 只有当 `page-chief` 基于文件状态判定页面环节已收口后，主入口才允许进入 `prd-chief`，再由其线性调度 `foundation-builder` 和 `prd-writer`。
- 进入 `prd-chief` 后，必须先由 `foundation-builder` 产出术语表 / Schema / API / foundation 交付清单；仅在技术地基完成后，才允许由 `prd-writer` 产出功能列表、主 PRD 和子 PRD。
- 若当前仅有需求摘要或 BRD，没有页面代码 / 页面交付清单，主入口不得把 S2 判定为“已开始完成”；若页面环节尚未被 `page-chief` 判定 DONE，或缺少 foundation 产物，同样不得把 S2 判定为“可进入 PRD 终稿”。

### Step 5：回写 — 写回正确的全局文件

| 内容类型 | 写入目标 |
|----------|----------|
| 项目目标澄清、阶段变化、协作模式、当前任务执行主体、入口信息、待确认项 | 项目画像文件 |
| 当前正式计划入口、当前活跃任务、下一步任务、完成标准摘要 | 执行计划文件 |
| 本轮做了什么、产出了什么、遇到的问题、下一轮建议 | 仅在阶段切换、正式交付、会话收口或需要交接时，合并调用 `project-devlog` 或 `tools/devlog-sync.mjs` 写入日志文件 |
| 稳定的长期规则（极少更新） | 全局规则文件 |

S2 页面环节收口后的回写口径：

- 当 `page-chief` 判定页面环节 DONE 后，主入口必须把项目画像中的 `当前轮应输出的交付物` 更新为 `技术地基 / PRD 环节输入`，把 `当前最大不确定项` 从页面布局/交互确认切换为术语表、Schema、API 等技术地基问题，并将页面相关待确认项标记为已确认或移出待确认列表。
- 同一轮必须把执行计划中的 `当前目标`、`进行中任务`、`下一步任务`、`完成标准` 从 page-designer / page-explainer 页面环节更新为 `prd-chief` 调度 `foundation-builder`；不得让“页面环节需确认”这类旧措辞继续留在驾驶舱里。
- 若本轮已产出 `page-delivery`、`explainer-delivery` 或 route-check 明确显示 `pageStageClosedForPrd.pass = true`，应作为一次阶段内交接点写入 `project-devlog`，方便下一轮直接进入 PRD 环节。

禁止：把 3 类全局文件职责混写到同一个文件；把详细任务明细、完整过程日志写入画像。
禁止：宿主已有对应全局文件时，再额外创建一份同职责模板文件。
禁止：状态回写类任务绕过 `project-devlog` 直接自由写入状态载体，除非宿主项目已有更强约束。

## Red Flags

出现以下想法时，默认说明你在绕开主入口纪律：

- "先建几个默认文件，后面再问也行"
- "先把 BRD 或页面代码做出来，回头再补项目画像"
- "这个阶段我顺手帮用户一起写完更省事"
- "我已经知道大概是操作页 / 配置页，不用再问给谁用"
- "宿主文件不完全符合模板，干脆再建一份标准文件"
- "这一轮先不回写，等全部做完再统一整理"
- "阶段切换了，先进子能力开始干活，日志后面再补"

碰到这些念头时，停下来，回到本文件开头，重新按“一页版总规则”执行。

## 2. 阶段判断表

| 阶段 | 适用信号 | 当前轮最小交付物 | 默认承载 |
|------|----------|------------------|----------|
| **S0 需求调研** | 首次必问尚未完成，启动最小必需字段包仍有缺口；需求仍停留在访谈和业务理解层 | 需求清单 | 主入口继续承载 |
| **S0.5 既有项目基线诊断** | 已有代码接入套件；项目维护性差，缺少 BRD / 页面说明 / foundation / PRD 等关键文件 | 项目画像草稿 + 关键维护文件缺口清单 | 进入 `project-baseline-auditor`，由其生成或更新 `project-profile.md`，并产出 `docs/baseline/baseline-audit-<slug>.json` |
| **S1 业务需求文档** | 首次必问已结束，启动最小必需字段包已齐；需要形成可评审的业务需求文档 | 业务需求文档 / BRD | 默认进入 `brd-writer` 能力，由其独占产出 BRD |
| **S2 页面设计、技术地基与完整版 PRD** | 已有业务需求文档；需要先完成页面代码、交互语义冻结与页面环节收口，再反推术语表 / Schema / API，最后沉淀 AI 可直接编码的 PRD | 首轮：页面代码 / 页面交付清单 + 待确认项；页面环节收口后：术语表 / Schema / API / foundation 交付清单；最终：功能列表 + 主 PRD + 子 PRD | 先进入 `page-chief` 调度页面环节（`page-designer` → `page-explainer`，必要时回环）；页面环节 DONE 后进入 `prd-chief` 调度 PRD 环节（`foundation-builder` → `prd-writer`） |
| **S3 任务拆解与开发计划** | 完整版 PRD 已形成；需要拆成开发任务并形成开发计划 | 正式开发计划文件组 + 当前执行摘要 | 进入 `delivery-planner` 能力，由其独占产出 `main-delivery-plan-<slug>.md`、`task-kanban-<slug>.md`、`sub-delivery-plan-<slug>-<TaskID>-<short-name>.md`；主入口回写 `execution-plan.md` 驾驶舱摘要 |
| **S4 开发执行** | 正式开发计划文件组存在、通过结构校验，且由 `delivery-planner` 完成 S4 前计划一致性校验；可以进入编码、联调和实现 | 当前任务的执行结果 + 任务状态更新 + 问题/决策记录 | 主入口先触发 `delivery-planner` 校验 main plan / kanban / sub plan 三者一致；通过后才进入 `coding-standards` 能力，由其独占承接开发执行 |
| **S5 测试用例生成** | 开发执行已完成，或当前版本已具备可验证基础；需要基于 PRD 生成标准化测试用例 | 验收文档 + 单域测试用例文件 + 验收矩阵 + 版本历史 | 进入 `test-case-chief` 调度链路（`prd-acceptance-reviewer` → `test-case-writer` → `test-case-reviewer`），由其独占产出测试用例 |
| **S6 测试执行** | 测试用例已准备好；需要执行测试并记录问题 | 验收结论 + 不符合项清单 + 补缺建议 + 阶段收口建议 | 进入 `test-case-runner` 能力，由其独占产出测试结果 |
| **S7 完工前安全扫描** | 已进入完工 / 交付前收口；需要完成固定安全闸门扫描并给出放行结论 | 安全扫描报告 + `PASS / BLOCK / WAIVER` 结论 + 输入证据缺口说明 | 进入 `security-scan` 能力，由其独占产出完工前安全闸门报告 |

决策规则：
1. 多个阶段可选时，优先较早阶段
2. 首次必问未结束、启动最小必需字段包未补齐前，不跳过 S0；补齐后即可进入 S1
3. 已有代码接入且目标是补维护知识底座时，进入 S0.5；若 `baseline-audit-<slug>.json` 已存在，主入口只按清单中的 BRD / 页面说明 / foundation / PRD 缺口路由
4. 用户要求"先给方案"→ S1 或 S2；要求"拆任务""拆开发任务""制定开发计划"→ S3
5. 用户要求实现某模块但上下文不足，先补最少上下文再进入 S3/S4
6. `当前阶段`（画像已有） vs `当前最适合进入的阶段`（主入口综合判断）冲突时，先解释差异再更新画像
7. 命中 S2 时，若页面代码或 `page-delivery` 尚未产出，则本轮只能停留在 `page-chief` 链路内，由其先调度 `page-designer` 产出页面代码 / 页面交付清单，不得提前交付 foundation 或 PRD
8. 命中 S2 时，若 `page-designer` 产物已存在，但 `page-chief` 尚未判定 DONE（包括页面方向未确认、`page-explainer` 产物未齐、交互语义存在 `open`、或仍有未解决 gap），则本轮只能继续页面环节，不得进入 `prd-chief`、`foundation-builder`、`prd-writer` 或 S3
9. 命中 S2 时，只有 `page-chief` 已 DONE，才允许进入 `prd-chief`；只有 `foundation-builder` 已完成，才允许由 `prd-chief` 继续推进 `prd-writer`
10. 命中 S7 时，主入口不得把"即将完工"当成口头结论直接放行；必须进入 `security-scan`，输出固定结构的安全扫描报告和 `PASS / BLOCK / WAIVER` 结论后，才算完成完工前闸门
11. 命中 S3 并由 `delivery-planner` 产出或更新正式开发计划后，主入口必须把主开发计划入口、任务看板入口、当前子开发计划入口、当前活跃任务与下一步动作同步回 `execution-plan.md`；不得把完整开发计划文件组直接塞进执行计划驾驶舱
12. S3 驾驶舱同步时，只能读取主开发计划入口、任务看板当前 Task 行和当前子开发计划；不得从完整开发计划文件组自由总结
13. 命中 S4 时，主入口必须先触发 `delivery-planner/scripts/check-plan-consistency.mjs`，以 `s4_pre_coding_plan_consistency_check` 目的校验 `main-delivery-plan-<slug>.md`、`task-kanban-<slug>.md` 和当前 `sub-delivery-plan-<slug>-<TaskID>-<short-name>.md` 三者一致；一致性通过后，才允许进入 `coding-standards`
14. 命中 S4 但正式开发计划文件组缺失、结构校验失败或计划状态一致性失败时，主入口不得进入 `coding-standards`；必须停留开发计划修复链路，调用 `delivery-planner` 生成、修复或校正 `docs/plans/delivery-plans/` 下的正式开发计划文件组

### 2.1 历史项目标准化模式

当 S0.5 进入既有项目资料标准化链路后，baseline 刷新由 `ai-project-manager` 负责，`brd-writer`、`page-explainer`、`foundation-builder`、`prd-writer` 不感知 `project-baseline-auditor`。

运行规则：
1. 无 `baseline-audit-<slug>.json` 时，主入口先路由到 `project-baseline-auditor` 生成项目画像草稿与当前缺口清单。
2. baseline 推荐某个补档 skill，且对应维护产物仍缺失时，主入口按该推荐路由给对应 skill 独占产出正式文档。
3. 补档 skill 完成后回到主入口；若旧 baseline 推荐缺口已被对应产物满足，主入口先刷新 baseline，再按最新缺口继续路由。
4. baseline 为 `summary.status = ready` 且 `recommended_next_skill = null` 时，视为 S0.5 维护知识底座补齐；主入口收口历史项目标准化，并重新判断后续阶段。
5. `project-baseline-auditor` 只读宿主源码和项目资料，不修改源码、配置、迁移脚本或业务实现文件；它只写项目画像和 `docs/baseline/` 下的诊断产物。

### 2.2 S2 设计流水线协议

S2 不是单纯“写方案文档”的阶段，而是 **页面环节收口 → PRD 环节收口** 的联动阶段，且顺序必须固定：

1. 先基于 BRD 进入 `page-chief`，由其观察文件状态并调度 `page-designer`
2. `page-designer` 先产出页面代码、页面交付清单，以及必要的中间文件；页面方向需先获得用户明确确认
3. 页面方向确认后，仍留在 `page-chief` 链路内，由其调度 `page-explainer` 产出流程、交互语义和 gap 文件
4. 只有当 `page-chief` 判定页面环节已收口时，才允许切换到 `prd-chief`
5. 进入 `prd-chief` 后，必须先调用 `foundation-builder` 反推术语表、数据库 Schema、API 接口和 foundation 交付清单
6. 技术地基完成后，才允许由 `prd-chief` 调度 `prd-writer` 基于页面代码、交互语义和 foundation 产物产出功能列表、主 PRD 和子 PRD
7. 完整版 PRD 形成后，才允许进入 S3 任务拆解

**页面环节收口标准：**
- `page-delivery-<slug>.md` 已存在，且其中列出的页面代码文件全部存在
- `page-explainer` 必需产物已存在：`explainer-flow`、`explainer-b-interaction`、`explainer-delivery`
- 所有 interaction 文件中的语义条目状态均为 `locked`
- 若存在 gap 文件，其中不得残留未解决的 `design_gap` / `logic_conflict`
- 以上条件满足后，由 `page-chief` 判定 DONE，才允许进入 `prd-chief`

**人工确认口径：**
- 只有用户明确表示“页面方向通过”“按这个页面继续”“页面确认”之类肯定结论时，才算 `page-designer` 的页面方向通过
- `page-explainer` 各阶段涉及的流程和交互语义，同样需要用户明确确认后才能冻结为 `locked`
- “继续看看”“先放着”“差不多”这类模糊表达，不自动视为页面确认或语义冻结确认

**禁止：**
- 禁止在没有页面代码 / 页面交付清单的情况下直接输出 foundation 或完整版 PRD
- 禁止把静态模块清单当作“页面产物”替代物
- 禁止页面未经用户确认、或页面环节未被 `page-chief` 判定 DONE，就进入 `prd-chief`、`foundation-builder` 或把 PRD 写成权威版本
- 禁止跳过 `page-explainer` 直接把页面代码当成页面环节已完成
- 禁止跳过 `foundation-builder` 直接让 `prd-writer` 自行定义术语、Schema 或 API
- 禁止在页面未确认或 foundation 未完成时进入 S3
- 禁止主入口绕过 `page-chief` 直接把 `page-designer`、`page-explainer`、`foundation-builder`、`prd-writer` 拼成非受控链路
- 禁止主入口绕过 `page-designer`，直接用任何未登记的页面专项能力充当 S2 正式阶段交付

### 2.3 全局伴随能力规则

`project-devlog` 不归属于单一阶段，而是全局伴随能力。

满足以下任一条件时，本轮结束默认调用 `project-devlog`：
- 产生了新的阶段交付物
- 当前阶段发生变化
- 当前会话准备收口，且已积累足够需要交接的决策或变更
- 如果不写日志，下一轮恢复成本会明显升高

默认不触发 `project-devlog` 的情况：
- 只确认了一个访谈答案或一个待确认项
- 只做了局部措辞澄清，没有形成新的阶段结论或交付边界
- 同一主题仍在连续收敛，且项目画像 / 执行计划已能承接当前上下文
- 刚更新完画像或计划，又立刻为同一小步动作重复写一次日志

合并写入原则：
- 同一主题下的连续小决策，优先先回写到项目画像 / 执行计划，待形成阶段性结论后再合并写入一条日志
- 同一会话内若还在持续追问同一组待确认项，默认继续累积，不按“问一次、写一次日志”执行
- `project-devlog` 的价值是承担阶段切换、正式交付、跨会话承接和复盘，不是逐条复刻对话

运行环境允许时，优先通过 `tools/devlog-sync.mjs` 完成结构化写入；`project-devlog` 继续承担日志内容组织、回写口径和人工复盘规则。

`project-link-indexer` 不归属于单一阶段，而是全局伴随能力。主入口只判断调起场景，不判断索引是否需要 build / refresh / noop；索引状态判断和索引写入由 `project-link-indexer` 自己完成。

主入口每次运行 `route-check.mjs` 后必须读取 `companionActions`。若其中包含 `project-link-indexer`，必须先加载并执行该 skill；执行完成后，再继续主阶段路由或本轮收口。

`project-link-indexer` 的调起条件分两层：

**第一层：脚本可判定条件。** `route-check.mjs` 只在已识别到项目画像时判定，且每次最多返回一条伴随动作（按下列顺序取首个命中）：
- 宿主已存在可用的 baseline-audit 清单（S0.5 诊断完成后；不限本次检查的目标阶段）
- 本次检查的目标阶段为 S2，且 BRD 已存在（对应“S1 BRD 完成后”）
- 本次检查的目标阶段为 S3 / S5，且 foundation 交付清单或完整版 PRD 的门禁已通过（对应“S2 页面 / foundation / PRD 产物形成后”）
- 本次检查的目标阶段为 S4，且正式开发计划文件组的门禁已通过（对应“S3 开发计划文件组形成或修复后”）
- 本次检查的目标阶段为 S6，且测试用例门禁已通过（对应“S5 验收文档 / 测试用例形成后”）

**第二层：语义条件。** 以下场景脚本拿不到判定输入，`companionActions` 不会返回；主入口自行判断命中后，必须直接调用 indexer，不等脚本提示：
- 阶段产物刚在当前轮形成或拆分，但本轮 route-check 尚未以下一阶段为目标重跑（例如 BRD 刚落盘、阶段判断仍停留在 S1 时），按 `artifact_files_added_or_split` 触发
- 用户询问坏链、回链或孤立文件时，按 `need_broken_link_or_reverse_link_check` 触发；询问文件之间怎么关联时，按 `need_file_relationship_diagnosis` 触发；询问改某个文件会影响哪些文件时，按 `need_impact_lookup` 触发。这三个是诊断类取值，只检查不写索引文件；`--trigger` 合法取值全表见 `skills/00-03-project-link-indexer/SKILL.md`

执行入口：`node <suite-path>/skills/00-03-project-link-indexer/scripts/run-project-link-indexer.mjs <hostRoot> --trigger <trigger> --json`。

## 3. 首轮极简访谈

**触发条件**：项目画像缺失，且协议中的“启动最小必需字段包”信息不足。

**目标**：用最少问题补齐协议中的“启动最小必需字段包”，足以判断下一阶段即可停止。

若当前会话已加载本套件，且用户已表达项目启动意图，则触发后**直接开始首轮极简访谈**；不要额外发一轮“是否使用 `ai-project-manager` / 是否按套件流程开始”的确认。

### 首次必问（按顺序单问单答，最多 4 问）

| # | 问题 | 对应字段 | 参考回答示例 |
|---|------|----------|------------|
| 1 | 这个项目叫什么？ | `项目名称` | "叫门店设备巡检助手" |
| 2 | 一句话说，这个项目是做什么的？ | `项目一句话目标` | "帮门店把巡检、异常上报和进度放到一个系统里" |
| 3 | 主要给谁用？ | `目标使用者` | "门店店长和巡检人员" |
| 4 | 你现在最想先解决的业务痛点是什么？ | `主要问题` | "现在靠 Excel 和微信群，漏检和跟进不及时" |

### 展示规则
- 严格按顺序单问单答：一次只问 1 个问题，拿到回答后再决定是否进入下一问
- 不允许把“首次必问”一次性整组抛给用户，也不要改写成问卷式多问题消息
- 每次提问附 1 条参考回答，标注"仅供参考，可按实际情况改写"
- 不展示 `对应字段` 给用户
- “启动最小必需字段包”的 4 个字段都已拿到答案（或可从既有回答直接推断）时即可停止，不必逐题问完；只要字段包仍有缺口，就继续按顺序补问缺口字段
- 首次必问一旦结束，且启动最小必需字段包已齐，主入口默认不再停留 S0，而是按 S1 承接后续 BRD 形成
- 一旦 `项目名称` 已确认，且当前工作目录疑似容器目录，后续骨架补齐时默认使用 `<当前目录>/<项目名称>/` 作为物理落点，除非用户明确指定其他位置

### 禁止
- 不问技术栈/数据库/中间件
- 不发十几条问卷式问题
- 不把多个首次必问合并成一条消息连续追问
- 不把系统推断字段（规则入口/计划入口/状态入口）拿来问用户
- 不把主入口回写字段（当前阶段/当前轮交付物）丢给用户作答
- 不把“是否采用 `ai-project-manager` 流程”当成首轮问题；套件已加载时，这件事默认成立
- 不等待业务方自己主动说出页面端别；命中页面任务时，主入口必须主动牵引补问
- 信息不足时不承诺具体实现方案

---

## 4. 协作模式判断

当前产品定位只聚焦 **solo 模式**。

- 主入口默认按 `业务单人 + AI执行` 处理，不再把协作模式作为启动阶段固定补问
- 项目画像中的 `协作模式` 默认回写为 `业务单人 + AI执行`

主入口只围绕当前操作者持续推进。

## 5. 默认交付链路

当用户没有指定研发流程时，按此顺序推进：

```
项目画像 → 需求清单 → 业务需求文档 → page-chief → 页面代码 / 页面交付清单 → 页面确认 → page-explainer 冻结交互语义 / gap 收口 → prd-chief → 术语表 / Schema / API / foundation 交付清单 → 功能列表 / 主 PRD / 子 PRD → 开发计划 → 开发执行 → 验收文档 + 测试用例 → 测试执行 → 完工前安全扫描
```

输出约束：
- 每轮留下可复用文档，不只停留在对话里
- 每轮写清楚下一步，不只给方向
- 用户时间有限时，优先输出当前阶段最小交付物（见 §2 表格）
- 命中 S2 时，若 `page-chief` 尚未判定页面环节 DONE，优先输出页面代码 / 页面交付清单 / explainer 待确认项，不输出 foundation 或 PRD
