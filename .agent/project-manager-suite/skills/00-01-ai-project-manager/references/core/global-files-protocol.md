## 对应实现与执行入口

对应关系：

- 结构化实现：
  - `lib/ai-pm-protocol/field-contracts.js`
  - `lib/ai-pm-protocol/validation.js`
  - `lib/ai-pm-protocol/rules-sync.js`
  - `lib/ai-pm-protocol/change-impact-map.js`
- 对应脚本：
  - `tools/validate-global-files.mjs`
  - `tools/bootstrap-host.mjs`
  - `tools/devlog-sync.mjs`

维护原则：

- 若字段语义、字段来源、字段层级、文件角色职责发生变化，先改本文件
- 若规则没变，只是识别、校验、创建、回写动作有缺口，再改结构化实现或脚本
- 不要让校验器或写入脚本反向定义字段含义

运行环境允许时，主入口应优先使用脚本完成结构识别和执行动作：

- `tools/validate-global-files.mjs`：识别权威入口、检查结构缺口
- `tools/route-check.mjs`：消费画像和计划，判断阶段与门禁
- `tools/bootstrap-host.mjs`：按协议补齐缺失骨架
- `tools/devlog-sync.mjs`：按协议完成结构化日志回写

关系原则：

- 协议定义“字段是什么、谁负责写、什么时候该补”
- 脚本负责“识别、校验、阻断、创建最小结构、落地受控写入”
- 主入口负责“访谈、业务判断、例外处理、对子能力的交接和解释”

## 1. 3 类全局文件 + 1 类状态回写能力职责

| 文件类型 | 默认文件名 | 核心职责 | 绝不能混入的内容 |
|----------|------------|----------|-------------------|
| **全局规则文件** | `project-rules.md` | 这个项目怎么运行、权威入口在哪、必须遵守的技术/协作边界 | 单次阶段任务、本轮状态、过程日志 |
| **项目画像文件** | `project-profile.md` | 项目目标快照、协作模式、当前阶段、执行主体、待确认项 | 长期规则细节、详细任务清单、大段开发日志 |
| **执行计划文件** | `execution-plan.md` | 当前执行驾驶舱：现在跟哪份正式计划、当前做到哪、下一步做什么、完成标准摘要 | 完整 Phase / Task 正文、发布闸门明细、整份开发计划副本 |
| **状态回写能力** | `project-devlog`（输出到 `logs/`） | 负责本轮动作实录、本轮产出结论、风险卡点、下一步建议的沉淀与收口 | 覆盖基础画像重写、全量变更历史重新陈述 |

## 2. 字段层级与来源定义

### 字段层级（决定能不能缺失）
- **必填**：协议底线，缺失时主入口必须推断或补齐
- **可选**：公开版允许缺失，增强版可扩展
- **自动推断**：优先从上下文推断，尽量不追问用户
- **禁止混入**：绝对不能写在当前文件里

### 字段来源（决定谁提供信息）
- **用户确认**：需要用户拍板的事实（来自对话意图或显性已有资料）
- **系统推断**：主入口从文件/角色/上下文中自动识别出的客观结论
- **主入口回写**：主入口综合判断后得出的阶段结论、交付物类型、执行主体等

### 字段包（用于解耦运行流程与模板结构）
- **启动最小必需字段包**：`项目名称`、`项目一句话目标`、`目标使用者`、`主要问题`
- **身份识别字段包**：`身份识别口径`
- **页面任务必补字段包**：`项目覆盖对象`、`当前页面主要给谁用`、`当前页面主要用途`、`页面定位标签`
- **运行时回写字段包**：`当前阶段`、`当前最适合进入的阶段`、`当前轮应输出的交付物`、`当前任务执行主体`、`当前最大不确定项`

约束：
- `runtime.md` 只引用字段包决定“何时补齐哪些信息”，不重复维护模板章节结构。
- `project-profile.md` 只负责展示和存储字段，不重复定义访谈顺序与路由规则。
- 当当前轮目标涉及页面、原型、前端界面或 App 页面收敛时，必须先补齐“页面任务必补字段包”，再进入 S2。
- `lib/ai-pm-protocol/field-contracts.js` 是本协议的结构化实现入口；稳定字段变更应先同步那里，再由工具层消费。
- `validate-global-files.mjs` 当前以结构性校验为主；字段语义仍以本协议和 `field-contracts.js` 为准，不应反过来由校验器定义字段含义。

## 3. 最小字段合同（全景表）

> [!IMPORTANT]
> 无论是主入口生成，还是高级代理修改，都必须遵循该合同结构。

### 3.1 全局规则文件 (`project-rules.md`)
| 字段名 | 层级 | 来源 |
|--------|------|------|
| 规则权威文件 / 计划权威文件 / 状态回写权威文件 | **必填** | 系统推断 |
| 是否要求每轮沉淀交付物 | **必填** | 系统推断 |
| 协作边界（长期有效） | 可选 | 用户确认 |
| 其他入口 / 核心目录结构 / 文档代码边界 | 可选 | 系统推断 |
| 默认工作语言 / 默认沟通方式 | 自动推断 | 系统推断 |

### 3.2 项目画像文件 (`project-profile.md`)
| 字段名 | 层级 | 来源 |
|--------|------|------|
| 项目名称 / 项目一句话目标 / 目标使用者 / 主要问题 | **必填** | 用户确认 |
| 协作模式 | **必填** | 系统推断（当前默认 `业务单人 + AI执行`） |
| 项目覆盖对象 / 当前页面主要给谁用 / 当前页面主要用途 | 可选（页面类任务时必补） | 用户确认 |
| 页面定位标签 | 可选（页面类任务时必补） | **主入口回写** |
| 第一版核心目标 | 可选 | 用户确认 |
| 当前阶段 / 当前最适合进入的阶段 / 当前轮应输出的交付物 / 当前任务执行主体 | **必填** | **主入口回写** |
| 第一版范围 | 可选 | 用户确认 |
| 当前最大不确定项 / 待确认项 | 可选 | **主入口回写** |
| 身份识别口径 / 已有文档、原型、材料清单 / 规则入口、计划入口、状态入口、PRD 入口 | 自动推断 | 系统推断 |

### 3.3 执行计划文件 (`execution-plan.md`)

`execution-plan.md` 的定位是**当前执行驾驶舱**，不是完整开发计划正文。

- 它负责记录：当前跟哪份正式计划执行、当前活跃 Phase / Task、下一步动作、阻塞与完成标准摘要
- 它不负责维护完整 Phase / Task 列表、完整发布闸门、完整风险与反向索引
- `main-delivery-plan-<slug>.md` 是 S3 `delivery-planner` 产出的正式开发计划入口，位于 `docs/plans/delivery-plans/`
- `task-kanban-<slug>.md` 是任务看板，承接 Task 状态与子开发计划入口，位于 `docs/plans/delivery-plans/`
- `sub-delivery-plan-<slug>-<TaskID>-<short-name>.md` 是单个 Task 的执行正文，位于 `docs/plans/delivery-plans/`

| 字段名 | 层级 | 来源 |
|--------|------|------|
| 当前正式计划文件组（若未生成则写 `待生成`） / 当前阶段 / 当前目标 / 当前活跃 Phase / Task / 当前子开发计划 / 下一步任务 / 完成标准摘要 | **必填** | **主入口回写** |
| 当前阻塞 / 前置依赖 / 待确认项 | 可选 | **主入口回写** |
| 可并行任务 / 任务优先级 / 候选后续任务 | 自动推断 | 系统推断 |

章节落点说明（与默认模板 `assets/global-files/execution-plan.md` 和结构校验器保持一致）：
- `当前活跃 Phase / Task` 落在驾驶舱的「进行中任务」章节，`完成标准摘要` 落在「完成标准」章节；两组名称同义，工具读取时两种章节名都接受，模板默认使用「进行中任务」「完成标准」
- `当前正式计划文件组` 是独立章节，内含主开发计划、任务看板、当前子开发计划三个入口条目；未生成时写 `待生成`
- 结构校验（`validate-global-files.mjs` 会检查这些章节 / 条目是否存在，缺失时报 `missing_required_markers`）以上述章节名为准

### 3.4 状态回写能力 (`project-devlog`)
| 字段名 | 层级 | 来源 |
|--------|------|------|
| 本轮时间 / 本轮动作 / 本轮产出 / 当前结论 / 下一步建议 | **必填** | **主入口回写** |
| 风险与阻塞 / 待确认项 / 本轮决策 / 本轮验收结论 | 可选 | **主入口回写** |
| 本轮涉及文件 / 本轮关联阶段 / 本轮执行主体 | 自动推断 | 系统推断 |

默认落地载体：
- 宿主项目已有日志目录时，`project-devlog` 必须优先写入宿主映射后的权威日志文件。
- 若宿主项目没有状态回写载体，`project-devlog` 默认在 `logs/` 下创建日志文件，不创建 `project-status.md`。

## 4. 统一读写规则

### 4.1 读与写的责任人

| 文件类型 | 谁负责读 | 谁负责写（或主导更新） |
|----------|----------|------------------------|
| 全局规则 | 主入口、全部子能力 | 主入口、项目维护者（人工） |
| 项目画像 | 主入口、计划/需求类能力 | **主入口主导**，`project-baseline-auditor` 可在既有代码接入时受控生成或补充，其余能力受控变更 |
| 执行计划（驾驶舱） | 主入口、计划/执行/验收类 | `ai-project-manager` 主导维护；其他能力只提供输入，不直接改驾驶舱正文 |
| 状态回写 | 主入口、全部子能力 | 默认由 `project-devlog` 承接；主入口或实际发生动作的执行单元负责提供输入 |

补充规则：
- `main-delivery-plan-<slug>.md`、`task-kanban-<slug>.md`、`sub-delivery-plan-<slug>-<TaskID>-<short-name>.md` 不属于 3 类全局文件之一，而是 S3 阶段的正式交付产物。
- `docs/plans/foundation-plans/foundation-change-requests-<slug>.md` 不属于 3 类全局文件之一，而是 S4 反哺 foundation 的待改 backlog；`coding-standards` 只追加条目，`foundation-builder` 只读 `待评审` 条目，`ai-project-manager` 在消费点翻状态并报告用户裁决；条目字段口径以 `skills/06-01-coding-standards/SKILL.md` 为准。
- `execution-plan.md` 不得复制完整开发计划文件组，只保留驾驶舱摘要和入口。
- `delivery-planner` 不得用 `execution-plan.md` 替代正式开发计划文件组。

禁止项：
- 若宿主项目已存在同职责全局文件，禁止再额外创建一份默认模板文件并并行维护。
- 子能力不得绕过主入口已确认的映射关系，自行按默认文件名创建第二权威源。
- 除非宿主项目已有更强约束，状态回写类任务不得绕过 `project-devlog` 直接自由写入状态载体。

脚本优先口径：
- 权威入口识别和结构校验，优先由 `validate-global-files.mjs` 执行。
- 阶段门禁依赖的字段读取，优先由 `route-check.mjs` 消费已识别入口执行。
- 目录骨架与最小模板创建，优先由 `bootstrap-host.mjs` 执行。
- 日志落地与规则候选池联动，优先由 `devlog-sync.mjs` 执行。
- 若脚本不可用，主入口才按本协议手工执行等价动作。

### 4.2 文件生命周期 / 更新时机

1. **项目启动且未明确结构时** → 先识别宿主现有全局文件的角色映射关系，不直接按默认文件名创建文件
2. **缺项目画像角色文件时** → 必须先完成必要访谈，再创建最小画像载体；若已有同职责权威文件，直接认定为项目画像文件，不重复创建 `project-profile.md`
3. **已有代码接入且需要补维护知识底座时** → 可由 `project-baseline-auditor` 基于代码生成或更新同一个 `project-profile.md`；必须保留 `【用户确认】` 字段，代码反推内容只能写为 `【系统推断】`，未能证明的内容进入单焦点待确认
4. **缺全局规则角色文件时** → 仅在确有长期规则承载需要，且无法通过现有文件映射时，才创建最小规则载体；若已有同职责权威文件，直接复用
5. **项目启动且缺计划载体时** → 将执行计划文件作为启动记忆骨架与当前执行驾驶舱补齐；若已有同职责权威文件，直接复用
6. **本轮推进需要状态回写时** → 由 `project-devlog` 承接；何时写入（默认只在阶段切换、正式交付、会话收口或需要交接时合并写入，小步动作先在画像 / 执行计划中累积）统一以 `runtime.md` §2.3「全局伴随能力规则」为准；优先写回已有日志文件，仅在无可映射载体时于 `logs/` 下补最小日志文件
7. **协作大范围调整、项目阶段跃升** → 回写项目画像文件或其宿主映射文件
8. **任务完成或目标调整** → 更新执行计划文件或其宿主映射文件
9. **S3 正式开发计划形成或更新后** → `delivery-planner` 负责维护 `main-delivery-plan-<slug>.md`、`task-kanban-<slug>.md`、`sub-delivery-plan-<slug>-<TaskID>-<short-name>.md`；`ai-project-manager` 负责把“主开发计划入口 + 任务看板入口 + 当前子开发计划入口 + 当前活跃摘要”同步回 `execution-plan.md`

执行计划驾驶舱同步规则：
- 仅在以下事件发生时同步 `execution-plan.md` 摘要：首次生成正式开发计划、当前活跃 Phase / Task 变化、阻塞状态实质变化、阶段跨越、S2 内部链路从页面环节切换到 PRD 环节
- `page-chief` 判定页面环节 DONE 时，属于 S2 内部链路切换；主入口必须同步 `execution-plan.md`，把当前目标、进行中任务、下一步任务、完成标准和待确认项从 page-designer / page-explainer 页面语境更新为 `prd-chief` / `foundation-builder` 语境
- 同步来源必须是主开发计划入口、任务看板当前 Task 行和当前子开发计划，不从完整开发计划文件组自由提炼
- 若只是风险补充、说明文字修订、反向索引调整等不影响“今天怎么继续做”的变更，不同步驾驶舱

若脚本可用，推荐执行顺序为：

1. `validate-global-files.mjs`
2. `route-check.mjs`
3. 仅在确有骨架缺口时执行 `bootstrap-host.mjs`
4. 一轮有效推进结束后执行 `devlog-sync.mjs`

> [!TIP]
> 永远**先识别现有体量，非必要不新建**（已有 `docs/rules.md` 就映射为全局规则文件，已有 `logs/day1.md` 就作为 `project-devlog` 的状态回写载体）。

### 4.3 缺口处理原则与默认模板

当找不到某个文件角色，且当前阶段确实需要该载体时，模板文件有且只有两种用途：

- **映射用途**：宿主项目已有对应职责文件时，模板仅作为字段合同、角色识别和指向关系参考，不创建物理文件。
- **创建用途**：宿主项目不存在对应职责文件，且已满足该角色文件的前置条件时，才允许基于模板创建最小载体。

默认模板入口如下：

*   全局规则参考模板：`skills/00-01-ai-project-manager/assets/global-files/project-rules.md`
*   项目画像参考模板：`skills/00-01-ai-project-manager/assets/global-files/project-profile.md`
*   当前执行计划参考模板：`skills/00-01-ai-project-manager/assets/global-files/execution-plan.md`

前置条件约束：
- `project-profile.md`：必须先完成最小访谈。若通过 `bootstrap-host.mjs` 创建，还必须提供可校验的结构化访谈输入，并覆盖“启动最小必需字段包”；单独的完成标记不构成创建依据。
- `project-rules.md`：仅在确有长期规则承载需要，且无法通过现有文件映射时才创建。
- `execution-plan.md`：属于启动骨架与 AI 持续记忆系统关键文件；若宿主缺失，应在初始化时创建最小驾驶舱载体。
- `logs/`：由 `project-devlog` 作为默认状态回写能力负责落地；不阻断当前轮；仅在本轮结束后仍找不到可映射状态载体时，于日志目录补最小日志文件。

脚本对应关系：
- `bootstrap-host.mjs` 负责执行目录创建、规则目录补齐和受控模板创建
- `generate-host-rules.mjs` 负责补齐宿主 `docs/rules/` 默认规则文件
- `bootstrap-host.mjs` 应复用 `generate-host-rules.mjs`，不重新复制一套规则同步逻辑

默认文件名说明：
- 默认文件名只是缺失时的创建落点，不代表宿主项目必须存在该命名文件。
- 全局文件角色与具体文件路径分离，识别职责优先于匹配文件名。
- 状态回写默认先决定能力归属，再决定落地文件路径；`project-devlog` 是默认承接能力，`logs/` 是其默认落地位置。

### 4.4 宿主专项规则生成协议

专项规则文件不属于 3 类全局文件本体，但属于宿主项目的默认规则资产。

- 套件默认专项规则源目录：`skills/00-01-ai-project-manager/references/rules/`
- 宿主专项规则权威目录：`docs/rules/`
- 当宿主项目首次创建 `docs/rules/`，或宿主缺少默认专项规则文件时，应从套件默认专项规则源生成同名规则文件到宿主目录
- 生成策略默认只补齐缺失文件，不覆盖宿主已存在的同名规则文件
- 一旦宿主存在同名规则文件，后续运行时读取应优先使用宿主文件，套件默认专项规则源仅作为缺失时的兜底回退
- `project-rules.md` 只声明专项规则入口，专项规则细则应沉淀在宿主 `docs/rules/`，不在规则层重复维护副本
- 若脚本可用，默认通过 `generate-host-rules.mjs` 或 `bootstrap-host.mjs` 完成该动作，而不是由主入口手工逐文件创建

**降级兜底**：不要因为个别可选字段未搜集到就死循环发问，优先填写为 `待确认` 以免阻断主流程。
