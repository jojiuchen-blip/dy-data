# 文档与计划任务规则包

## 适用任务

- 执行计划修订
- 需求 / 方案文档修订
- 文档交叉引用同步
- 任务状态回写
- 缺陷单、日志、计划的闭环更新

## 加载时机

当任务涉及执行计划、需求文档、方案文档、缺陷单、任务状态、完成标准、文档结构同步时，必须在读取全局入口后加载本文件。

## 执行顺序

1. 先读取项目规则文件
2. 读取本文件
3. 读取当前执行计划、项目画像与相关文档
4. 确认文档之间的引用关系
5. 修改文档并回写状态

## 核心规则

### 1. 所有任务转述必须回溯原始计划

编写待办事项、判断下一步、转述任务内容时，必须回溯原始计划文档。

禁止凭对话摘要、记忆或阶段性总结照搬任务措辞。

### 2. 任务状态回写是计划体系的一部分

任务完成后必须同步回写当前执行计划文件中的对应任务状态。

若该任务影响下游前置关系或 Phase 状态，需在同次修改中一并更新。

### 3. 完成标准变更必须显式改文档

若联调环境、验收口径、依赖关系发生变化，不能只在对话里说明。

必须回写到计划文档的完成标准或任务说明中，否则后续执行会继续沿用旧口径。

### 4. 计划与需求 / 方案依据必须形成双链

执行计划中的任务必须能追到对应需求或方案章节。

若计划缺少任务级需求 / 方案锚点，AI 容易脱离业务上下文实现错误逻辑。

### 5. 文档修改要同步交叉引用

修改任何章节号、表名、路由、接口路径等结构信息时，必须全局搜索并同步更新关联文档。

### 6. 修复闭环必须反向更新状态文档

修复缺陷后，必须同步更新缺陷单、计划状态或其他跟踪文档。

否则后续成员会继续重复排查同一问题。

### 7. 验收口径升级必须写回计划，不得停留在对话里

若本次任务补充了新的必测项，例如：
- 真实数据库查询验证
- 真实保存链路验证
- 页面加载后回显验证
- 接口 `code / message / data` 口径核对

则必须同步回写到计划文档或任务完成标准中。

否则后续执行仍会沿用旧口径，重复出现“构建通过但联调失败”的问题。

### 8. 完成标准必须避免抽象描述

当任务涉及固定配置表、固定枚举表、直接驱动页面的配置表时，完成标准不能只写：
- “固定行数表已补齐”
- “配置表数据完整”
- “关键种子已导入”

必须写成可核查的显式条目，例如：
- `button_strategy=3`
- `primary_label_config=3`
- `detail_button_config=5`

否则执行时无法判断具体漏了哪张表，容易把数据库 readiness 问题拖到前端联调阶段才暴露。

## 推荐检查清单

- 任务标题与完成标准是否还是当前真实口径
- 需求 / 方案依据是否还能准确定位到需要读取的章节
- 关联文档是否已经同步更新
- 状态回写是否会影响次日启动判断

## 禁止行为

- 禁止只修代码不修计划状态
- 禁止只在日志里记录变更而不更新计划文档
- 禁止修改结构信息后不查交叉引用
- 禁止把新增验收要求只留在聊天记录，不回写到计划或规则文档
- 禁止在完成标准中使用无法核查的抽象验收表述
