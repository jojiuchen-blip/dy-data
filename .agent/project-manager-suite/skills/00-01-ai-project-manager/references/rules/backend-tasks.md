# 服务端开发任务规则包

## 适用任务

- 服务端接口开发与改造
- `Controller / Service / Mapper` 链路实现
- 聚合接口、配置接口、数据管理接口开发
- DTO、查询逻辑、配置读取链路改造

## 加载时机

当任务进入服务端写代码阶段，必须在读取全局入口后加载本文件。

## 执行顺序

1. 先读取项目规则文件
2. 读取本文件
3. 读取当前执行计划中的服务端任务
4. 读取任务对应的需求依据、接口方案、数据库 Schema 或相关设计资料
5. 明确数据来源、兜底逻辑、配置链路后再写代码

## 核心规则

### 1. 先读业务链路，再写接口代码

写服务端代码前，必须能明确回答：
- 这条接口要实现哪条业务规则
- 数据来自哪些表或外部源
- 哪些字段由配置表驱动
- 兜底逻辑是什么

答不上来时，停止写代码，先补读需求或方案依据。

### 2. 严守分层职责

- `Controller`：接收请求、参数校验、返回响应
- `Service`：承接业务规则与聚合逻辑
- `Mapper`：负责数据库访问，不承担业务拼装

禁止把业务规则直接塞进 Controller 或 Mapper。

### 3. 接口字段必须对齐计划与需求依据

开发或改造接口时，响应字段必须以当前执行计划和接口需求 / 方案依据为准。

禁止凭当前代码现状或前端已有字段反推接口契约。

### 4. 前端是纯渲染层，服务端必须输出 display-ready 数据

若需求或方案已明确某项逻辑由服务端负责计算，服务端必须直接输出可渲染结果。

禁止把状态判定、标签匹配、按钮拼装等逻辑甩回前端。

### 5. 配置链路必须真实接表

若需求或方案说明字段应来自配置表，服务端实现时必须真实读取配置表。

禁止为了先跑通而临时硬编码文案、颜色、图标、按钮或标签。

### 6. 写完代码前先核对下游依赖

服务端接口若被前端页面或其他任务依赖，必须在回写状态前确认：
- 字段名已稳定
- 配置表或数据表已就绪
- 计划前置关系满足

## 最小验收口径

至少验证：
- 接口返回结构与需求 / 方案 / 计划一致
- 关键配置字段来自真实配置表
- 主要业务分支可通过现有测试数据验证
- 任务状态已回写到计划文档

### 7. 涉及数据库或配置表的接口，必须做真实链路验证

只要接口读取配置表、业务表、种子表，或保存后会影响数据库状态，在宣称完成前必须做一次真实链路验证。

最低要求：
- 至少真实调用一次接口，确认 `code / message / data` 与前端约定一致
- 至少核对一次实体、Mapper、SQL 与真实表结构一致，不能只依赖 mock 单测推断 ORM 映射正确
- 若接口是读取类，必须验证关键字段确实来自真实表数据，而不是默认值、旧字段或代码兜底假象
- 若接口依赖枚举值、状态值、类型值、条件键等口径字段，必须核对这些值在需求依据、配置表、业务表、聚合表、接口返回之间一致；不能只看到“有值”或“接口 200”就视为通过
- 若接口是保存类，必须执行一次真实保存请求，并通过接口回读或数据库查询确认已正确落库
- 若接口被前端页面直接消费，至少完成一次最小上下游对接验证，确认页面不会因返回码、字段名或枚举值口径不一致而误判

若真实链路问题的根因落在数据库口径而非当前代码实现，必须先修正数据或补齐幂等修复脚本，再重新执行接口验证；禁止只在服务端临时兜底后跳过源数据回查。

### 8. 单测通过不能替代真实数据库验证

Controller / Service 单测、MockMvc、Mockito 只能证明局部逻辑。

遇到以下场景，必须补真实环境验证：
- MyBatis Plus 实体字段映射
- SQL 查询字段与真实表结构一致性
- 配置表固定枚举、种子值、行数依赖
- 业务表与聚合表之间的枚举值口径一致性
- 前后端对 `success code`、`message`、空数据返回的契约一致性

若未完成上述验证，任务不得标记为“接口已真实验证完成”。

## 禁止行为

- 禁止未读对应需求或方案依据就直接写接口
- 禁止用硬编码代替配置驱动
- 禁止在未核对完成标准前推进下一服务端任务
- 禁止只凭单测、Mock 数据或日志推断，就宣称接口读写链路已完成
