# 开发日志任务规则包

## 适用任务

- 写日志
- 总结今天
- 日志收口
- 当日任务补充更新

## 加载时机

当用户明确要求“写日志 / 总结今天 / 日志收口”，或完成了一个有实质产出的任务准备追加到当日日志时，必须在读取全局入口后加载本文件。

## 核心规则

### 1. 日志类任务必须优先命中 `project-devlog`

日志、日报、开发总结类任务，不应直接按印象写作。

必须先按 `project-devlog` 的结构与规则执行。

### 2. 日志文件按天分离

宿主项目已有日志目录时，优先沿用宿主项目位置。默认建议文件位置：
- `logs/YYYYMMDD_refactor_log_<用户名>.md`

`<用户名>` 是当前操作者标识（默认取 git 配置里的 `user.name`）；该命名与套件日志回写脚本 `tools/devlog-sync.mjs` 实际生成的文件名一致，同一天同一操作者只应存在一份日志文件。

### 3. 追加而非覆盖

若当日日志已存在，后续产出应在文件末尾追加“补充更新”章节。

不得覆盖已有日志内容。

### 4. 收口时必须汇总真实记录

用户说“写日志”或“总结今天”时，应汇总：
- 当日提交记录
- 当日已完成任务
- 复盘与待跟进事项

### 5. 日志会影响次日启动判断

每日启动规则依赖最近一天日志中的“待跟进事项”。

因此日志中的任务状态、待办项和计划状态必须保持一致。

### 6. 命中规则升级信号时必须同步写入复盘池

如果日志中的复盘、经验总结、补充更新里出现以下任一表述：
- 建议提炼为规则
- 建议写入规则
- 建议升级为规则

则必须在同次任务中同步更新宿主项目已有的规则候选承载位置。若宿主项目尚无固定承载方式，默认创建独立候选池文件：
- `docs/retrospectives/rule-candidates-YYYYMM.md`

执行约束：
- 若宿主项目已有固定承载方式（例如状态日志中的固定章节，或独立候选池文件），优先沿用，不额外创建第二份权威源
- 若需首次创建独立候选池文件，请参考 `skills/00-01-ai-project-manager/assets/devlog/rule-candidates-template.md`
- 更新候选池时，优先合并同主题已有候选，避免同月重复造多条近义规则
- 候选池只记录“待升级规则”，正式采纳后应补记其最终落点

未完成该动作前，本次日志不算收口完成。

## 推荐检查清单

- 是否使用了正确的文件名
- 是否是追加而不是覆盖
- 是否同步更新执行概要
- 是否记录了复盘与待跟进事项
- 是否与计划状态一致
- 若出现规则升级信号，是否已同步更新规则候选记录

## 禁止行为

- 禁止绕过 `project-devlog` 直接自由发挥
- 禁止写完日志却不反映真实计划状态
- 禁止把未完成任务写成已完成
- 禁止在日志中写“建议提炼为规则”后不更新复盘候选池
