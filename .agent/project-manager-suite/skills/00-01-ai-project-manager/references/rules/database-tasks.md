# 数据库任务规则包

## 适用任务

- 数据库建表、改表、迁移、种子补齐
- `init_schema.sql`、`mock_data.sql`、修复 SQL 的编写与执行
- 云端开发数据库的数据验收与回归验证

## 加载时机

当任务涉及数据库结构、种子数据、迁移脚本、真实数据库验证、数据修复时，必须在读取全局入口后加载本文件。

## 执行顺序

1. 先读取项目规则文件
2. 读取本文件
3. 读取当前执行计划中的数据库任务
4. 按计划中的需求 / 方案依据读取数据库 Schema、接口定义和相关设计资料
5. 开始修改 SQL、执行修复、做主库验收

## 核心规则

### 1. 真实运行环境切换优先于任务完成判断

若项目实际真实运行环境已从本地切换到云端，必须先同步更新计划中的完成标准和验收口径。

在未显式更新前，一律以真实联调主环境为准，不得沿用旧的本地口径宣称任务完成。

### 2. 目标开发数据库验收是唯一完成依据

数据库结构、种子、迁移类任务在标记完成前，必须对目标联调主库执行验收查询，并记录关键依赖表的行数或结构证据。

临时库、临时表、演练库只能证明脚本可执行，不能替代开发数据库就绪验收。

对于固定行数配置表、直接驱动页面的配置表，计划或日志中的验收记录必须写明：
- 具体表名
- 目标行数
- 当前开发数据库实际行数

禁止只写“固定行数表已补齐”这类抽象表述。

### 3. 脚本可执行不等于主库已就绪

脚本在临时库执行成功，只能说明 SQL 语法和基本数据链路可用。

任务是否完成，必须看开发数据库是否达到了计划中定义的完成标准。

### 4. 主库已有业务数据时，不得粗暴重跑全量 mock

若主库中已存在业务测试数据，优先采用定向、幂等、可重复执行的补齐脚本。

禁止为了补齐少量缺失配置而直接重跑整份全量 mock，避免撞重复键或污染已有数据库数据。

### 5. 迁移验证样例不得长期残留在开发数据库

像 `T1.3` 这类迁移验证脚本产生的演练样例，只能用于验证迁移行为。

验证完成后必须清理，避免开发数据库行数虚高、误导下游接口开发与验收。

## 推荐验收步骤

### A. 结构类任务

至少验证：
- 目标表存在
- 字段名、类型、是否可空与需求 / 方案依据一致
- 索引或唯一约束存在

### B. 种子类任务

至少验证：
- 固定行数表的记录数
- 关键枚举或唯一键是否完整
- 配置字段是否可被下游接口读取
- 代码中依赖这些表的实体字段、枚举值与真实开发数据库一致

若这些表直接驱动页面展示或接口匹配，验收记录中必须逐张列出表名与目标行数，例如：
- `button_strategy=3`
- `detail_button_config=5`
- `primary_label_config=3`

若这些表还承担枚举源或条件匹配源职责，必须额外核对：
- 需求或方案中的目标枚举值
- 源业务表真实枚举值
- 聚合表 / 缓存表真实枚举值
- 下游配置表引用值

禁止只核对“字段存在”或“记录数正确”，而不检查值域是否一致。

### C. 迁移类任务

至少验证：
- 迁移脚本可重复执行
- 迁移前后核心字段一致
- 演练样例是否清理

### D. 联调支撑类任务

若数据库任务会直接影响接口查询、页面回显或保存链路，至少验证：
- 真实开发数据库中的字段名、可空性、默认值与代码实体映射一致
- 固定行数表的真实枚举值与前后端常量一致
- 若存在源业务表和聚合表并存，必须确认两边枚举值口径一致；若不一致，优先用定向幂等 SQL 修复源数据，再做下游接口复验
- 保存类接口落库后，回读结果与页面或接口期望一致
- 不能让“SQL 已执行”替代“下游真实可用”

## 推荐记录方式

数据库任务完成时，建议在日志或计划回写中记录：
- 验收环境：本地 / 云端开发数据库 / 临时库
- 关键表行数
- 是否为幂等脚本
- 是否残留演练数据

## 禁止行为

- 禁止只凭“我执行过 SQL”就宣称数据库任务完成
- 禁止只在临时库验证后回写主计划状态
- 禁止在未核对完成标准前推进下一数据库任务
- 禁止忽略代码实体与真实表结构的偏差，导致下游接口在运行期才暴露 ORM/字段错误
- 禁止把固定配置表的验收写成抽象概述而不列出具体表名和目标行数
- 禁止发现真实问题后只修下游接口表象，不修开发数据库中的源枚举或聚合枚举口径
