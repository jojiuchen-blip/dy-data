# 故障排查任务规则包

## 适用任务

- Bug 排查
- 前后端对接异常
- 测试失败
- “看起来完成了但实际没完成”的状态失真问题
- 多组件、多数据源、多环境之间的异常定位

## 加载时机

当出现任何异常、失败、和预期不一致或结果可疑时，必须在读取全局入口后加载本文件。

## 执行顺序

1. 先读取项目规则文件
2. 读取本文件
3. 读取当前任务对应规则包
4. 回查计划、日志、最近改动
5. 先取证、再定位根因、最后修复

## 核心规则

### 1. 不允许跳过根因调查直接修

发现异常时，先确认：
- 错误现象是什么
- 是否可稳定复现
- 发生在哪个环境
- 最近改了什么

在没有证据前，不得凭印象直接打补丁。

### 2. 优先查真实环境，而不是先猜

若问题与前后端对接、数据库、部署环境、配置源有关，优先查询真实环境状态。

不要只凭脚本、代码、日志片段推测结果。

### 3. 多层系统要按链路逐层取证

排查顺序建议：
- 计划与完成标准
- 当前环境状态
- 数据源 / 配置表
- 服务端逻辑
- 前端渲染

每层都要回答“输入是什么、输出是什么、哪里开始偏离”。

### 4. 修复前先确认是规则问题、实现问题还是状态问题

许多异常不一定是代码 bug，也可能是：
- 完成标准过期
- 计划状态误写
- 数据库数据未落地
- 日志与计划口径不一致

先把问题归类，再决定改哪里。

### 5. 修完后必须反向验证上下游

修复完成后，至少检查：
- 根因是否真的消失
- 是否影响下游任务判断
- 计划、日志、缺陷单是否需要回写

## 推荐取证清单

- 当前环境：本地 / 云端开发数据库 / 临时库
- 计划中的完成标准
- 最近相关提交
- 关键表行数 / 关键接口返回
- 计划状态与日志状态是否一致

## 禁止行为

- 禁止把症状修复当根因修复
- 禁止连续多次“试试看”式补丁而不回查证据
- 禁止修完代码却不回写状态文档
