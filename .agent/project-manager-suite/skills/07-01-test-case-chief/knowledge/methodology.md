# 测试用例设计方法论

## 1. 方法论总览

测试用例设计有 5 种核心方法，每种适用于不同场景。实际项目中通常需要组合使用。

---

## 2. 等价类 + 边界值

### 适用场景
- 接口参数校验（必填/选填/格式）
- 分页参数（page/pageSize）
- 计数类接口
- 空值/null 校验
- 列表长度（0/1/N）

### 设计思路

将输入划分为"等价类"（同类输入预期相同结果的分组），然后在每个类的**边界**处取值：

```
输入: userId 参数
├── 有效等价类: 有订单的用户 → 返回订单列表
├── 有效边界: 只有 1 个订单的用户 → 返回长度 1 的数组
├── 无效等价类: 无订单的用户 → 返回空数组
└── 异常类: 不传 userId → 返回错误码
```

### 用例组织模式

典型的等价类+边界值用例组：
1. **正常 N 值** — 多条数据正常返回（Correct）
2. **边界 1 值** — 只有 1 条数据（Border）
3. **边界 0 值** — 空数据 / 空数组（Border）
4. **异常输入** — 参数缺失 / 格式错误（Error）

### 预期结果要点

- 精确到数组长度、字段数量、数据类型
- 空结果必须区分 `[]`（空数组）vs `null` vs 错误码
- 参数错误必须验证 HTTP 状态码和错误提示

## 3. 决策表

### 适用场景
- 多条件组合判定（如状态判定规则）
- 配置驱动的展示规则
- 多角色业务条件组合
- 徽章/标签命中条件

### 设计思路

列出所有条件变量和动作结果，构建条件组合矩阵：

```
条件:
  C1: 是否已下单           [是/否]
  C2: 订单状态             [PENDING/PAID/SHIPPED/DELIVERED]
  C3: 是否超过配送时效      [是/否]
  C4: 是否在售后期内        [是/否]

动作:
  A1: status = ?
  A2: badgeColor = ?

决策表:
| C1  | C2        | C3  | C4  | → status          | → color |
|-----|-----------|-----|-----|-------------------|---------|
| 否  | -         | -   | -   | NO_ORDER          | gray    |
| 是  | PENDING   | -   | -   | AWAIT_PAYMENT     | amber   |
| 是  | PAID      | -   | -   | PROCESSING        | blue    |
| 是  | SHIPPED   | 否  | -   | IN_TRANSIT        | blue    |
| 是  | SHIPPED   | 是  | -   | DELAYED           | red     |
| 是  | DELIVERED | -   | 是  | WITHIN_WARRANTY   | green   |
| 是  | DELIVERED | -   | 否  | COMPLETED         | gray    |
```

### 用例组织模式

决策表中的每一行 → 一条用例。条件组合就是前提条件，动作结果就是预期。

### 预期结果要点

- 每个条件组合独立成用例，便于定位失败
- 预期中要覆盖该组合下所有受影响的字段，不只是状态字段
- 标注"-"表示该条件不影响结果（don't care）

## 4. 状态迁移法

### 适用场景
- 状态流转（如订单状态流转）
- 模式切换（如普通模式 ↔ 售后模式）
- 生命周期管理
- 进度推进（如物流进度 4 节点）

### 设计思路

绘制状态迁移图，识别所有合法的状态转换：

```
PENDING ──[支付]──→ PAID ──[发货]──→ SHIPPED ──[签收]──→ DELIVERED
    │                                  │                    │
    └──[取消]──→ CANCELED               └──[超时]──→ DELAYED │
                                                       [申请售后]
                                                             ▼
                                                        REFUNDING
```

### 用例组织模式

- 每个合法转换 → 一条用例（验证转换后的状态正确）
- 非法转换 → 一条用例（验证系统拒绝或忽略）
- 关注转换触发条件和转换后的副作用

### 预期结果要点

- 验证目标状态的所有相关字段（不只是 status 字段）
- 验证转换的副作用（如切换收货地址后总价和时效联动刷新）
- 关注"转换过程中的中间态"是否可见（如骨架屏）

## 5. 场景法

### 适用场景
- 端到端用户操作链路
- 多步骤交互流程
- 涉及多个组件协作的场景
- 跨端联动链路

### 设计思路

按用户的操作路径构建场景：

```
场景: 用户切换收货地址
  1. 用户打开订单结算页 → 看到当前默认地址
  2. 点击切换地址入口 → 弹出地址列表
  3. 选择另一个地址 → 弹窗关闭
  4. 等待运费重算 → 订单总价 / 配送时效区域显示新地址对应值
```

### 用例组织模式

- 主流程（Happy Path）→ 一条完整场景用例
- 分支路径（如取消操作）→ 独立用例
- 异常路径（如切换中网络断开）→ 独立用例

### 预期结果要点

- 每个步骤后都要有检查点
- 最终状态要覆盖所有受影响的区域
- 交互响应要验证视觉反馈（如骨架屏、loading）

## 6. 错误推测法

### 适用场景
- 历史 bug 的回归验证
- 已知薄弱点的强化测试
- 经验驱动的边界探索

### 设计思路

基于已知的缺陷历史或经验，推测系统可能出错的位置：

```
历史教训:
  - 筛选面板在列表为空时没有隐藏 → 测空列表下筛选面板可见性
  - 切换地址后旧运费残留 → 测切换后总价 / 时效同步更新
  - 详情按钮 link 中 {orderId} 占位符未替换 → 测 link 中不含模板变量
```

### 用例组织模式

- 每个已知缺陷 → 至少一条回归用例
- 每个薄弱点 → 一组防御性用例
- 标注缺陷编号，便于追踪

## 7. 方法论选型矩阵

在实际项目中，一个测试文件通常需要组合多种方法。以下是选型指引：

| 验收点特征 | 首选方法 | 辅助方法 |
|-----------|---------|---------|
| 接口返回 N 个字段 | 等价类 | — |
| 接口参数必填/选填 | 边界值 | — |
| 多条件决定状态 | 决策表 | 边界值（条件边界） |
| 多状态之间切换 | 状态迁移 | 场景法（切换流程） |
| 用户操作流程 | 场景法 | 错误推测（异常分支） |
| 配置驱动展示 | 决策表 | 场景法（端到端验证） |
| 历史缺陷 | 错误推测 | — |
| 空值/兜底展示 | 边界值 | — |
| 分页/翻页 | 边界值 | — |
| 多类型筛选 | 等价类 | — |

### 组合示例

以"订单状态卡片区"为例：
- 7 态展示规则 → **决策表**（每态一条用例验证所有字段）
- 新用户 vs 老用户文案 → **边界值**（orderCount = 0 vs >= 1）
- 徽章命中条件 → **决策表**（多条件组合）
- 主按钮文案 → **决策表**（按状态 × 权益组合）
- 进度条展示 → **边界值**（0% / 50% / 100%）
- 切换收货地址刷新 → **场景法**（端到端操作链路）
- 字段为空兜底 → **边界值**（addressLine=null / receiverName=null）

## 8. BCDE 覆盖检查

每组相关用例完成后，用 BCDE 检查覆盖度：

| 维度 | 检查项 | 示例 |
|------|--------|------|
| **B**order | 是否覆盖了 0/1/空/极大值？ | 0 个订单、1 个订单、999 个订单 |
| **C**orrect | 是否覆盖了正常输入的正确结果？ | 3 个订单正常返回 |
| **D**esign | 是否与验收文档条目逐条对应？ | 验收文档 §3#2 → TC-order-01 |
| **E**rror | 是否覆盖了异常和兜底？ | 参数缺失、接口超时 |

如果任何一个维度缺失，需要补充用例。
