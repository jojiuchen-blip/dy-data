# 通用模板（所有测试类型共享）

## 1. 文件头模板

每个测试用例文件的开头必须包含元信息和文档头。

```markdown
<!-- @meta docId="{docId}" status="active" baseline="v1.0.0" -->
# {文件标题}

> **基线版本**: v1.0.0 | **最后更新**: {日期}
> **编号前缀**: {PREFIX}-
> **基准验收文档**: [{验收文档文件名} §{章节}]({相对路径})
> **设计规范**: [{规范文件名}]({规范相对路径})
> **设计方法**: {使用的方法论列表}

---
```

| 字段 | 说明 | 示例 |
|------|------|------|
| `docId` | 文件唯一标识 | `tc-api`, `tc-order` |
| `PREFIX` | 用例编号前缀 | `TC-API`, `TC-ORDER`, `TC-CMS` |
| `验收文档文件名` | 对应的验收文档（区块子文件） | `acceptance-<slug>/<区块名>.md` |
| `设计方法` | 本文件使用的方法论 | `等价类 + 边界值、决策表` |

---

## 2. 标准用例模板（基础结构）

所有类型用例的基础结构，遵循四段式闭环：Setup → Execute → Verify → Teardown。
各类型有自己的特化模板（见 `type-*.md`），但都基于此基础结构。

```markdown
### {PREFIX}-{NN}：{用例标题}

> 验收文档：`{文档名}` §{章节号}
> 目标：验证{单一验证点描述}
> 执行方式：{curl / 浏览器 UI / 联调验证}

**前提条件**：
- {环境或数据状态要求（人类可读描述）}

**数据准备**：
```sql
-- 构造所需数据状态
INSERT INTO {表名} ({字段列表}) VALUES ({值列表});
```

**步骤**：
1. {操作步骤}

**预期结果**：

| 检查项 | 预期 |
|:---|:---|
| {字段/文案/元素} | {具体值或判定规则} |

**数据清理**：
```sql
DELETE FROM {表名} WHERE {条件};
```
```

## 3. 验收矩阵模板

每个文件末尾必须有验收矩阵：

```markdown
## {N}、验收矩阵

| 场景 | 输入条件 | 预期输出 | 用例编号 |
|------|---------|---------|---------|
| {业务场景1} | {数据状态} | {关键验证结果} | {PREFIX}-01 |
| {业务场景2} | {数据状态} | {关键验证结果} | {PREFIX}-02, 03 |
```

**组织原则**：
- 按**业务场景**组织（不是按技术字段平铺）
- 覆盖正常路径、边界路径、异常路径
- 场景描述要让非技术人员也能理解
