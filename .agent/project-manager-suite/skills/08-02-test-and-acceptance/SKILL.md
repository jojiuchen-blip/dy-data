# Test and Acceptance

本 skill 是 `project-manager-suite` 的验收收口支撑能力。

**定位说明**：本 skill 不是 S6 / S7 阶段的路由入口——S6 测试执行由 test-case-runner 承接（产出测试报告与缺陷清单），S7 完工安全门禁由 security-scan 承接（产出 PASS / BLOCK / WAIVER 结论）；主入口 ai-project-manager 的自动路由不会路由到本 skill。它的位置在两者之间和之后：测试执行完成后、安全扫描前后，由用户显式调用，把已有的测试结果、缺陷清单等材料对照完成标准收口成验收结论。

它负责：

- 对照完成标准验收任务结果
- 识别缺失项和不符合项
- 输出验收结论和补缺建议
- 触发阶段收口和状态沉淀

它不负责：

- 需求澄清和方案设计
- 任务拆解和排期
- 具体编码实现和 bug 修复
- 定义项目长期规则

## 触发条件

本 skill 只由**用户显式调用**触发，不由主入口自动路由。用户提出以下诉求且满足前提时使用：

- 用户明确要求"验收""检查结果""阶段收口"
- 已有可验证的开发结果（如 test-case-runner 产出的测试报告与缺陷清单）
- 需要在测试执行完成后、或安全扫描前后做一次人工点检准备与验收收口
- 当前任务审核主体已明确

## 使用原则

1. 验收必须对照完成标准，不凭印象判断。
2. 区分"已通过""需补充""不通过"三个结论。
3. 补缺建议应具体可执行，不停留在"需要改进"。
4. 涉及自动化测试或测试用例文档时，加载 `coding-standards` 能力。
5. 验收结论必须回写，不只停留在对话中。

## 默认输入

优先读取：

- 当前执行计划中的任务和完成标准
- 本轮开发产出（代码、文档、方案）
- 项目画像中的当前审核主体

必要时再读取：

- 全局规则文件中的交付件底线
- `coding-standards` 能力中的测试和测试用例规范

## 读取的全局文件

- 当前执行计划文件：读取任务、完成标准、前置依赖
- 项目画像文件：读取当前审核主体、当前阶段
- 全局规则文件：读取交付件底线、阶段最小交付件要求
- 项目状态回写文件：读取最近产出，作为验收输入

## 默认工作流

### Step 1：确认验收范围

- 从执行计划中读取待验收任务和完成标准
- 确认审核主体
- 收集本轮开发产出

### Step 2：逐项验收

- 对照完成标准逐项检查
- 涉及测试代码或测试文档时加载 `coding-standards` 规范
- 标注每项的验收结论（通过 / 需补充 / 不通过）

### Step 3：输出验收结论

- 汇总验收结果
- 输出补缺建议（如有）
- 触发状态回写和阶段收口

## 最小输出物

- 验收结论（通过 / 需补充 / 不通过）
- 不符合项清单（如有）
- 补缺建议
- 阶段收口建议

## 回写规则

- 回写到项目状态回写文件：本轮验收结论、发现的问题、补缺建议
- 回写到当前执行计划文件：任务状态变更（已验收 / 需返工）
- 回写到项目画像文件：当前阶段变化（如进入下一阶段）
- 不得回写：长期规则定义（除非验收中发现需长期执行的新规则）

## 输出要求

- 验收结论应具体到每项完成标准
- 补缺建议应可直接执行
- 面向业务团队时，用"做了什么 / 还差什么 / 下一步"的结构

## 资源入口

- 编码与测试规范：`../06-01-coding-standards/SKILL.md`（辅助能力，按需加载）

## V1 边界

本 V1 先解决：

- 对照完成标准的基础验收
- 验收结论和补缺建议输出
- 验收结果回写闭环

本 V1 暂不解决：

- 自动化回归测试编排
- 复杂多角色审批流程
- 用户验收测试（UAT）编排
