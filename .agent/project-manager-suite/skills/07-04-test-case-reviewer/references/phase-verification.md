# TC 独立核查方法论

> **父文件**：[SKILL.md](../SKILL.md) §4 工作流

## 1. 核查定位

test-case-writer 生成 TC 之后，test-case-reviewer 进行一轮独立核查，验证的是 **"TC 与验收文档条目是否语义一致"**——TC 断言的每个字段、值、条件、逻辑，在验收文档条目中是否有明确支撑，且没有曲解或自创。

核心原则：
- **验收文档为唯一 Oracle**：`acceptance-<slug>/<区块名>.md` 的验收条目是对错判定的唯一基准；PRD / foundation / API 契约 / 宿主代码只作上下文或辅助
- **从 TC 出发反查**：先逐条读 TC 断言，回查验收文档条目看是否支撑；再从验收文档条目反查覆盖完整性；不从 PRD 全文主动扫入
- **不审查上游正确性**：默认验收文档 / PRD / foundation 都是对的；核查过程中只对单条 TC 涉及的段落做被动察觉
- **不主动扫 PRD 全文 vs 验收文档**：跨段的文字脱节不是 reviewer 的主动核查项

---

## 2. 核查范围

### 2.1 必查项（每个域都要做）

| 核查维度 | 核查内容 | 发现问题类型 |
|---------|---------|------------|
| **断言准确性** | TC 预期结果中的每个断言是否有验收文档条目支撑 | TC 错误 |
| **字段存在性** | TC 引用的字段名是否在 foundation-api / foundation-schema 中存在 | TC 错误 |
| **逻辑一致性** | TC 描述的业务逻辑（回退、优先级、条件、降级）是否与验收文档条目一致 | TC 错误 |
| **覆盖完整性** | 验收文档条目的每条是否被 TC 覆盖 | TC 遗漏 |

### 2.2 选查项（视复杂度按需）

| 核查维度 | 核查内容 | 适用场景 |
|---------|---------|---------|
| **术语一致性** | TC 使用的术语是否与 foundation-glossary 一致 | 术语集较复杂的项目 |
| **SQL 数据链路** | SQL 构造的数据是否确实能触发验收文档条目描述的场景 | 数据链路复杂的域 |
| **跨域引用** | TC 中引用其他域的断言是否与被引用域的验收文档条目一致 | 多域交叉的场景 |

### 2.3 被动察觉项（仅在核查单条 TC 时触发）

| 核查维度 | 核查内容 | 处理方式 |
|---------|---------|---------|
| **被动察觉事实** | 核查某条 TC 时，TC 引用的 PRD 正文段落和验收文档对同一规则描述不一致 | 仅记录事实到 issues，本轮终态写 `需用户接手`；不裁决 |

被动察觉有三条硬约束（与 SKILL.md §1 一致）：
- **被动触发**：只在核查单条 TC 时顺带触发，不主动扫 PRD 全文 vs 验收文档找脱节
- **只记录事实**：贴 TC 原文 + PRD 段落 + 对应验收条目三份文字，不做对错裁决
- **不改上游**：不改 PRD、不改验收文档、不改 foundation / BRD / delivery-plan

## 3. 逐域核查流程

### 3.1 准备阶段

1. 列出待核查的域清单（按业务域组织）
2. 确认每个域对应的验收文档子文件（`acceptance-<slug>/<区块名>.md`）和 TC 文件（索引层 + 详情层）
3. 确认每个域对应的 SQL 文件（场景数据 `<业务域>/sql/<PREFIX>-<NN>.sql` 与种子数据 `<PREFIX>-SEED.sql`；`<PREFIX>` 为该域编号前缀）

### 3.2 单域核查步骤

**每次只核查一个域**。对每个域按以下步骤执行。

#### Step 1：重读验收文档子文件 + 区块内 §X

重新读取该域对应的验收文档子文件。关键是**刷新上下文**，不复用 writer 生成期的理解——当时可能已形成固化认知。重点关注：区块内 §X 各条目的值 / 条件 / 描述、标注 `[待确认]` 的条目、区块内条目 ID 编号。

若某条 TC 引用了 PRD 正文段落，**仅在这一步顺带读**该段——只读 TC 引用到的位置，不主动扫 PRD 全文。

#### Step 2：逐条比对 TC 断言

逐条读取 TC 的预期结果表格，对每个断言问：

| 核查问题 | 期望答案 | 不通过的处理 |
|---------|---------|------------|
| 这个字段在 foundation-api / foundation-schema 中存在吗？ | 是 | 标记为 TC 错误（引用不存在的字段） |
| 这个值 / 条件在验收文档条目中有明确描述吗？ | 是 | 标记为 TC 错误（自创逻辑） |
| TC 的逻辑表述与验收文档条目语义一致吗？ | 是 | 标记为 TC 错误（曲解条目） |
| TC 引用的验收文档 §X / 条目 ID 正确吗？内容匹配吗？ | 是 | 标记为 TC 错误（条目引用错位） |

常见 TC 错误模式：
1. **引用不存在的字段**：TC 断言了验收文档未声明的响应字段，可能来自宿主实现
2. **自创回退路径**：TC 描述了"先查 A 再回退查 B"的逻辑，但验收文档只定义了单数据源
3. **混淆独立概念**：不同业务概念被混为一谈（如将两类独立数据源的字段视为同一概念）
4. **从 mock 数据推断限制**：从 mock 样本中的元素数量推断"最多 N 个"，但验收文档未声明
5. **用"固定"描述"默认"**：验收文档写"固定 X"，但该字段实际是可编辑配置字段的默认值

#### Step 3：检查覆盖完整性

将验收文档条目每条与 TC 逐行对照：

```
验收条目 #1  →  TC 有覆盖？  →  覆盖方式合理？（API / UI）
验收条目 #2  →  TC 有覆盖？  →  覆盖方式合理？
...
```

未覆盖的条目一律标记为 `TC-GAP-`，issues 里记录需补充用例；如果受前端交互 / 网络异常 / 特殊环境限制，仍保留 `TC-GAP-` 编号，并在原因字段说明限制和建议验证方式。

> 注：旧方法论里"主动核查 PRD 段落之间一致性"的那一步在新架构中**不保留**。PRD 段落之间的一致性不是 reviewer 的主动核查项；只在 §2.3 被动察觉项触发时记录事实。

## 4. 问题记录规范

### 4.1 issues 文件模板

每轮核查产出一份 `tc-reviews/<日期>-issues.md`，同日多轮追加 `-2`、`-3` 后缀。头部固定为：

```markdown
# 待裁定 TC 问题清单

> **来源**：test-case-reviewer 独立核查
> **日期**：YYYY-MM-DD
> **结论**：已完工 / 需 writer 续改 / 需用户接手

## 一、TC 错误（N 项）

### TC-ERR-01：[简述]
**TC 写法**（文件名 第 N 行）：
> [TC 中的具体文字]

**验收文档**（文件名 §X 第 N 行）：
> [验收条目的具体文字]

**问题本质**：[分析两者差异的根本原因，1-3 句]
**处理动作**：[reviewer 自决修复则写"已修"；否则列出裁定选项或转 writer 建议]

## 二、TC 遗漏（N 项）

### TC-GAP-01：[简述]
**未覆盖的验收条目**（文件名 §X 第 N 行）：
> [验收条目的具体文字]

**建议覆盖位置**：[业务域 / 号段建议]
**处理动作**：[可自决补则标"已补"；否则转 writer 续改]

## 三、被动察觉事实记录（N 项，可选）

### OBS-01：[简述]
**TC 片段**（文件名 第 N 行）：
> [TC 引用的 PRD 正文段落]

**对应验收条目**（文件名 §X 第 N 行）：
> [验收条目的具体文字]

**事实**：[两段文字对同一规则的描述差异，具体为...]
**说明**：仅记录事实，不裁决；本轮结论 `需用户接手`

## 四、处理汇总表

| 编号 | 类型 | 域 | 处理结果 | 备注 |
|------|------|------|---------|------|
| TC-ERR-01 | TC 错误 | XXX | 已修 / 需 writer 续改 / 需用户接手 | |
| TC-GAP-01 | TC 遗漏 | XXX | | |
| OBS-01 | 被动察觉事实记录 | XXX | 需用户接手 | |
```

### 4.2 问题编号规则

| 类型 | 前缀 | 编号格式 |
|------|------|---------|
| TC 错误 | TC-ERR- | TC-ERR-01, TC-ERR-02, ... |
| TC 遗漏 | TC-GAP- | TC-GAP-01, TC-GAP-02, ... |
| 被动察觉事实记录 | OBS- | OBS-01, OBS-02, ...（也可无编号，但必须可定位） |

> 注：被动察觉事实记录不是独立的问题分类，只是附录段；不复用旧 generator 的冲突编号规则。

### 4.3 记录要求

1. **必须引用原文**：不能只写"TC 和验收文档不一致"，必须贴出 TC 原文和验收文档原文的具体文字
2. **必须标注行号**：方便定位和复核
3. **必须分析本质**：不是简单罗列差异，要分析差异的根本原因（如"来自宿主实现""来自 mock 数据"）
4. **需用户接手的问题必须提供至少 2 个裁定选项**：让用户能直接选，不需要追加讨论

## 5. 修复流程（三路分发）

### 5.1 可自决的 TC 错误

reviewer 直接修改 TC 文件，修改范围按需：TC 详情文件修正断言 / 标题 / 验收条目回链；TC 索引文件同步更新测试范围表中的"测试重点"描述；TC 验收矩阵同步更新覆盖状态 / 说明；SQL 文件如断言变化导致数据准备需要调整；TC 版本历史追加变更记录。

issues 对应项的"处理动作"字段写"已修"作审计线索。本轮若只剩"已修"项，结论写 `已完工`。

### 5.2 需 writer 续改的问题

超出 reviewer 原地修正能力（SQL 需重构、验收矩阵重算、跨域用例重组等）。在 issues 里写完整问题 + 影响范围 + 建议修改方向，结论写 `需 writer 续改`。

### 5.3 需用户接手的问题

包括被动察觉事实记录、无法给唯一修正方案的疑问。在 issues 里贴事实 + 裁定选项，结论写 `需用户接手`。reviewer 不改 PRD、不改验收文档、不改 foundation / BRD / delivery-plan、不改 PRD 元信息或版本文件。

### 5.4 TC 遗漏补充

可自决的补充（如已有 SQL 可直接用）：在对应域文件合适号段新增用例；编写 / 复用 SQL 数据准备文件；更新索引层测试范围表；更新验收矩阵；更新 `tc-main-<slug>.md` 用例统计。超出 reviewer 能力（跨域重构、SQL 重写等）则转 writer 续改，结论写 `需 writer 续改`。

## 6. 核查产出物

| 产出物 | 说明 |
|--------|------|
| 待裁定 TC 问题清单 | `tc-reviews/<日期>-issues.md`，所有发现的问题 + 处理动作 + 本轮终态结论 |
| 修正后的 TC 文件 | reviewer 自决修复的 `<业务域>/tc-<业务域>.md` 和 `<业务域>/sql/*.sql` |
| 更新后的 TC 主索引 | 仅在 reviewer 原地修 TC 导致统计变化时更新 `tc-main-<slug>.md` 的用例统计 / 覆盖率 |

## 7. 实践经验

### 7.1 高频问题类型

1. **被动察觉事实记录，不做主动核查**：TC 引用的 PRD 正文段落和验收文档对同一规则的描述有差异时，仅记录事实；不裁决对错、不改上游；不主动扫 PRD 全文找脱节
2. **TC 引用宿主实现或未在 foundation-api 暴露的字段**：生成 TC 时如果做了数据链路研究，容易不自觉地把宿主字段写入断言，但验收文档可能未声明；核查时统一按验收文档 + foundation-api 判定
3. **概念混淆**：不同业务概念被混为一谈（如将两类独立数据源的字段视为同一概念）
4. **验收文档固定值与 foundation-schema 可编辑配置字段之间的疑点**：验收文档写"固定 X"，但该字段在 foundation-schema 中是可编辑配置；此时按 §2.3 被动察觉记录事实或按验收文档口径修 TC

### 7.2 核查效率建议

- **批量发现，逐条处理**：先完成所有域的核查，生成完整的待裁定问题清单，再分类处理；避免核查一个域就暂停等裁定
- **分类处理**：reviewer 可自决的 TC 错误直接改；需 writer 续改的问题写清楚影响范围；需用户接手的问题提交用户裁定
- **关联问题合并**：如果多个问题来自同一根因（如同一处字段引用错位扩散到多条 TC），应合并处理，issues 里一次性列出所有受影响的 TC
