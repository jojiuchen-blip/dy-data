# Reviewer 核查自检清单

> 本文件供 test-case-reviewer 做 TC 独立核查时自检使用。
> 验收文档条目是唯一 Oracle；PRD / foundation / API 契约 / 宿主代码只作上下文或辅助。
> reviewer 只记录 TC 错误、TC 遗漏，或被动察觉事实；不审查上游正确性。

本清单与 `references/phase-verification.md` 的方法论配合使用——方法论规定"怎么做核查"，本清单规定"每个域核查完后该过哪些自检问题"。按独立核查的阶段分四组：逐域比对硬项（§1）、问题记录硬项（§2）、裁定与修复闭环（§3）、常见核查期陷阱速查表（§4）。任何一条自检不通过就不要进入下一阶段，先回到对应域补齐再往下走。

使用时机：test-case-writer 产出 `tc-main-<slug>.md` 和业务域 TC 文件、reviewer 开始独立核查前读一遍；每轮核查收尾写 issues 前再过一遍 §2 / §3。清单只约束 reviewer 自己的动作，不替上游（验收文档 / PRD / foundation）做对错裁决。

## 1. 逐域比对

每个业务域独立核查时，都要逐项通过以下 6 条自检：

- [ ] 重新读取了该域的验收文档子文件，刷新上下文，不复用生成期记忆
- [ ] TC 预期结果中的每个断言都有验收文档条目支撑（能准确指向某条验收条目的某个值 / 条件 / 描述）
- [ ] TC 引用的字段名在 foundation-api / foundation-schema 中存在（不把宿主实现的内部字段写入 TC 预期）
- [ ] TC 描述的业务逻辑（回退、优先级、条件、降级）与验收文档条目一致（不自创验收文档未定义的执行路径）
- [ ] TC 引用的验收文档 §X / 条目 ID 正确且内容匹配（不保留已改动的旧引用）
- [ ] 验收文档条目的每条至少被一条 TC 覆盖（无覆盖的条目记为 TC 遗漏）

逐域比对时每次只处理一个业务域，不跨域跳读——跨域跳读容易把不同域的验收条目混淆。比对过程中发现的 TC 错误 / 遗漏先记下来，不立刻动手改；所有修正在 §3 "裁定与修复闭环" 阶段统一处理。

## 2. 问题记录

核查中发现的每个问题提交到 issues 之前，都要过以下 5 条自检：

- [ ] 每个发现的问题已分类为 `TC 错误` / `TC 遗漏`，或放入 `被动察觉事实记录`（不自创第四类）
- [ ] 每个问题引用了 TC 和验收文档的原文，不只是摘要（贴真实文字，不自己改写）
- [ ] 每个问题标注了文件名和行号（方便定位和复核）
- [ ] 每个问题分析了差异根本原因（不是简单罗列差异，要判断是 TC 自创、引用错位、字段不存在等）
- [ ] 每个需要用户接手的问题提供了至少 2 个裁定选项（让用户能直接选，不需要追加讨论）

`被动察觉事实记录` 的触发条件严格限定为：核查单条 TC 时，TC 引用的 PRD 正文段落和验收文档对同一规则的描述不一致。不从 PRD 全文主动扫入，不替上游裁决，仅把"TC 引用的 PRD 段落 ↔ 对应验收条目"的文字差异作为事实记录下来。

## 3. 裁定与修复闭环

本轮核查结束前，过以下 5 条自检以确保 issues 文件可以交付：

- [ ] issues 文件头部结论字段已写清楚本轮核查终态
- [ ] 结论字段只能是 `已完工` / `需 writer 续改` / `需用户接手`，与 `test-case-chief/SKILL.md` 的判定表对齐，不自造同义词
- [ ] 所有 reviewer 可自决的 TC 错误已原地修复（issues 里记"已修"作审计）
- [ ] 所有需用户接手的问题已提交用户裁定（附事实记录和裁定选项）
- [ ] TC 修正后版本历史已更新（如因本轮 reviewer 修正导致统计变化，`tc-main-<slug>.md` 的用例统计 / 覆盖率也已同步）

issues 文件按 `tc-reviews/<日期>-issues.md` 命名；同日多轮从第 2 轮起加 `-2` / `-3` 数字后缀，不覆盖旧 issues。新轮次直接新建文件，历史 issues 保留作审计；chief 侧按"日期最大、同日内数字后缀最大者为最新"读取（无后缀视为第 1 轮，不按整个文件名的字典序取末尾）。

## 4. 常见核查期陷阱速查表

以下 4 条是独立核查时最容易踩到的模式，编号 R1-R4；逐域比对时比照一遍，命中就按"修复"列处理，不自造更细分的编号。
R1-R4 编号独立于 writer 侧 `references/self-check.md` 的生成期陷阱编号，不要混用。

| 编号 | 陷阱 | 症状 | 修复 |
|------|------|------|------|
| R1 | TC 断言引用验收文档未定义的字段 | TC 预期中出现 `<未在 foundation-api 中定义的字段>`，可能来自宿主实现但验收文档未声明 | 核查 TC 断言与 foundation-api / foundation-schema；删除或标注无源断言 |
| R2 | TC 自创回退 / 降级路径 | TC 描述"先查 A 再回退查 B"，但验收文档只定义了单数据源 | 核查 TC 逻辑描述是否有验收文档条目支撑，无支撑则修正 TC |
| R3 | 被动察觉 TC 引用的 PRD 正文段落与验收文档对同一规则描述不一致 | 核查单条 TC 时，TC 引用的 PRD 正文段落和验收文档对同一规则描述不一致 | 仅记录事实到 issues，本轮终态写 `需用户接手`；不裁决对错、不改 PRD、不改验收文档；不主动扫 PRD 全文 vs 验收文档找脱节 |
| R4 | 混淆"默认值"与"固定值" | 验收文档写"固定颜色"等固定值，但该字段实际来自 foundation-schema 里的可编辑配置表 | 标 `需用户接手` 或按验收文档明确口径修 TC；不直接裁决 |

R1-R4 合起来对应两种问题形态：R1 / R2 属于 `TC 错误`，由 reviewer 自决或转 writer 续改；R3 / R4 通常归入 `被动察觉事实记录`，本轮终态写 `需用户接手`。不要把 R3 / R4 当作独立的问题分类——reviewer 的问题分类始终只有 `TC 错误` 和 `TC 遗漏` 两类，加上一类 `被动察觉事实记录` 作为附录。
