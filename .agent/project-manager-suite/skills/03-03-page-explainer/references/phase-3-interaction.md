# Phase 3: 交互描述

> 本文件在进入 Phase 3 时由 SKILL.md 指令加载。

## 目标

逐页产出结构化交互语义，同步执行方法论四条线识别交互盲区。

## 执行步骤

对 page-delivery 中列出的每个页面：

1. **打开运行页面**：优先使用 page-delivery「本地预览」段中的启动命令与访问地址；旧交付清单缺该段时按工程目录启动页面。优先用 Playwright / Browser 打开真实运行页面并完成截图、可见元素检查和必要点击。若 Playwright / Browser 不可用，必须先标记 BLOCKED 并说明缺少运行态浏览器验证能力；不得用源码阅读、curl、构建日志、静态 DOM 或静态截图替代。只有用户明确批准降级时，才可使用可控制的 Chrome 可见窗口或等价方案；替代方案必须能打开真实页面、截图取证、检查可见元素、执行必要点击，否则不能产出 locked 语义。
2. **读取页面代码**：从 delivery 中的文件路径读取页面代码文件（技术栈以 page-delivery 交付清单声明为准，默认见 tech-stack.md），用来佐证点击行为、状态联动、禁用条件和 mock 数据流；不能只读代码判断页面布局。
3. **识别原型 / mock 边界**：判断样例文件、预置列表、固定数据是否只是 page-designer 阶段的 mock 表达。mock 展示态不得直接判定为真实系统功能缺失。
4. **分模块描述页面**：基于运行页面，用大白话介绍这个页面由哪些模块组成，每个模块长什么样，页面目的与覆盖角色。
5. **逐模块产出语义条目**：对每个模块中的每个可交互元素，填写完整的结构化字段（id/actor/trigger/system_behavior/fallback/status 等），并在描述中说明证据来自运行页面还是代码佐证。
6. **逐元素解释检查**：是否有说不清的元素？先判断是否属于 mock 原型态；仍说不清的产出 `clarification` 类型差异，主动问用户。
7. **正向追踪检查**：从流程文件中该页面涉及的流程出发，每一步是否能落在运行页面的具体可见模块、可交互元素或明确原型占位态上？核心业务路径完全无法表达时才产出 `design_gap`。
8. **CRUD 完整性检查**：仅对管理实体的页面执行（如列表页/表单页/实体详情编辑页）。操作流程页、审核页、修复页、dashboard、只读详情页、审批页、数据报表页、原型演示页跳过本检查，不因缺少 CRUD 入口产出 gap。
9. **业务态覆盖检查**：每个语义条目的 `fallback` 字段是否覆盖可能业务态。若运行页面未展示某个业务态，但可以在语义中合理补充，先写入语义；只有 BRD 或用户确认要求当前页面显式呈现该业务态时，才产出 `design_gap`。

## 差异判定边界

仅当差异来自运行页面中的业务交互问题时，才产出 `explainer-b-gap-<slug>.md`。

| 不应判为 gap | 正确处理 |
|--------------|----------|
| 原型预置一个样例文件 | 描述为“文件上传/选择后的展示态” |
| mock 检测列表是固定数据 | 描述为“检测报告生成后的结果态” |
| 下载、上传、检测服务尚未接后端 | 作为后续 foundation / PRD 细化输入，不在页面阶段判缺失 |
| 代码存在但运行页面不可见的结构 | 以运行页面为准，不写入用户可见交互语义 |

| 应判为 gap | 分类 |
|-----------|------|
| 已确认主流程在运行页面中完全无表达 | `design_gap` |
| 页面允许“未确认检测结果就修复” | `logic_conflict` |
| 页面表现为“覆盖原文件”而非“生成副本” | `logic_conflict` |
| 用户确认某业务态必须当前页面显式呈现，但页面没有 | `design_gap` |

模板：读取 `templates/interaction.md`

## 差异文件结构

文件名：`explainer-b-gap-<slug>.md`

仅在发现差异时产出。模板：读取 `templates/gap.md`

## 冻结流程

1. 所有语义条目产出后，初始 status 为 `open`
2. 向用户展示，逐条或批量确认
3. 用户确认后标记为 `locked`
4. 全部 locked 后 Phase 完成
