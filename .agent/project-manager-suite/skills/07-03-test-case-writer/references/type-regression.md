# 缺陷回归测试 — 生成策略 + 模板

> 回归用例的唯一目的是"历史 bug 不复发"。本文件只覆盖回归型 TC 的生成模板与策略。
> 核查类职责（TC 是否与验收文档对齐、TC 与 PRD 冲突等）不属于 writer，归 test-case-reviewer。

## 1. 适用条件

**只有输入资料里已有缺陷记录时才生成回归用例**。来源包括：
- 宿主项目缺陷单 / bug tracker 导出
- 已修复 PR 的描述里明确写出的缺陷场景
- 用户或产品口述的"上一轮出过 X 问题"——需要形成书面缺陷记录后才生成

没有缺陷记录则**不生成**回归用例文件；不要凭空造回归。

## 2. 生成策略

- 每条已知缺陷至少一条回归用例
- 用例重现原始缺陷触发场景，验证修复后的正确行为
- 标注缺陷编号便于追踪
- 使用错误推测法补充相关边界用例
- 数据准备必须能构造原缺陷触发条件的最小充分集
- 回归用例**仍需绑定验收文档条目**——用来锚定"修复后的正确行为"这一预期

## 3. 回归用例模板

```markdown
### {PREFIX}-{NN}：{缺陷标题}-回归

> 关联缺陷：{缺陷编号}（{缺陷简述}）
> 验收文档：`acceptance-<slug>/<区块名>.md` §<X> 条目 <ID>
> 目标：验证 {缺陷修复后的正确行为}
> 执行方式：{按原缺陷场景——API / UI / 管理台}

**前提条件**：
- {重现原缺陷的环境条件（人类可读描述）}

**数据准备**：
> 执行 [sql/{PREFIX}-{NN}.sql](sql/{PREFIX}-{NN}.sql) 中的"数据准备"段
> 构造原缺陷触发条件的源表数据

**步骤**：
1. {重现原缺陷的操作步骤}
2. {检查修复后行为}

**预期结果**：

| 检查项 | 预期（修复后，源自验收文档条目） |
|:---|:---|
| {原缺陷表现字段} | {正确行为} |

**数据清理**：
> 执行 [sql/{PREFIX}-{NN}.sql](sql/{PREFIX}-{NN}.sql) 中的"数据清理"段
```

## 4. 无验收条目时的标记规则

并非所有历史缺陷都能在验收文档里找到对应条目（例如只影响一次性场景的缺陷、验收文档未覆盖的灰色地带）。这类用例按以下规则生成：

- 在用例头部用 `[待确认-缺陷无验收条目]` 标记
- **预期结果整列标 `[待确认]`**——不得用代码行为反推预期；代码只能用于构造"原缺陷触发条件"的数据准备，不参与预期 oracle
- 步骤和数据准备照常写（能构造、能执行），只是预期栏留空并挂 `[待确认-需补验收条目]`
- 在 `tc-main-<slug>.md` 的"[待确认] 汇总"里登记，等产品/用户补齐验收条目后再回填预期

`[待确认-缺陷无验收条目]` 是 writer 的标记动作，不是推动回环；核查由 reviewer 走 test-case-reviewer 的流程。
