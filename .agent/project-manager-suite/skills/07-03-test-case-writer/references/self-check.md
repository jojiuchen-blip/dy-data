# Writer 自检清单

> 本文件供 test-case-writer 在生成 TC 时自检使用。
> 验收文档条目是唯一测试目标；上游规格、foundation、API 契约、宿主代码只作上下文或数据链路辅助。
> 本文件只包含生成期自检内容；独立核查侧的自检清单见 test-case-reviewer 的 `references/verification-check.md`。

## 1. 单条用例自检

每条用例提交前必须通过以下检查。

### 1.1 验收文档条目回链

- [ ] 用例标注了验收文档条目（`验收文档：acceptance-<slug>/<区块名>.md §X 条目 <ID>`）
- [ ] 验收文档条目 ID 确实存在且内容匹配
- [ ] 如果验收文档条目修订过，用例已同步更新

**反例**：`验收文档：略` 或 `验收文档：acceptance-<slug>.md`（没有条目 ID）。

### 1.2 可复现前提

- [ ] 前提条件足够明确，非作者也能照着复现
- [ ] 测试数据指向明确实体（如用户、订单、配置项）
- [ ] 如果依赖特定数据库状态，写明了数据来源

**反例**：`前提条件：准备一个用户` —— 什么状态的用户？什么数据特征？

### 1.3 数据准备可执行

- [ ] 包含 `数据准备` 段，且内容是可直接运行的 SQL 或 API 调用
- [ ] 数据准备的结果能满足前提条件中描述的数据状态
- [ ] 包含 `数据清理` 段，确保测试后环境可还原
- [ ] 数据清理覆盖了数据准备中插入或修改的所有记录
- [ ] 没有假设“数据已存在”而不说明怎么造

**反例**：`前提条件：数据库中存在 <用户标识> 的 3 条记录` —— 怎么造这 3 条记录？用什么 SQL？

### 1.4 计算字段禁写

- [ ] 已先完成本域字段分层：`输入源字段` vs `服务端计算字段`
- [ ] SQL 仅写入输入源字段，未写入任何服务端计算字段
- [ ] 没有通过预置结果字段来“伪造正确输出”
- [ ] 若该域存在计算快照字段或缓存字段，已确认其在本接口链路中不是判定输入

**反例**：接口 `<结果字段>` 由源表和配置实时计算，却在 SQL 里写入 `<服务端计算快照表>.<状态字段>='<某状态>'` 作为前提。

### 1.5 可判定预期

- [ ] 预期结果可直接判定 PASS/FAIL
- [ ] 具体到字段值、颜色代码、数量、布尔值、或可追溯的数据引用（如 SEED SQL 行引用）中的至少一类
- [ ] 没有使用模糊词汇

禁止词汇清单：
- “看起来正常”
- “基本符合”
- “页面正常展示”
- “数据正确”
- “应该有值”
- “符合预期”
- “A 或 B”式模糊预期（如“200 或 400”“null 或空字符串”）；查 API 契约确认唯一预期，无法确认则标注 `[待确认-需产品明确]`

### 1.6 BCDE 覆盖

- [ ] 同组用例覆盖了 Border（边界）
- [ ] 同组用例覆盖了 Correct（正常）
- [ ] 同组用例覆盖了 Design（验收文档条目对应）
- [ ] 同组用例覆盖了 Error（异常/兜底）

### 1.7 单一验证点

- [ ] 一条用例只验证一个逻辑点
- [ ] 没有多个逻辑点混在一条用例的问题
- [ ] 如果验收文档条目影响多状态/多模块，已拆成多条用例

### 1.8 目标态原则

- [ ] 用例验证的是验收文档条目的目标态描述
- [ ] 没有测试变更差异（如“旧字段已删除”“新增字段 X”）
- [ ] 没有引用上游文档的“现状差异”或“变更说明”章节作为预期

## 2. 文件级自检

每个测试文件完成后检查。

### 2.1 结构完整性

- [ ] 文件头包含 `@meta` 标签（docId/status）
- [ ] 文件头包含编号前缀、验收文档引用、设计方法
- [ ] 有测试范围表
- [ ] 有验收矩阵
- [ ] 有版本历史

### 2.2 编号正确性

- [ ] 编号前缀与 TC 主索引（`tc-main-<slug>.md`）登记的一致
- [ ] 编号按号段组织：01-09 API、11-19 UI
- [ ] 各号段内连续无跳号
- [ ] 编号格式统一（两位数字：01、02）

### 2.3 分组组织

- [ ] 用例按功能模块或接口分组
- [ ] 不同类型的用例没有混写
- [ ] 每组有清晰的标题

### 2.4 分工边界

- [ ] 域文件内 API 验证用例（01-09）只验 JSON 字段值，不验 UI 渲染
- [ ] 域文件内 UI 验证用例（11-19）只验渲染和交互，不验 API 数据逻辑
- [ ] 同一条用例没有混合两种验证角度

### 2.5 验收文档覆盖率

- [ ] 验收文档每条条目都有对应用例
- [ ] 验收文档中异常兜底类条目都有对应用例
- [ ] 没有遗漏的验收文档条目

### 2.6 验收矩阵

- [ ] 矩阵覆盖全部验收文档条目（每条条目至少一行）
- [ ] 矩阵按业务场景组织，不按字段平铺
- [ ] 覆盖正常、边界、异常三条路径
- [ ] 每行的用例编号可追溯

### 2.7 数据声明与数据准备

- [ ] 每条用例的前提条件足够具体
- [ ] 每条用例包含可执行的 `数据准备` 段（SQL 或 API 调用）
- [ ] 每条用例包含 `数据清理` 段
- [ ] 使用样本标识（如“用户 A”）时，样本在数据字典中有定义
- [ ] 没有硬编码不可变更的具体 ID

### 2.8 方法论匹配

- [ ] 设计方法与场景匹配，不是全部用同一种方法
- [ ] 文件头标注的设计方法与实际使用一致

## 3. 体系级自检

全部文件完成后做整体检查。

### 3.1 跨文件一致性

- [ ] 不同域文件之间没有重复测试同一逻辑
- [ ] 同一业务域的 API 验证、UI 验证在同一个域文件中，不分散到多个文件
- [ ] 编号前缀全局唯一，没有两个文件用同一前缀

### 3.2 TC 主索引（`tc-main-<slug>.md`）更新

- [ ] `tc-main-<slug>.md` 索引表中的状态已更新
- [ ] 用例统计表已填写
- [ ] 验收矩阵汇总已更新

### 3.3 测试数据完整性

- [ ] 数据样本字典覆盖了所有用例的数据需求
- [ ] 每个样本标识在至少一条用例中被引用
- [ ] 没有用例引用了不存在的样本

### 3.4 缺陷映射

- [ ] 每个已知缺陷有对应的回归用例
- [ ] 缺陷映射表已更新
- [ ] 回归用例编号可在文档中找到

### 3.5 覆盖率统计

- [ ] 全部验收文档条目的覆盖率已计算
- [ ] 未覆盖的验收文档条目已标注原因
- [ ] 总体覆盖率达到目标（通常 >= 90%）

## 4. 常见生成期陷阱速查表

| # | 陷阱 | 症状 | 修复 |
|---|------|------|------|
| 1 | 测变更差异而非目标态 | 用例标题含“已删除”“新增”；验证旧版字段被删除 | 改为验证目标态：响应只包含验收文档规定的字段 |
| 2 | 自上而下预设测试数据 | 先设计数据字典再写用例，覆盖率严重不足 | 先写用例再归堆数据 |
| 3 | API 验证和 UI 验证混在一条用例 | 一条用例里既验 JSON 字段值又验 UI 渲染 | 拆为两条：一条 curl 验 JSON，一条浏览器验渲染 |
| 4 | 模糊预期 | 预期含“正常”“基本”“页面正常展示” | 改为可判定值。验收文档固定值直接写；管理台可配置值引用数据准备 |
| 5 | 从旧文件搬运 | 用例结构和旧文件一模一样，没基于验收文档重新设计 | 旧文件结构和覆盖范围可能已过时，必须从验收文档条目出发重写 |
| 6 | 编号混乱 | 两个文件都用同一编号前缀，跨文件引用无法定位 | 每个文件必须有独立前缀 |
| 7 | 数据准备缺失 | 前提写“数据库中存在 X”但无 SQL | 补充可执行的 INSERT / UPDATE 和清理语句 |
| 8 | 往缓存或预计算表塞数据 | 直接构造 `<缓存快照表>.<字段>='<某状态>'` 然后验证 API 返回该状态 | API 可能不读缓存表。必须查宿主实现确认数据链路，往源表构造数据让 API 自己算 |
| 9 | 多逻辑混写 | 一条用例有 5 个以上检查项且覆盖不同规则 | 拆分为多条，每条只验一个逻辑点 |
| 10 | 前提模糊 | “准备一个对象”但没有说明状态和数据特征 | 指定具体数据特征（用户、订单、配置项、数值） |
| 11 | 缺验收文档条目回链 | 用例没有标注验收文档条目 | 补充 `验收文档：acceptance-<slug>/<区块名>.md §X 条目 <ID>` |
| 12 | 缺异常覆盖 | 全部用例都是正常路径 | 补充 Error / Border 用例 |
| 13 | 数据清理缺失 | 测试后遗留脏数据影响后续用例 | 补充 DELETE、还原 UPDATE 或管理台还原步骤 |
| 14 | 用代码行为定义预期结果 | 按代码逻辑写 expected（如 `<业务阈值类型> >= <数值>`），但代码本身有 bug | 预期行为**必须从验收文档条目获取**。数据链路文档只解决“往哪写数据”，不定义预期 |
| 16 | 枚举值杜撰 | SQL 中插入了不存在的枚举值 | 写 SQL 前查 Schema 确认枚举字段的全部合法值，验收文档引用的枚举值也必须与 Schema 一致 |
| 17 | 单文件过长 | 用例详情超过 200 行，编辑卡顿、上下文过大 | 拆为两层结构：索引层 + 详情层，双向引用 |
| 18 | 一次性生成所有域 | 试图一次性生成多个域的用例，导致上下文过大 | 每次只处理一个域，完成并经用户确认后再处理下一个 |
| 19 | “A 或 B”式模糊预期 | 验收文档没明确定义参数缺失时返回 200 还是 400，就写“200 或 400” | 查 API 契约或宿主实现确认唯一预期；仍不确定则标注 `[待确认-需产品明确]` |
| 20 | 硬编码配置项运行值 | 把 SEED SQL 中的展示文案（如 `<可配置文案字段>='<某具体文案>'`）直接写进预期结果 | 配置字段的运行时值由配置项决定，不是验收文档固定值。预期应引用数据准备 |
| 21 | SEED 隐式依赖执行顺序 | SEED-A 对共用配置表只有 UPDATE，假设 SEED-B 已先跑过并插入了该行 | SEED 必须自包含：对共用配置表用 INSERT ... ON DUPLICATE KEY UPDATE |
| 22 | 全局破坏性测试操作 | 用 DELETE FROM 配置表模拟“无数据”，清理依赖人工重跑 SEED | 用等效非破坏操作，如 UPDATE isActive=0；清理只需定向 UPDATE 还原 |
| 23 | 散装数据复用 | “复用 SQL-A + 额外在步骤中插入 B”，执行步骤散落、容易遗忘清理 | 当多条用例需同一对象的多种状态叠加时，提取为独立 SQL 文件并被这些用例共用 |
| 24 | 预置服务端计算结果字段 | 测试 SQL 写入 `<服务端计算快照表>.<状态字段>` / `<缓存快照表>.<字段>` 等结果字段，并据此断言接口输出 | 先做字段分层，只构造驱动计算的输入源数据，禁止预置结果字段 |
| 28 | 从 mock 数据推断业务限制 | mock 数据有 5 个节点，就在 TC 中写“最多 5 个节点” | 区分 mock 数据样本与业务规则。验收文档未声明的限制不应出现在 TC 断言中 |
