# 业务域测试 — 生成策略 + 模板

> 每个业务域对应一个测试文件夹，域内按验证方式分两组（API / UI）。
> 本文件只覆盖业务域型 TC 的生成模板与策略；缺陷回归见 `type-regression.md`。

## 1. 核心思想

- **域即最小生成单位**：一个业务域（验收文档里聚合出的一个功能/规则簇）对应一个测试文件夹，内含一份 TC markdown 和 `sql/`
- **每次只处理一个业务域**：完成并经用户确认后再处理下一个；禁止一次性生成所有域
- **域内按验证方式分两组**：
  - **API 验证组**（编号 01-09）：curl 调接口 → 验 JSON 字段（测业务计算逻辑）
  - **UI 验证组**（编号 11-19）：浏览器打开页面 → 验渲染和交互（测前端展示）
- **两组共用同一套 SQL 数据准备**
- 号段之间留空位供后续新增，号段一旦分配不重排

> v2.0.0 删除旧版"配置闭环组"（号段 21-29）——配置变更通过 UI 验证组覆盖即可。

## 2. 数据链路研究（SQL 构造前的必做前置步骤）

数据链路研究**只为 SQL 构造服务**；预期结果不从数据链路研究得出。

| 研究项 | 用途 |
|--------|------|
| Controller → Service 调用链 | 定位实际读取数据的源表 |
| 读取的全部表列表 | 区分源表与缓存/预计算表；**只往源表构造数据** |
| 关键计算函数与逻辑 | 确认哪些字段是服务端计算出来的，登记为"禁写字段" |
| 降级 / 默认值逻辑 | 构造异常兜底用例时能命中对应分支 |
| 本域**服务端计算字段**清单 | SQL 中禁止写入这些字段 |

**Oracle 优先级**（预期值从哪来）：
1. **验收文档条目**——权威，唯一预期源；PRD / foundation / API 契约 / 代码只做上下文或数据链路辅助
2. foundation-api / foundation-schema 或 API 契约文档（仅辅助：字段名、类型、状态码这类技术事实）
3. 宿主代码实现（仅用于"从哪里读数据、如何构造输入"，不用于"应该返回什么"）
4. 仍无法确定 → 在用例预期列标注 `[待确认-需产品明确]`

**计算字段禁写门（强制）**：每个域生成 SQL 前先列"输入源字段"与"服务端计算字段"两类清单；SQL 只能写输入源字段。即便数据库中存在某个字段，若它是服务端实时计算的结果，就禁止作为测试输入构造。

## 3. 文件内部结构

### 3.1 单层结构（用例详情 ≤ 200 行）

```markdown
<!-- @meta docId="tc-<业务域>" status="active" baseline="v1.0.0" -->
# <业务域>

> **编号前缀**: <PREFIX>-
> **基准验收**: acceptance-<slug>/<区块名>.md §<X>
> **数据链路**: <源表列表 / dc-*.md 引用>
> **预期来源**: 验收文档条目 ID <ID 范围>
> **设计方法**: <方法论，如 决策表 + 边界值>

## 一、测试范围

| 用例编号 | 测试重点 | 验证方式 | 验收文档条目 ID |
|---------|---------|---------|----------------|
| <PREFIX>-01 | <场景> | API | <ID> |
| <PREFIX>-11 | <场景> | UI | <ID> |

## 二、前置条件（全部用例共用）

**种子数据**：执行 [sql/<PREFIX>-SEED.sql](sql/<PREFIX>-SEED.sql) 初始化配置表。

## 三、API 验证用例
## 四、UI 验证用例
## 五、验收矩阵
## 六、版本历史
```

### 3.2 两层结构（用例详情 > 200 行）

超过约 200 行时拆为索引层 + 详情层两个文件，双向引用：

| 层 | 文件名 | 内容 |
|----|-------|------|
| 索引层 | `tc-<业务域>.md` | 元信息、测试范围表、前置条件、验收矩阵、版本历史、指向详情层 |
| 详情层 | `tc-<业务域>-用例详情.md` | 全部用例步骤 + 数据准备 + 预期结果，按 API / UI 分节 |

双向回链：
- 索引层头部：`> **用例详情**: [tc-<业务域>-用例详情.md](tc-<业务域>-用例详情.md)`
- 详情层头部：`> **索引文件**: [tc-<业务域>.md](tc-<业务域>.md)`

## 4. API 验证用例模板

```markdown
### <PREFIX>-<NN>：<用例标题>

> 验收文档：`acceptance-<slug>/<区块名>.md` §<X> 条目 <ID>
> 目标：验证 <业务条件下 API 的返回字段值>
> 执行方式：curl

**数据准备**：
> 执行 [sql/<PREFIX>-<NN>.sql](sql/<PREFIX>-<NN>.sql) 中的"数据准备"段

**步骤**：
1. `curl "<完整 URL>"`
2. 解析响应 JSON

**预期结果**：

| 检查项 | 预期 |
|:---|:---|
| HTTP 状态码 | <200 / 400 / ...> |
| code | <code 字段预期> |
| data.<字段> | <具体值 / 类型约束 / 引用 SEED 中某行某列> |

**数据清理**：
> 执行 [sql/<PREFIX>-<NN>.sql](sql/<PREFIX>-<NN>.sql) 中的"数据清理"段
```

**特殊情况**：
- 参数校验类用例通常不需要数据准备
- 空结果类用例的数据准备是"确保目标数据不存在"，用等效非破坏操作（如 `UPDATE isActive=0`）

## 6. 验收矩阵模板

验收矩阵列出本域所有相关的验收文档条目，映射到 API / UI 用例。

```markdown
## 验收矩阵

| 验收文档条目 ID | 验收场景 | API 用例 | UI 用例 | 覆盖状态 |
|----------------|---------|---------|---------|---------|
| <ID-1> | <场景描述> | <PREFIX>-01 | <PREFIX>-11 | ✅ |
| <ID-2> | <场景描述> | <PREFIX>-02 | — | ✅ |
| <ID-3> | <场景描述> | — | — | ⚠️ 无唯一预期，`[待确认]` |
```

覆盖状态含义：
- ✅ 至少一条用例能验证该条目
- ⚠️ 无唯一预期或无法构造数据，已标 `[待确认]` 并在 `tc-main-<slug>.md` 汇总
