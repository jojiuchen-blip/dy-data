# 宿主项目测试资产落地约定参考

> 本文件是 test-case-writer 在新宿主项目**首次搭建** `docs/test-case/` 目录时使用的通用参考。
> 它不是 pipeline 依赖文件，不是宿主项目最终规则来源——宿主项目一旦有了自己的 `docs/test-case/README.md` 或 `tc-main-<slug>.md`，以宿主文件为准。
> 文件内使用的目录名、域名、前缀都是占位符，不绑定任何具体业务。

## 1. 职责与使用方式

本文件回答一个问题："新宿主项目第一次产测试资产时，目录怎么摆、文件怎么命名、编号怎么定、SQL 怎么放？"

使用时机：
- Phase 2 规划业务域和文件结构时读取
- 宿主项目 `docs/test-case/` 为空，或只有零散文件无统一约定时

优先级规则（冲突时取前者）：
1. 宿主项目 `docs/test-case/README.md` 已登记的结构/命名/编号规则
2. 宿主项目 `tc-main-<slug>.md` 已注册的业务域目录与编号前缀
3. 本文件的通用模板

前两者皆无时，writer 按本文件模板创建首版约定，并在 `tc-main-<slug>.md` 里登记。

## 2. 推荐目录结构

```
<host>/docs/test-case/
├── tc-main-<slug>.md                    ← 全局 TC 主索引
├── README.md                            ← 本宿主项目落地约定（可选）
├── acceptance-<slug>.md                 ← 验收文档主索引（prd-acceptance-reviewer 产出）
├── acceptance-<slug>/                   ← 验收文档区块子目录
│   └── <区块名>.md
│
└── <业务域>/                             ← 业务域（一个域一个子目录）
    ├── tc-<业务域>.md                   ← 索引层（或单文件）
    ├── tc-<业务域>-用例详情.md          ← 详情层（超约 200 行时拆出）
    └── sql/
        ├── <PREFIX>-SEED.sql            ← 种子/配置数据，多用例共用
        ├── <PREFIX>-01.sql              ← 场景数据，与用例编号一一对应
        └── ...
```

组织原则：
- 每个业务域一个子目录；业务域以**验收文档里的区块/功能语义**为聚合单位，不按 PRD 文件机械切分
- 域内 markdown 与 `sql/` 同级；SQL 不跨域共享

## 3. 编号前缀生成规则

编号前缀按"域简称"组合，不预设固定清单；新增域时取一个未与主索引冲突的短名。

| 类型 | 前缀形态 | 示例形态 |
|------|---------|---------|
| 业务域 | `TC-<DOMAIN>-` | `TC-<业务域简称>-01` |
| 缺陷回归 | `TC-REG-` | `TC-REG-01` |

`<DOMAIN>` 取 2-4 个字母，能从字母看出域含义即可；与 `tc-main-<slug>.md` 已登记的前缀对齐或新增。

每个域文件内按验证方式分号段，号段之间留空位供后续新增：

| 号段 | 验证方式 |
|------|---------|
| 01-09 | API 验证 |
| 11-19 | UI 验证 |

号段一旦分配，进入验收矩阵或缺陷单后不重排；新增用例在对应号段内顺延。

## 4. SQL 文件命名与引用规则

| 文件类型 | 命名 | 说明 |
|---------|------|------|
| 种子数据 | `<PREFIX>-SEED.sql` | 配置表初始化；可被多条用例共用；必须自包含（`INSERT ... ON DUPLICATE KEY UPDATE` 而非裸 UPDATE） |
| 场景数据 | `<PREFIX>-<NN>.sql` | 与用例编号一一对应；必须同时包含"数据准备"段和"数据清理"段 |

存放位置：每个业务域自己的 `sql/` 子目录，路径 `<业务域>/sql/<PREFIX>-<NN>.sql`。

Markdown 中引用（相对于域内 markdown 文件）：
- 使用：`> 执行 [sql/<PREFIX>-<NN>.sql](sql/<PREFIX>-<NN>.sql) 中的"数据准备"段`
- 复用：`> 复用 [sql/<PREFIX>-<NN>.sql](sql/<PREFIX>-<NN>.sql)`
- 参数校验类用例可无 SQL 引用

## 5. 两层文件拆分规则

单域用例详情总行数超过约 200 行时，拆为索引层 + 详情层两个文件，双向引用。

| 层 | 命名 | 内容 |
|----|------|------|
| 索引层 | `tc-<业务域>.md` | 元信息、测试范围表、前置条件、验收矩阵、版本历史 + 指向详情层的链接 |
| 详情层 | `tc-<业务域>-用例详情.md` | 全部用例的步骤、数据准备、预期结果，按 API / UI 分节 |

双向回链：
- 索引层头部：`> **用例详情**: [tc-<业务域>-用例详情.md](tc-<业务域>-用例详情.md)`
- 详情层头部：`> **索引文件**: [tc-<业务域>.md](tc-<业务域>.md)`

用例详情 ≤ 200 行的域不拆；索引与详情合在单个文件里。

## 6. 主索引与 README 职责

| 文件 | 路径 | 职责 |
|------|------|------|
| TC 主索引 | `docs/test-case/tc-main-<slug>.md` | 登记全部业务域文件、编号前缀、用例统计、验收覆盖率、全量 `[待确认]` 汇总 |
| 落地 README（可选） | `docs/test-case/README.md` | 记录本宿主项目选定的目录结构、编号前缀清单、SQL 命名等约定 |

生成/更新顺序：每生成或更新一个业务域 TC 文件后，回头同步更新 `tc-main-<slug>.md` 的域登记与覆盖率统计；不依赖 README 也能工作，但已有 README 时以 README 登记为准。

## 7. 业务样本字典占位规则

稳定样本字典（Phase 4 的样本归堆产物）由宿主项目自行定义，本文件不预置具体样本。

样本记录格式：

| 样本类型 | 业务含义 |
|---------|---------|
| `<样本类型>` | 该样本在业务里代表的人/车/订单/数据状态的一句话描述 |

生成规则：
- 样本类型命名使用宿主项目的业务术语（取自 foundation-glossary 或 PRD）
- 每个样本类型应能对应到一组可稳定构造的数据准备 SQL 片段
- 样本一旦稳定，后续用例优先复用而非临时拼装；多域共用的样本提取为独立 SQL 文件被共享
